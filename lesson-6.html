Service Layer, API & Implementation Plan

Bring the gathering system to life: implement the orchestrator service, API endpoints, migrate existing data, and execute a phased build.

Scenario

You've just been assigned to build the TAM gathering system. You need to know: What orchestrates the pipeline? What API do frontend and MCP clients call? What files do I create, modify, or reuse? In what order do I build it? How do I import 25,000 existing company records without losing source attribution?

GatheringService — The Orchestrator

The GatheringService is a single Python class that orchestrates the entire gathering pipeline. It validates inputs, calls adapters, stores results, triggers downstream processing, and provides history endpoints.

    # ── GATHER ──

    async def scrape_companies(

How start_gathering() Works Internally

When the API receives a POST /api/pipeline/gathering/start request, the service orchestrates these 9 steps:

  1. Create GatheringRun record — Save project_id, source_type, filters, status='running', created_by, notes.
  2. Validate filters — Use the Pydantic schema for source_type to check filter structure and required fields.
  3. Load adapter — Look up adapter class from registry by source_type (e.g., "apollo.people.emulator" → ApolloOrgAPIAdapter).
  4. Execute adapter — Call adapter.execute(filters) → returns list of raw company dicts.
  5. Update raw_results_count — Set GatheringRun.raw_results_count = len(results).
  6. Deduplicate — Call dedup_and_store(run_id, results) → normalizes domains, checks/creates DiscoveredCompanies, creates CompanySourceLinks.
  7. Run blacklist check — Call run_blacklist_check(run_id) → marks rejected companies, creates gate if any found.
  8. Enqueue scraping — For accepted companies, queue scrape_companies(run_id) in background job queue (Celery/RQ).
  9. Return GatheringRun — Return created run with status='running' and job_id for frontend polling.

API Endpoints — All 16

These endpoints form the complete public interface for the gathering pipeline:

Method Endpoint Description
POST /api/pipeline/gathering/start Start a new gathering run. Body: project_id, source_type, filters, triggered_by, notes. Returns: GatheringRun with status and job_id.
GET /api/pipeline/gathering/runs List all gathering runs for a project. Query: project_id, source_type (optional), status (optional), date_range (optional). Returns: paginated GatheringRun list with counters.
GET /api/pipeline/gathering/runs/{id} Get full run detail: metadata, statistics (raw_count, new_companies, duplicates, rejected, accepted), filter summary, phase timeline. Returns: GatheringRunDetail.
GET /api/pipeline/gathering/runs/{id}/companies Get paginated companies from a run. Query: page (default 1), limit (default 50). Returns: PaginatedCompanies with DiscoveredCompany + source link metadata.
POST /api/pipeline/gathering/continue/{id} Resume pipeline from a specific phase. Body: next_phase (e.g., 'scrape', 'analyze'). Returns: PhaseResult with job_id and phase status.
POST /api/pipeline/gathering/estimate Get cost estimate without executing. Body: source_type, filters. Returns: { estimated_cost_usd, estimated_duration_seconds, billable_entities }.
GET /api/pipeline/gathering/sources List all available source adapters with their filter JSON schemas. Returns: list of { source_type, description, schema, example_filters }.
GET /api/pipeline/gathering/sources/{type}/schema Get JSON schema for a specific source type. Returns: Pydantic schema as JSON Schema spec for frontend form generation.
GET /api/pipeline/gathering/approval-gates List pending approval gates (status='pending'). Query: project_id (optional), gate_type (optional). Returns: list of ApprovalGate with scope details.
POST /api/pipeline/gathering/approval-gates/{id}/approve Approve a gate. Body: operator (user ID), note (optional). Updates gate.status='approved', resumes pipeline.
POST /api/pipeline/gathering/approval-gates/{id}/reject Reject a gate. Body: operator (user ID), note (required). Updates gate.status='rejected', triggers cleanup/retry logic.
GET /api/pipeline/gathering/scrapes/{company_id} Get all scrapes for a company. Returns: list of CompanyScrape with versions, timestamps, pages, content summaries.
POST /api/pipeline/gathering/scrapes/refresh Trigger re-scrape for expired content. Body: company_ids (list). Returns: job_id and estimated completion time.
GET /api/pipeline/gathering/analysis-runs List analysis runs. Query: project_id, status, date_range. Returns: paginated AnalysisRun list with result summary.
GET /api/pipeline/gathering/analysis-runs/{id} Get analysis run detail with result summary by confidence band (high/medium/low), model metadata.
GET /api/pipeline/gathering/analysis-runs/{a}/compare/{b} Compare two analysis runs. Returns: { agreements, disagreements (by confidence delta), new findings, dropped findings, metrics diff }.
GET /api/pipeline/gathering/blacklist-check Dry-run blacklist check on domains. Query: domains (comma-separated). Returns: { blacklisted: [...], reasons: {...}, sources: [...] }.

Migrating Existing EasyStaff Data

We have ~25,000 companies and 12,000 people records already gathered. We must import them into the new system while maintaining source attribution and run metadata.

What Exists

File Records Source

data/uae_god_search_companies.json 5,602 apollo_people_ui, Strategy B (seniority + location)

Migration Script: scripts/migrate_existing_tam.py — 5 Steps

Step 1: Create GatheringRuns for each historical search

  • Run 1 (Strategy A, keyword search):
    • source_type: apollo.people.emulator
    • source_subtype: strategy_a
    • filters: { "person_locations": ["Dubai, UAE"], "q_organization_name": "32 keywords", "max_pages": 10 }
    • status: completed
    • created_by: migration_script
    • raw_results_count: 295 (set after dedup)
  • Run 2 (Strategy B, seniority + location):
    • source_type: apollo.people.emulator
    • source_subtype: strategy_b
    • filters: { "person_locations": ["Dubai, UAE", "Abu Dhabi, UAE", "Sharjah, UAE"], "person_seniorities": ["founder", "c_suite", "owner"], "organization_num_employees_ranges": ["1,10", "11,50", "51,200", "201,500", "501,1000", "1001,5000", "5001,10000", "10001,+" ], "max_pages": 10 }
    • status: completed
    • raw_results_count: 5,602
  • Run 3 (Companies tab, industry + employee tags):
    • source_type: apollo.companies.emulator
    • source_subtype: industry_tags
    • filters: { "organization_locations": ["United Arab Emirates"], "organization_industry_tag_ids": [...], "organization_num_employees_ranges": [...], "max_pages": 100 }
    • status: completed
    • raw_results_count: 7,782

Step 2: Create/update DiscoveredCompanies for each unique domain

  • Read each company record from the JSON files.
  • Extract domain (or construct from company name if missing).
  • Call domain_service.normalize_domain(raw_domain) to standardize (lowercase, remove www, etc.).
  • Upsert DiscoveredCompany by (project_id, domain). If exists, update only empty fields (name, employees, linkedin_url) with richest source data.
  • Link to GatheringRun via gathering_run_id column.
  • Set first_source_date to migration timestamp, last_source_date = now, source_count = 1 (incremented when linked to multiple runs below).

Step 3: Create company_source_links for each (company, run) pair

  • For every company in the JSON files, create a CompanySourceLink record.
  • Fields: company_id (FK to DiscoveredCompany), gathering_run_id (FK to GatheringRun from Step 1), source_data (the raw JSON record), source_rank (position in original results list, 1-indexed).
  • This preserves full attribution: which company came from which run and at what rank.
  • When a company appears in multiple runs (e.g., both Strategy A and Strategy B), increment source_count in DiscoveredCompany.

Step 4: Import people as ExtractedContacts (from uae_god_search_people.json)

  • Parse the 12,201 people records from uae_god_search_people.json.
  • For each person, extract name, title, email, company, location.
  • Link to DiscoveredCompany by domain (extracted from company name or previous mapping).
  • Create ExtractedContact record with source=APOLLO, gathering_run_id=Run2 (Strategy B).
  • Store all raw fields in contact_data (JSON) for later enrichment or de-duping.

Step 5: Update GatheringRun counters

  • For each GatheringRun, compute and store counters:
  • raw_results_count = count of records in JSON file.
  • new_companies_count = count of DiscoveredCompanies created (not already in project).
  • duplicate_count = raw_count - new_count (companies that already existed).
  • rejected_count = count of records that failed blacklist or domain validation (if any).
  • accepted_count = new + duplicates (all successfully stored).

Migration Order

  1. Run migration for largest dataset first (uae_god_search = 5,602 companies). This establishes the base DiscoveredCompany set and avoids later merges.
  2. Then migrate keyword search (295) — mostly duplicates of Strategy B, but creates source_links for attribution.
  3. Then migrate companies tab (7,782) — many new (different discovery method), some overlapping. Updates DiscoveredCompany with richer employee/industry data.
  4. Separately, import people records (12,201) — link to existing DiscoveredCompanies by domain.

Implementation Order — Phases A–F

Follow this phased approach to build and deploy the system incrementally, testing as you go.

Phase A: Data Layer (1 day)

  • Task A1: Create all SQLAlchemy models in backend/app/models/gathering.py. Define: GatheringRun, CompanySourceLink, CompanyScrape, AnalysisRun, AnalysisResult, ApprovalGate. Add all columns, relationships, and indexes. (2 hours)
  • Task A2: Write Pydantic schemas in backend/app/schemas/gathering.py. Include per-source filter schemas (ApolloPeopleFilters, ApolloCompaniesFilters, ClayFilters, CSVImportFilters, ManualFilters) and all request/response models. (2 hours)
  • Task A3: Create Alembic migration in backend/alembic/versions/202603201_gathering_system.py. Add all 6 new tables, foreign keys, indexes. Modify discovered_companies and search_jobs. Test locally. (1.5 hours)
  • Task A4: Create materialized view for CRM blacklist in database. Add refresh function. Test query performance. (2 hours)

Phase B: Core Service + Apollo Adapter (2 days)

  • Task B1: Create backend/app/services/gathering_adapters/base.py with GatheringAdapter ABC. Define validate_filters() and execute() signatures. (1 hour)
  • Task B2: Create adapter registry in backend/app/services/gathering_adapters/__init__.py. Implement get_adapter(source_type) and list_adapters(). Auto-register all adapters on import. (1 hour)
  • Task B3: Create backend/app/services/gathering_adapters/apollo_org_api.py — simplest adapter. Wraps apollo_service.search_organizations(). Test with sample filters. (3 hours)
  • Task B4: Create GatheringService core methods: start_gathering(), dedup_and_store(), run_blacklist_check(). Implement all 9 internal steps. Unit test each. (8 hours)
  • Task B5: Create backend/app/api/gathering.py with endpoints: /start, /runs, /runs/{id}, /runs/{id}/companies. Wire up to GatheringService. Test with curl/Postman. (4 hours)

Phase C: Scraping + Analysis (1–2 days)

  • Task C1: Add scrape_to_db() method to backend/app/services/scraper_service.py. Write CompanyScrape records to DB. Test schema and content hashing. (2 hours)
  • Task C2: Implement scrape_companies() method in GatheringService. Handle TTL checks, force refresh, per-page scraping. Enqueue background jobs. (3 hours)
  • Task C3: Implement start_analysis() and compare_analysis_runs() methods. Extract AI logic from company_search_service. Store results in analysis_runs/analysis_results tables. (4 hours)
  • Task C4: Create endpoints: /analysis-runs, /analysis-runs/{id}, /analysis-runs/{a}/compare/{b}. Test with sample data. (2 hours)

Key Takeaway

GatheringService is the single orchestrator for the entire pipeline. It calls adapters to fetch data, stores results in a normalized schema with full source attribution, runs deterministic filters (blacklist, domain validation), triggers scraping and analysis, manages approval gates, and serves history/detail endpoints. The 16 API endpoints expose all these capabilities to the frontend and MCP clients. By building in phases (data → core → scraping → migration → gates), you keep each layer testable and independent. The migration script brings in 25K historical records without data loss or broken lineage.
