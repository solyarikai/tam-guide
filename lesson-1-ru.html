Урок 1

Общая картина

Разберитесь в архитектуре системы сбора TAM, её принципах и в том, как конвейер организует обнаружение компаний, анализ и квалификацию в детерминированном потоке под контролем оператора.

Чему вы научитесь
  • Три основных принципа системы сбора TAM
  • Как конвейер движется от сбора к загрузке лидов в SmartLead
  • 10 фаз и 3 критических контрольных точки
  • Что означают «новая», «дубликат» и «отклонена» на каждой фазе
  • Как ворота одобрения оператора предотвращают случайные траты и обеспечивают корректность данных
  • Почему строгий порядок фаз защищает бизнес-логику и контроль затрат
  • Как восстанавливаются сбои сеанса без потери данных
Сценарий

Вы создаёте систему поиска лидов для B2B SaaS компании, запускающей несколько кампаний таргетирования (агентства ОАЭ, стартапы tech в ЕС, рекрутеры в Австралии). Вам нужно собрать более 5000 перспектив компаний, проверить их против существующих кампаний чтобы избежать дубликатов, спарсить их веб-сайты, проанализировать соответствие ICP, и загрузить квалифицированные в ваш инструмент аутрича. Весь процесс должен быть прозрачным, проверяемым, и давать оператору контроль в нужные моменты — не слишком рано (потеря времени), не слишком поздно (потеря денег). Это система сбора TAM.

Что мы строим

Переиспользуемую систему, которая помнит всё о сборе TAM:

  • Какие фильтры были применены в каком источнике (Apollo, Clay, Sales Navigator и т.д.)
  • Какие компании были найдены из каких фильтров
  • Содержимое сайтов (многие страницы, версионировано с TTL)
  • Запуски AI анализа (несколько моделей, несколько промптов, всё сохранено)
  • Проверки чёрного списка против CRM/кампаний
  • Ворота одобрения перед шагами траты кредитов

Три основных принципа

  1. MCP-ready: Каждый источник — модульный адаптер. Сегодня это Puppeteer эмуляторы; завтра пользователи подключат свои ключи API Apollo/Clay. Слой БД одинаков.
  2. Переиспользуемо в разных проектах: EasyStaff Global, OnSocial, Inxy, любой будущий проект — те же таблицы, тот же конвейер.
  3. Работает только на Hetzner: Все скрипты, парсеры, миграции, запросы к БД выполняются на продакшн-сервере (46.62.210.24). Локальные машины только для редактирования кода. Claude Code SSH на Hetzner для любого выполнения.

Поток конвейера — строгий линейный, без пропусков

Система принудительно обеспечивает строгую линейную последовательность без ярлыков:

GATHER + DEDUP → BLACKLIST → ★ CP1 — STOP → PRE-FILTER → SCRAPE → ANALYZE → ★ CP2 — STOP → VERIFY → ★ CP3 — STOP → GOD_SEQ → PUSH

CP1 это реальное подтверждение проекта. Показывает кампании и контакты проекта так чтобы оператор подтвердил что находится в правильном проекте И что область видимости чёрного списка корректна. Это нельзя пропустить — это approval_gate в базе данных.

Машина состояний фазы

Система отслеживает прогресс через детерминированную машину состояний сохранённую в gathering_run.current_phase:

gathered → awaiting_scope_ok → scope_approved → filtered → scraped → analyzed → awaiting_targets_ok → targets_approved → awaiting_verify_ok → verify_approved → verified → pushed

На фазах awaiting_* запуск физически заблокирован. В БД существует запись approval_gate. ЕДИНСТВЕННЫЙ способ вперёд это POST /approval-gates/{gate_id}/approve. Без кода пути это не пропустишь. Это переживает краши, перезагрузки сеанса, перезагрузки сервера — состояние в базе данных.

Детали фазы: все 10 фаз + 3 контрольные точки

# Фаза Стоимость Авто? Что происходит
1 GATHER varies auto Выполнить адаптер, получить сырой список компаний из источника
2 DEDUP $0 auto Нормализовать домены, сопоставить с существующими DiscoveredCompanies, создать source_links. «Уже известна» = существует в discovered_companies для ЭТОГО проекта из предыдущего запуска сбора. ЕЩЁ не в чёрном списке — это дальше.
3 BLACKLIST $0 auto Проверка CRM с областью проекта + project_blacklist + enterprise_blacklist
— ★ КОНТРОЛЬНАЯ ТОЧКА 1 — STOP Подтверждение проекта + проверка чёрного списка. Показывает: имя/ID проекта, всего контактов, ВСЕ активные кампании (имя, платформа, лиды), разбор отклонений по кампании, списание предприятия, кросс-проектные предупреждения. Оператор подтверждает: «Да, это мой проект, и область видимости правильная.» Принудительно в коде — нельзя пропустить. Если неправильный проект → отменить запуск. Если неправильная кампания → исправить campaign_filters.
4 PRE-FILTER $0 auto Детерминированное отклонение: офлайн отрасли (ресторан, отель, строительство), мусорные домены (.gov, .edu), вредные паттерны. БЕЗ AI. Отклоняет 40-60%.
5 RESOLVE $0-low auto Для компаний без домена: LinkedIn URL → домен. Имя компании → Google поиск. Пропустить если домен уже известен.
6 SCRAPE $0 (httpx) auto Получить содержимое сайта. Проверка TTL — пропустить если свежее. Несколько страниц. Дешевле — одобрение не нужно.
7 ANALYZE ~$0.01-0.05 auto AI анализ (GPT-4o-mini). Оценки: industry_match, size_match, service_model. Дешевле — одобрение не нужно.
— ★ КОНТРОЛЬНАЯ ТОЧКА 2 — STOP Оператор проверяет список целей. Должен видеть: каждую компанию отмеченную как цель, её оценку уверенности, обоснование, сегмент. Оператор может принять, отклонить отдельные компании, переопределить вердикты. Только после подтверждения оператором список целей конвейер продолжает. Здесь оператор убеждается что AI не галлюцинировал. Кредиты не тратятся пока это не одобрено.
8 VERIFY $$$ BLOCKED Проверка email FindyMail. Это дорогой шаг. Запускается только на одобренные операторм целях.
— ★ КОНТРОЛЬНАЯ ТОЧКА 3 — STOP Оператор одобряет траты FindyMail. Должен видеть: сколько emails проверять, предполагаемую стоимость, какие компании. Оператор может удалить компании перед проверкой.
9 GOD_SEQUENCE ~$0.08 auto Создать 5-шаговую последовательность из 3-уровневого знания: универсальные паттерны + бизнес-знание (same sender_company) + project ICP. Оператор проверяет черновик перед push. POST /generate-sequence/ → POST /approve/
10 PUSH $0 BLOCKED POST /generated/{id}/push/ создаёт SmartLead кампанию (DRAFT) с GOD_SEQUENCE выводом. Оператор добавляет лиды + активирует.

Что означают «новая» и «дубликат» на каждой фазе

Это критично. Слова меняют смысл в зависимости от того где вы находитесь:

После фазы «Новая» означает «Дубликат» означает «Отклонена» означает
GATHER+DEDUP Домен не в discovered_companies для ЭТОГО проекта из любого предыдущего запуска сбора Домен уже известен из предыдущего запуска сбора (получает новый source_link, не новую запись) Ничего пока — нет применённых фильтров
BLACKLIST Прошла все проверки чёрного списка N/A В АКТИВНЫХ кампаниях ЭТОГО проекта, ИЛИ в project blacklist, ИЛИ в enterprise blacklist. НЕ в других проектах — это предупреждение, не отклонение.
PRE-FILTER Прошла матчинг паттернов N/A Офлайн отрасль, мусорный домен, вредный паттерн
ANALYZE AI говорит is_target=true N/A AI говорит не цель для этого ICP
КОНТРОЛЬНАЯ ТОЧКА 2 Оператор подтвердил как цель N/A Оператор отклонил / переопределил

Ошибки парсинга сайтов сообщаются

Когда запускается анализ, компании без спарсенного текста ПРОПУСКАЮТСЯ (не анализируются). Ответ анализа включает:

  • total_eligible: сколько компаний могло быть проанализировано
  • skipped_no_scraped_text: сколько пропущено потому что парсинг сайта не удался
  • total_analyzed: сколько было фактически проанализировано

Claude Code ДОЛЖЕН сообщить это на контрольной точке 2: «Проанализировано 300 из 500 возможных компаний. 200 пропущено (парсинг сайта не удался — нет текста сайта).»

Восстановление после сбоя сеанса

Всё состояние контрольной точки сохранено в базе данных (gathering_run.current_phase + таблица approval_gates).

🔄 Восстановление после сбоя сеанса: пошагово

Что происходит если Claude Code сеанс сбойнулся на контрольной точке 1:

  1. Следующий сеанс: Claude Code читает CLAUDE.md который говорит «сначала проверить запущенные»
  2. Запрос: GET /runs?project_id=X → находит запуск с current_phase=awaiting_scope_ok
  3. Запрос: GET /approval-gates?project_id=X → находит ожидающие ворота с gate_type=scope_verification
  4. Читает gate.scope которое содержит ПОЛНЫЙ детальный чёрный список (rejected_domains, warning_domains, разбор кампании)
  5. Показывает те же результаты контрольной точки 1 оператору
  6. Оператор одобряет → конвейер продолжает

Никакие данные не потеряны. JSON ворот сохраняет всё необходимое чтобы переконструировать отображение контрольной точки.

Почему этот порядок ненегосцируем

  1. BLACKLIST перед PRE-FILTER: Чёрный список это проектно-область бизнес-логика (кампании, CRM). Pre-filter это универсальный матчинг паттернов. Сначала бизнес-логика — если компания уже в вашем аутрече, не важно совпадает ли она с офлайн паттерном.
  2. КОНТРОЛЬНАЯ ТОЧКА 1 после BLACKLIST: Оператор ДОЛЖЕН убедиться что система корректно определила кампании проекта. Если campaign_filters неправильно сконфигурированы, чёрный список неправильный. Лучше поймать это перед потратой времени на 10K компаний.
  3. SCRAPE и ANALYZE авто (без контрольной точки): httpx парсинг бесплатен. GPT-4o-mini анализ стоит ~$0.01-0.05 для 500 компаний. Не стоит прерывать оператора за $0.03.
  4. КОНТРОЛЬНАЯ ТОЧКА 2 после ANALYZE: Здесь оператор видит фактический список целей. До этой точки всё одноразовое. После этой точки деньги тратятся.
  5. КОНТРОЛЬНАЯ ТОЧКА 3 перед VERIFY: FindyMail это реальные деньги ($0.01/email × 1000 emails = $10). Оператор должен одобрить точный список и стоимость.
  6. Нет Apollo API в потоке по умолчанию: Обогащение по умолчанию это Apollo UI эмулятор (Puppeteer) — парсит данные контактов с Apollo UI бесплатно. Apollo API стоит кредиты. Если нужны кредиты, это отдельные ворота одобрения.

Ключевой вывод

Система сбора TAM это детерминированный, поддерживаемый БД конвейер который принудительно применяет ворота одобрения в критических финансовых и решающих точках. Никаких ярлыков, никаких пропусков, никаких ошибок оператора — путь каждой компании прослеживаемый, каждая фаза обратима через состояние БД машины состояний, и каждый потраченный доллар явно одобрен заранее.
