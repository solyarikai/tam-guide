Урок 5

CRM-блэклист и Approval Gates

Узнайте, как система предотвращает дублирование контактов и контролирует расходы перед операциями, требующими кредитов.

Сценарий

Ваш оператор запускает сбор данных для Project 10 (Inxy, крипто-платежи). Они нашли 134 домена. 45 находятся в их собственных кампаниях. Но 89 появляются в активных кампаниях EasyStaff RU (Project 40, зарплатные услуги). Должна ли сбор данных Inxy автоматически отклонить все 89, чтобы избежать "контакта с одними и теми же компаниями"?

Критическое решение по дизайну

Ответ: Нет, абсолютно нет. Эти 89 доменов находятся в кампаниях EasyStaff, а не Inxy. Они представляют разные ценовые предложения, продаваемые потенциально разным personas покупателей в одной и той же компании.

Правило межпроектного взаимодействия: Только кампании одного проекта вызывают автоматическое отклонение. Кампании других проектов показываются как предупреждения. Это сохраняет TAM между проектами и уважает автономность параллельных направлений бизнеса.

Слои проверки блэклиста (по порядку)

Система проверяет домены по четырем слоям в строгой последовательности. Каждый слой имеет разные область действия и действие.

# Слой Область действия Действие Что это перехватывает
1 Блэклист проекта На проект Автоматическое отклонение project_blacklist таблица — ручные блокировки оператором
2 Кампании одного проекта На проект Автоматическое отклонение Контакты с доменами в активных кампаниях ЭТОГО проекта
3 Глобальный блэклист Глобальный Автоматическое отклонение enterprise_blacklist.json — конкуренты, запрещенные организации
4 Кампании других проектов Все проекты Только предупреждение Домены в кампаниях ДРУГИХ проектов (информационное)

Что видит оператор

Когда проверка блэклиста завершается, система возвращает два массива: отклоненные домены (с причинами) и домены-предупреждения (только информационно). Вот точная структура JSON:

Пример: JSON rejected_domains + warning_domains

Граничные случаи

Эти шесть сценариев показывают, как система ведет себя в сложных ситуациях:

Сценарий Поведение
Домен в кампаниях проекта A, сбор для проекта B Предупреждение (не отклонено) — разное ценовое предложение
Домен в кампаниях проекта A, сбор для проекта A Отклонено — уже находится в рассылке
Домен в глобальном блэклисте Отклонено независимо от проекта
Домен в блэклисте проекта A, сбор для проекта B Не отклонено — блэклист привязан к проекту
Кампания неправильно назначена проекту Оператор видит названия кампаний в деталях → может исправить отображение
Контакт имеет NULL project_id Игнорируется — неназначенные контакты не вызывают блэклист

Материализованное представление: быстрый поиск доменов

Система поддерживает материализованное представление, привязанное к проекту, всех активных доменов кампаний. Это позволяет выполнять проверки со скоростью менее миллисекунды во время сбора данных:

CREATE MATERIALIZED VIEW active_campaign_domains (SQL) 
CREATE MATERIALIZED VIEW active_campaign_domains AS
SELECT DISTINCT
  lower(c.domain) AS domain,
  c.project_id,
  p.name AS project_name,
  array_agg(DISTINCT camp.id) AS campaign_ids,
  array_agg(DISTINCT camp.name) AS campaign_names,
  count(DISTINCT c.id) AS contact_count
FROM contacts c
JOIN projects p ON p.id = c.project_id
JOIN campaigns camp ON camp.project_id = c.project_id
  AND camp.status = 'active'
WHERE c.domain IS NOT NULL AND c.domain != ''
GROUP BY lower(c.domain), c.project_id, p.name;

CREATE UNIQUE INDEX ON active_campaign_domains(domain, project_id);
CREATE INDEX ON active_campaign_domains(project_id);

Паттерны запросов:

  • Проверка одного проекта: WHERE domain = ANY(:domains) AND project_id = :pid
  • Проверка между проектами: WHERE domain = ANY(:domains) AND project_id != :pid

Стратегия обновления: REFRESH MATERIALIZED VIEW CONCURRENTLY после синхронизации кампании в crm_scheduler.py и после любого push кампании. Блокировки не требуются.

Также проверка: ранее обнаруженные компании

Система также предотвращает переобработку компаний из более ранних запусков сбора данных в одном проекте:

SELECT запрос для discovered_companies (SQL) 
-- Компании уже в системе для ЭТОГО проекта (не глобально)
SELECT DISTINCT domain FROM discovered_companies
WHERE domain = ANY(:domains) AND project_id = :pid
  AND status != 'REJECTED'

Это предотвращает переобработку компаний из предыдущих запусков сбора данных в одном проекте, экономя время и избегая дублирования анализа.

Approval Gates: контрольные точки оператора

Конвейер паузируется перед любой операцией, требующей расходования кредитов. Оператор проверяет объем, предполагаемую стоимость и список компаний, затем одобряет или отклоняет gate.

Типы Gates

Пять типов gates защищают различные этапы конвейера:

Тип Gate Когда активируется Стоимость
pre_scrape_crona Перед пакетным скрейпингом через Crona API Crona кредиты
pre_analysis Перед анализом AI (GPT/Gemini batch) LLM токены
pre_enrichment Перед пакетным обогащением Apollo 1 кредит/человек
pre_verification Перед проверкой email FindyMail FindyMail кредиты
pre_push Перед push в SmartLead/GetSales Нет (финальная проверка)

Бесплатный скрейпинг: Скрейпинг через httpx бесплатен — gate не требуется. Gates требуются только для шагов, требующих кредитов.

Жизненный цикл Gate (8 шагов)

  1. Конвейер создает gate: status = 'pending', created_at = now()
  2. База данных сохраняет: Полный объем (ID компаний, предполагаемая стоимость, использование кредитов)
  3. Оператор видит alert: Дашборд показывает pending gates, сгруппированные по проекту и типу
  4. Оператор проверяет детали: Названия компаний, предполагаемые стоимости, обоснование
  5. Оператор одобряет или отклоняет: POST /api/approval-gates/{id}/approve или /reject
  6. Конвейер возобновляется: Если одобрено, перейти к protected шагу. Если отклонено, пометить компании как 'not_approved' и остановиться.
  7. Автоматическое истечение: Если не решено в течение 48 часов, gate автоматически истекает (опционально, в зависимости от конфига)
  8. Audit trail: Все решения логируются: decided_by, decided_at, decision_note

Безопасность при сбоях: нет потерянной работы

Если система падает после одобрения gate, но до завершения protected операции, конвейер автоматически возобновляется. Сам gate остается неизменяемым — переповторяется только зависимая работа (обогащение, проверка).

Полный пример: BLACKLIST → pre_enrichment Gate

Вот 10-шаговый поток, показывающий ровно что происходит, когда оператор собирает 200 компаний, система их фильтрует, и создает gate обогащения:

  1. Сбор завершен: 200 уникальных доменов обнаружено (найдено в Apollo People, source_type = "apollo.people.emulator")
  2. Запускается проверка блэклиста:
    • 45 доменов отклонено (кампании одного проекта)
    • 12 доменов отклонено (блэклист проекта)
    • 8 доменов отклонено (глобальный блэклист)
    • 89 доменов возвращено как предупреждения между проектами (не отклонено)
    • Чистое одобрение: 135 доменов
  3. Скрейпинг происходит: 135 одобренных доменов скреплены (бесплатно через httpx)
  4. Результат скрейпинга: 128 сайтов полностью скреплены, 7 недоступны
  5. Запускается pre-filter: Детерминированные проверки (без AI)
    • Удаляет 8 доменов (офлайн индустрии, припаркованные сайты, спам домены)
    • Чистое после pre-filter: 120 компаний
  6. Фаза анализа: 120 компаний отправлено в GPT-4o-mini для классификации
  7. Анализ завершен: 45 помечено как targets, 75 как non-targets
  8. Создан pre-enrichment gate:
    • Объем: 45 ID целевых компаний
    • Предполагаемая стоимость: 45 Apollo кредитов (~$0.90)
    • Предполагаемое количество людей на компанию: 2.3 (в среднем)
    • Итого предполагаемое обогащение: ~103 контактных записи
    • Метка: "Одобрить 45 компаний для обогащения Apollo (~$0.90)"
  9. Оператор одобряет gate: decision = approved, decided_by = "alice@company.com"
  10. Выполняется обогащение: Конвейер продолжает, вызывает enrich_apollo_batch(45 companies). 103 контактные записи создаются в таблице extracted_contacts. Статус переходит на следующий этап.
Ключевой вывод

Автономия между проектами + защиты, привязанные к проекту. Система блэклиста признает, что разные проекты продают разные продукты потенциально разным decision-makers в одной и той же компании. Предупреждения держат операторов в курсе; отклонения предотвращают дублирование контактов. Approval gates гарантируют операторы остаются в контроле расходов.
