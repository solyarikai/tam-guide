Source-Agnostic архитектура и адаптеры

Как добавить новый источник данных без изменения базы данных — Принцип открытости/закрытости в действии.

Чему вы научитесь
  • Почему мы отвергаем типизированные модели базы данных для каждого источника (комбинаторный взрыв)
  • Как паттерн адаптера отделяет источники от базы данных
  • Валидация на уровне адаптера с Pydantic и extra='allow'
  • Полный реестр адаптеров и способ добавления нового источника
  • Все 15 текущих адаптеров и их поля фильтров
  • Как адаптеры clay обрабатывают лимит в 5000 внутри
  • Добавление Indeed или любого нового источника завтра (без изменения БД)
Сценарий из реальной жизни

Ваш вице-президент говорит: «Добавь Indeed как источник — найди всех Python-разработчиков в Остине. Не трогай базу данных.»

Вы улыбаетесь, пишете IndeedJobsAdapter, регистрируете его, развертываете. Готово. БД никогда не изменяется. Пайплайн обнаруживает его автоматически.

1. Проблема: почему не использовать типизированные модели для каждого источника?

Ваше первое желание может быть: «Давайте создадим Pydantic модель для каждого источника в базе данных. Фильтры Apollo здесь, фильтры Clay там.»

Проверка реальности:

  • Только Apollo: People UI, Companies UI, People API, Org API — 4 варианта с различными фильтрами
  • Clay: Companies UI, Companies API, People UI, People API — еще 4
  • Другие платформы: Sales Navigator, Google Maps, LinkedIn, Crunchbase, Indeed, Google Sheets, CSV, Manual, Google SERP — еще 9
  • Взрыв фильтров: Каждая платформа добавляет новые фильтры каждый месяц. Обновления моделей = миграции схемы = простой
  • Комбинаторный кошмар: платформы × цели × методы = слишком много моделей для поддержки

Решение: Source-agnostic база данных + паттерн адаптера.

Три измерения любого источника

Каждый источник сбора данных можно описать тремя ортогональными измерениями:

Измерение Колонка Примеры
Платформа source_platform apollo, clay, sales_navigator, google_maps, crunchbase, csv, google_sheets, manual, google
Цель source_target companies, people, jobs
Метод source_method api, emulator, manual

База данных объединяет их в: source_type = "{platform}.{target}.{method}" — например apollo.companies.emulator, clay.people.api, indeed.jobs.api.

Но вот ключевой момент: source_type — это просто бесплатный VARCHAR. База данных никогда не разбирает его. У неё нет никакого мнения о том, какие источники существуют. Семантику владеет адаптер.

Завтра: Добавьте indeed.jobs.api в реестр. На следующей неделе: linkedin_recruiter.people.api. Схема БД? Не изменилась.

2. Дизайн фильтров — базе данных все равно

Таблица gathering_runs имеет одну колонку для фильтров:

filters JSONB NOT NULL

Вот и все. База данных хранит то, что туда кладет адаптер. Она никогда не запрашивает отдельные поля фильтра. Она использует фильтры для:

  1. Хранение — записать один раз при запуске сбора
  2. Отображение — прочитать и показать оператору в UI
  3. Хеширование — SHA256 для обнаружения дубликатов (одинаковые фильтры = одинаковый хеш)
  4. Переповтор — передать адаптеру без изменений

Валидация Pydantic не происходит на уровне базы данных. Валидация — это работа адаптера, не схемы.

3. Валидация на уровне адаптера — extra='allow'

Каждый адаптер определяет СВОЮ собственную Pydantic модель. Это код адаптера, не код схемы. И каждая модель фильтра адаптера имеет одно критическое правило: 

class Config:
    extra = "allow"

Почему? Когда Apollo добавит новый фильтр на следующей неделе, пользователи передадут его через. Адаптер еще не знает о нем, но поле сохраняется в JSONB. Когда вы обновите код адаптера для его обработки, старые записи загружаются без проблем. Никакого сбоя.

Это секрет будущей защиты схемы фильтров без миграций базы данных.

💡

Валидация всегда живет в коде адаптера. Неизвестные поля проходят через. Когда требуется новый фильтр, адаптер его реализует — БД уже готова, потому что JSONB бессхемен.

4. Базовый класс GatheringAdapter

Все адаптеры наследуют от GatheringAdapter, абстрактного базового класса, который определяет контракт:

Полный код Python GatheringAdapter 
from abc import ABC, abstractmethod
from typing import Optional, Type, Callable
from pydantic import BaseModel

class EstimateResult:
    """Result of estimating a gathering before execution."""
    def __init__(self, estimated_companies: int, estimated_credits: float, cost_usd: float):
        self.estimated_companies = estimated_companies
        self.estimated_credits = estimated_credits
        self.cost_usd = cost_usd

class GatheringResult:
    """Result of executing a gathering."""
    def __init__(self, companies: list, metadata: dict):
        self.companies = companies
        self.metadata = metadata

class GatheringAdapter(ABC):
    """Base class for all gathering source adapters.

    The system is source-agnostic. The DB stores opaque JSONB filters.
    Each adapter owns its filter schema and execution logic.
    Adding a new source = write adapter + register. Zero DB changes.
    """

    source_type: str = None                            # "apollo.companies.emulator"
    filter_model: Optional[Type[BaseModel]] = None     # Pydantic model (optional)

    @abstractmethod
    async def validate(self, raw_filters: dict) -> dict:
        """Validate & normalize filters. Returns cleaned dict. Raises on invalid.

        If filter_model is set, validates via Pydantic. Otherwise, passes through.
        All filter models MUST have Config.extra = 'allow' for forward compat.

        Args:
            raw_filters: Dictionary of raw filter input from user/API

        Returns:
            Cleaned and normalized filter dictionary

        Raises:
            ValueError: If filters are invalid for this adapter
        """
        pass

    @abstractmethod
    async def estimate(self, filters: dict) -> EstimateResult:
        """Estimate cost/results without executing.

        Used to show user the approximate number of results and cost before running.
        For API adapters, may call API with dry_run flag. For UI adapters, may
        load the page and count visible results without scrolling.

        Args:
            filters: Validated filter dictionary

        Returns:
            EstimateResult with estimated_companies, estimated_credits, cost_usd
        """
        pass

    @abstractmethod
    async def execute(self, filters: dict, on_progress: Callable = None) -> GatheringResult:
        """Execute gathering. Returns companies list + metadata.

        Handles pagination, rate limiting, retries internally.
        Yields progress updates via on_progress callback.
        Normalizes all results to standard company dict format.

        Args:
            filters: Validated filter dictionary
            on_progress: Optional callback fn(current, total, message) for progress tracking

        Returns:
            GatheringResult with companies list and metadata
        """
        pass

    def get_filter_schema(self) -> Optional[dict]:
        """JSON Schema for MCP tool registration. None if fully dynamic.

        Returns the Pydantic model's JSON schema for use in API docs
        and MCP tool definitions. If this adapter doesn't have a strict
        filter schema, returns None.

        Returns:
            JSON Schema dict or None
        """
        if self.filter_model:
            return self.filter_model.model_json_schema()
        return None

    def get_capabilities(self) -> dict:
        """What this adapter can do. Used by UI and MCP for discovery.

        Returns a dictionary describing the adapter's capabilities,
        used by the UI to show which adapters are available and what
        features they support.

        Returns:
            Dictionary with keys:
                - source_type: str (e.g. "apollo.companies.emulator")
                - has_estimate: bool (can estimate before executing)
                - has_filter_schema: bool (has formal filter schema)
                - cost_model: str (free, per_result, per_page, per_credit)
                - requires_auth: bool (needs API key in integration_settings)
        """
        return {
            "source_type": self.source_type,
            "has_estimate": True,
            "has_filter_schema": self.filter_model is not None,
            "cost_model": "free",         # free | per_result | per_page | per_credit
            "requires_auth": False,       # needs API key in integration_settings
        }

Базовый класс определяет четыре ключевых метода:

  • validate(raw_filters): Валидация Pydantic + нормализация. Все модели используют extra='allow'.
  • estimate(filters): Сухой запуск для показа пользователю стоимости перед выполнением.
  • execute(filters, on_progress): Фактически собрать данные. Обрабатывает пагинацию, аутентификацию, повторы внутри.
  • get_filter_schema(): Возвращает JSON Schema для документации API.
  • get_capabilities(): Метаданные для обнаружения UI.

5. Реестр адаптеров

Глобальный реестр хранит все зарегистрированные адаптеры. Адаптеры саморегистрируются через декоратор.

Полный код Python реестра адаптеров 
from typing import Type, List

# Global registry — adapters self-register on import
ADAPTER_REGISTRY: dict[str, Type[GatheringAdapter]] = {}

def register_adapter(cls: Type[GatheringAdapter]) -> Type[GatheringAdapter]:
    """Decorator. @register_adapter on adapter class auto-registers it.

    Usage:
        @register_adapter
        class MyAdapter(GatheringAdapter):
            source_type = "my_platform.companies.api"
            ...

    Args:
        cls: The adapter class to register

    Returns:
        The adapter class (for use as a decorator)
    """
    if not cls.source_type:
        raise ValueError(f"Adapter {cls.__name__} must define source_type")
    ADAPTER_REGISTRY[cls.source_type] = cls
    return cls

def get_adapter(source_type: str) -> GatheringAdapter:
    """Get an adapter instance by source_type.

    Args:
        source_type: e.g. "apollo.companies.emulator"

    Returns:
        An instance of the adapter

    Raises:
        ValueError: If source_type not in registry
    """
    cls = ADAPTER_REGISTRY.get(source_type)
    if not cls:
        available = sorted(ADAPTER_REGISTRY.keys())
        raise ValueError(
            f"Unknown source: {source_type}. Available: {available}"
        )
    return cls()

def list_adapters() -> List[dict]:
    """Returns all registered adapters + capabilities.

    Used by /gathering/sources endpoint and MCP tool discovery.
    Returns capabilities for each registered adapter.

    Returns:
        List of capability dicts, one per adapter
    """
    return [
        get_adapter(st).get_capabilities()
        for st in sorted(ADAPTER_REGISTRY.keys())
    ]

Вот и все. Когда бэкэнд запускается, все адаптеры импортируются и саморегистрируются. Эндпоинт API /gathering/sources возвращает список динамически. UI обнаруживает новые адаптеры без перезагрузки.

Адаптер Google Sheets — автоопределение

Адаптер Google Sheets уникален: члены команды вручную собирают лиды в Apollo, Clay или LinkedIn, вставляют их в общую Google Таблицу и передают URL пайплайну.

Автоопределение маппинга колонок: Адаптер читает строку заголовков и автоматически маппит колонки на основе общих паттернов:

  • domain, website, url → домен компании
  • company, name, organization → имя компании
  • linkedin, linkedin_url → URL профиля LinkedIn
  • employees, headcount, size → количество сотрудников
  • industry, vertical → индустрия
  • country, location → страна
  • city → город
  • email, contact_email → электронная почта

Требуется: Таблица должна быть поделена с доступом «Anyone with link».

7. Добавление нового источника — без изменения базы данных

Скажем, вам нужно добавить Indeed как источник вакансий. Вот полный код:

Пример: полная реализация IndeedJobsAdapter 
# backend/app/services/gathering_adapters/indeed_jobs.py

from pydantic import BaseModel, Field
from typing import Optional
from .base import GatheringAdapter, EstimateResult, GatheringResult, register_adapter

class IndeedJobsFilters(BaseModel):
    """Filter model for Indeed job search."""

    job_title: str = Field(..., description="Job title to search for")
    location: str = Field(..., description="Location (city, state, or postal code)")
    radius_miles: Optional[int] = Field(default=25, description="Search radius")

    company_exclude: Optional[list[str]] = Field(default=[], description="Companies to exclude")
    job_type: Optional[list[str]] = Field(default=[], description="Full-time, part-time, contract, temporary")
    salary_min: Optional[int] = Field(default=None, description="Minimum salary in USD")
    date_posted: Optional[str] = Field(default="anytime", description="Last 7 days, last 30 days, anytime")

    max_results: Optional[int] = Field(default=500, description="Max jobs to retrieve")

    class Config:
        extra = "allow"  # Future-proof: unknown fields pass through

@register_adapter
class IndeedJobsAdapter(GatheringAdapter):
    """Adapter for Indeed job search API."""

    source_type = "indeed.jobs.api"
    filter_model = IndeedJobsFilters

    async def validate(self, raw_filters: dict) -> dict:
        """Validate filters via Pydantic."""
        validated = IndeedJobsFilters(**raw_filters)
        return validated.model_dump()

    async def estimate(self, filters: dict) -> EstimateResult:
        """Estimate job count via Indeed API dry-run."""
        # Call Indeed API with dry_run=true
        # Parse response to get estimated count
        # Assume 0 cost (Indeed doesn't charge per query)
        import httpx

        async with httpx.AsyncClient() as client:
            response = await client.get(
                "https://api.indeed.com/v2/jobs/search",
                params={
                    "query": filters["job_title"],
                    "location": filters["location"],
                    "radius": filters.get("radius_miles", 25),
                    "sort": "date",
                    "count": 0  # dry-run, no results
                },
                headers={"Authorization": f"Bearer {self.indeed_api_key}"}
            )
            data = response.json()
            estimated_jobs = data.get("totalResults", 100)

        return EstimateResult(
            estimated_companies=estimated_jobs,
            estimated_credits=0,
            cost_usd=0
        )

    async def execute(self, filters: dict, on_progress=None) -> GatheringResult:
        """Execute Indeed job search and normalize results."""
        import httpx

        companies = []
        max_results = filters.get("max_results", 500)
        retrieved = 0
        page = 0

        async with httpx.AsyncClient() as client:
            while retrieved < max_results:
                response = await client.get(
                    "https://api.indeed.com/v2/jobs/search",
                    params={
                        "query": filters["job_title"],
                        "location": filters["location"],
                        "radius": filters.get("radius_miles", 25),
                        "offset": page * 25,
                        "count": 25,
                        "sort": "date"
                    },
                    headers={"Authorization": f"Bearer {self.indeed_api_key}"}
                )

                data = response.json()
                jobs = data.get("results", [])

                if not jobs:
                    break

                for job in jobs:
                    if retrieved >= max_results:
                        break

                    # Normalize to company dict
                    company = {
                        "domain": self._extract_domain(job.get("company_url", "")),
                        "name": job.get("company_name"),
                        "industry": None,
                        "employees": None,
                        "country": None,
                        "city": filters.get("location"),
                        "source_data": {
                            "job_id": job.get("job_id"),
                            "job_title": job.get("job_title"),
                            "salary": job.get("salary"),
                            "url": job.get("url"),
                        }
                    }

                    if company["domain"] and company["name"]:
                        companies.append(company)
                        retrieved += 1

                        if on_progress:
                            on_progress(retrieved, max_results, f"Fetched {retrieved} jobs")

                page += 1

        return GatheringResult(
            companies=companies,
            metadata={
                "total_fetched": retrieved,
                "method": "indeed_jobs_api",
                "filters_applied": filters
            }
        )

    def _extract_domain(self, url: str) -> Optional[str]:
        """Extract domain from company URL."""
        if not url:
            return None
        from urllib.parse import urlparse
        parsed = urlparse(url)
        return parsed.netloc if parsed.netloc else None

    def get_capabilities(self) -> dict:
        return {
            "source_type": self.source_type,
            "has_estimate": True,
            "has_filter_schema": True,
            "cost_model": "free",
            "requires_auth": True,  # Needs Indeed API key
        }

Вот и все. Три простых шага для добавления Indeed или любого нового источника:

  1. Напишите класс адаптера с моделью фильтра Pydantic
  2. Украсьте декоратором @register_adapter
  3. Разверните. Система обнаруживает его автоматически.

База данных? Без изменений. API? Автоматически обновляется. UI? Обнаруживает на обновлении. Новые адаптеры работают с существующим пайплайном, шлюзами одобрения и логикой дедупликации без прикосновения к основному коду.

8. Внутренние операции адаптера — прозрачны для пайплайна

Адаптеры обрабатывают сложность внутри, невидимо для остальной системы:

Лимит Clay 5000

Когда предполагаемое количество результатов Clay превышает 5000, адаптер автоматически разделяет по географии. Он создает несколько gathering_runs под одинаковым pipeline_run_id, каждый с _geo_split_label в фильтрах. Остальная система видит один запрос; адаптер обрабатывает разделения безупречно.

Пагинация Apollo

Адаптеры UI и API Apollo обрабатывают итерацию страниц внутри. Адаптер проходит по страницам, агрегирует результаты и возвращает их как один нормализованный список. Пайплайн никогда не видит логику пагинации.

Ограничение скорости

Каждый адаптер реализует ограничение скорости внутри:

  • Apollo: 0.3 секунды на запрос (консервативно)
  • Clay: Соблюдает ограничения времени экспорта
  • Sales Navigator: Соблюдает ограничения скорости LinkedIn
  • Google APIs: Элегантно обрабатывает исчерпание квоты

Аутентификация

Адаптеры API читают ключи аутентификации из integration_settings (сохраненные API ключи пользователя). Адаптеры UI (на основе Puppeteer) обрабатывают вход прямо с сохраненными учетными данными.

Повтор и откат

Все адаптеры реализуют экспоненциальный откат для 429 (ограничение скорости), 503 (служба недоступна) и переходящих ошибок. Тайм-ауты обрабатываются элегантно, быстро завершаются, если источник недоступен.

Ключевой вывод

Паттерн адаптера отделяет реализации источников от схемы базы данных. Новые источники добавляются как код (адаптеры) + регистрация, не миграции базы данных. Паттерн extra='allow' в Pydantic гарантирует, что фильтры могут развиваться без нарушения старых записей. Это Принцип открытости/закрытости в действии: открыт для новых источников, закрыт для изменений базы данных.
