TAM Gathering Architecture

Полный справочник по pipeline лидогенерации — от поиска компаний до email-кампаний. Архитектура, потоки данных, API-контракты и пример реализации на практике.

Начало работы

Что делает система, зачем она нужна и ключевые принципы проектирования за каждым решением.

Обзор системы

Переиспользуемая система, которая запоминает всё о сборе TAM:

  • Какие фильтры были применены в каком источнике (Apollo, Clay, Sales Navigator и т.д.)
  • Какие компании были найдены из каких фильтров
  • Содержимое сайтов (несколько страниц, версионировано с TTL)
  • Запуски AI анализа (несколько моделей, несколько промптов, всё сохранено)
  • Проверки блэклиста против CRM/кампаний
  • Approval gates перед этапами, требующими кредитов

MCP-готовая: Каждый источник — подключаемый модуль адаптера. Сегодня это эмуляторы Puppeteer; завтра пользователи подключат свои ключи Apollo/Clay API. Слой БД остаётся неизменным.

Переиспользуемая по проектам: EasyStaff Global, OnSocial, Inxy, любой будущий проект — те же таблицы, тот же пайплайн.

Запускается только на Hetzner: Все скрипты, скраперы, миграции, запросы к БД выполняются на production сервере (46.62.210.24). Локальные машины используются только для редактирования кода. Claude Code подключается по SSH на Hetzner для всех операций.

Ключевые концепции

Система строится на нескольких фундаментальных идеях:

  • Source-agnostic DB: БД хранит непрозрачный JSONB. Она не знает про Apollo, Clay или любую другую платформу.
  • Filter memory: Каждый поиск сохраняется с точными параметрами. Переиспользуемо. Перезапускаемо.
  • Multi-source dedup: Одна компания, найденная 5 источниками = 1 DiscoveredCompany + 5 source_links.
  • Cheapest first: Бесплатная проверка паттернов → бесплатный блэклист → дешёвый скрапинг → дорогой AI → очень дорогое обогащение.
  • Full provenance: gathering_run → DiscoveredCompany → ExtractedContact → CRM Contact → Meeting → Deal

Пайплайн

Как компании проходят через систему — от поиска до email-кампании. Шаги, чекпойнты, контроль расходов и источники данных.

Поток пайплайна — строго линейный, без пропусков

GATHER + DEDUP → BLACKLIST → ★ CP1 — STOP → PRE-FILTER → SCRAPE → ANALYZE → ★ CP2 — STOP → VERIFY → ★ CP3 — STOP → GOD_SEQ → PUSH

CP1 — это настоящее подтверждение проекта. Оно показывает кампании и контакты проекта, так что оператор подтверждает, что находится в правильном проекте И что область блэклиста правильная. Это не может быть пропущено — это approval_gate в базе данных.

Конечный автомат фазы (gathering_run.current_phase):

gathered → awaiting_scope_ok → scope_approved → filtered → scraped → analyzed → awaiting_targets_ok → targets_approved → awaiting_verify_ok → verify_approved → verified → pushed

На фазах awaiting_* run физически заблокирован. Запись approval_gate существует в БД. Единственный способ вперёд — POST /approval-gates/{gate_id}/approve. Ни один путь кода это не пропускает. Это пережидает крахи, перезагрузки сессий, перезапуски сервера — состояние в базе данных.

Детали фаз

# Фаза Стоимость Авто? Что она делает
1 GATHER varies авто Выполнить адаптер, получить сырой список компаний из источника
2 DEDUP $0 авто Нормализовать домены, сопоставить с существующими DiscoveredCompanies, создать source_links. "Уже известно" = существует в discovered_companies для ЭТОГО проекта из предыдущего запуска сбора. ЕЩЁ НЕ в блэклисте — это дальше.
3 BLACKLIST $0 авто Project-scoped проверка CRM + project_blacklist + enterprise_blacklist
— ★ CHECKPOINT 1 — СТОП Подтверждение проекта + проверка блэклиста. Показывает: имя/ID проекта, всего контактов, ВСЕ активные кампании (имя, платформа, лиды), разбор отклонений по кампании, enterprise blacklist, cross-project предупреждения. Оператор подтверждает: "Да, это мой проект, и область правильная." Принудительно кодом — не может быть пропущено. Если неправильный проект → отменить run. Если неправильная кампания → исправить campaign_filters.
4 PRE-FILTER $0 авто Детерминированное отклонение: offline индустрии (ресторан, отель, строительство), мусор домены (.gov, .edu), мусор паттерны. БЕЗ AI. Отклоняет 40-60%.
5 RESOLVE $0-низко авто Для компаний без домена: LinkedIn URL → домен. Имя компании → Google поиск. Пропустить если домен уже известен.
6 SCRAPE $0 (httpx) авто Получить содержимое сайта. Проверка TTL — пропустить если свежо. Несколько страниц. Дешёво — одобрение не требуется.
7 ANALYZE ~$0.01-0.05 авто AI анализ (GPT-4o-mini). Оценки: industry_match, size_match, service_model. Дешёво — одобрение не требуется.
— ★ CHECKPOINT 2 — СТОП Оператор проверяет список целей. Должен видеть: каждую компанию, отмеченную как целевая, её confidence оценку, рассуждение, сегмент. Оператор может принять, отклонить отдельные компании, переопределить вердикты. Только после подтверждения оператором пайплайн продолжается. Здесь оператор гарантирует, что AI не галлюцинирует. Никакие кредиты не тратятся до одобрения.
8 VERIFY $$$ BLOCKED FindyMail email верификация. Это дорогой этап. Запускается только для одобренных оператором целей.
— ★ CHECKPOINT 3 — СТОП Оператор одобряет FindyMail расходы. Должен видеть: сколько email проверить, предполагаемая стоимость, какие компании. Оператор может удалить компании перед верификацией.
9 GOD_SEQUENCE ~$0.08 авто Создать 5-шаговую последовательность из 3-уровневого знания: универсальные паттерны + деловое знание (та же sender_company) + project ICP. Оператор проверяет черновик перед пушем. POST /generate-sequence/ → POST /approve/
10 PUSH $0 BLOCKED POST /generated/{id}/push/ создаёт SmartLead кампанию (DRAFT) с GOD_SEQUENCE выводом. Оператор добавляет лиды + активирует.

Значения "new" и "duplicate" на каждой фазе

Это критично. Слова меняют смысл в зависимости от того, где вы находитесь:

После фазы "New" означает "Duplicate" означает "Rejected" означает
GATHER+DEDUP Домен не в discovered_companies для ЭТОГО проекта из любого предыдущего запуска сбора Домен уже известен из предыдущего запуска сбора (получает новый source_link, не новую запись) Ничего ещё — никакая фильтрация не применена
BLACKLIST Прошёл все проверки блэклиста N/A В ЭТОМ проекте активные кампании, ИЛИ в project blacklist, ИЛИ в enterprise blacklist. НЕ в других проектах — это предупреждение, не отклонение.

ANALYZE AI говорит is_target=true N/A AI говорит не целевая для этого ICP
CHECKPOINT 2 Оператор подтвердил как целевую N/A Оператор отклонил / переопределил

Восстановление после сбоя сессии

Все состояния чекпойнтов сохранены в БД (gathering_run.current_phase + таблица approval_gates).

Что происходит, если сессия Claude Code падает на чекпойнте 1:

  1. Следующая сессия: Claude Code читает CLAUDE.md, которая говорит "сначала проверить текущие запуски"
  2. Запросы: GET /runs?project_id=X → находит run с current_phase=awaiting_scope_ok
  3. Запросы: GET /approval-gates?project_id=X → находит pending gate с gate_type=scope_verification
  4. Читает gate.scope, которая содержит ПОЛНЫЕ детали блэклиста (rejected_domains, warning_domains, разбор кампаний)
  5. Показывает те же результаты чекпойнта 1 оператору
  6. Оператор одобряет → пайплайн продолжается

Никакие данные не теряются. JSON scope хранит всё необходимое для восстановления отображения чекпойнта.

Отчёт об ошибках скрапинга

Когда анализ запускается, компании без скрепленного текста ПРОПУСКАЮТСЯ (не анализируются). Ответ анализа включает:

  • total_eligible: сколько компаний могли быть проанализированы
  • skipped_no_scraped_text: сколько пропущено, потому что скрапинг не удался
  • total_analyzed: сколько фактически проанализировано

Claude Code ДОЛЖЕН сообщить это на чекпойнте 2: "Проанализировано 300 из 500 eligible компаний. 200 пропущено (скрапинг не удался — нет текста сайта)."

Конфигурация проекта, которая управляет поведением pipeline — описание ICP, сегменты, промпты классификации и маппинг аккаунтов.

Контекст проекта

Система сбора работает не в вакууме. Она читает существующее знание проекта для умных решений.

Режимы ввода оператора

Когда оператор (или MCP агент) запускает сбор, они могут предоставить ввод несколькими способами:

Mode 1: Natural Language (самый простой, MCP-оптимизированный)

"Find 5000 digital agencies in UAE with <100 employees"

→ AI отображает на фильтры, используя project's ICP + segment контекст
→ Предлагает лучший источник (Apollo для широкого, Clay для конкретного)
→ Возвращает предполагаемые результаты + стоимость

Mode 2: Structured Filters (опытные операторы, точные)

{
"source_type": "apollo.people.emulator",
"filters": {
"person_locations": ["Dubai, United Arab Emirates"],
"person_seniorities": ["founder", "c_suite"],
"organization_num_employees_ranges": ["1,10", "11,50", "51,100"]
}
}

→ Пропускает AI отображение, прямое выполнение

Mode 3: Lookalike (из case studies или существующих клиентов)

"Find companies similar to: frizzon.ae, 10xbrand.com, zopreneurs.com"

→ Reverse-engineer паттерны (индустрия, размер, местоположение, tech stack)
→ Создать фильтры, которые соответствуют паттерну
→ Уже частично существует: reverse_engineering_service.py

Mode 4: Expand/Repeat (итерирование предыдущих запусков)

"Run the same search as gathering_run #42 but for Singapore instead of Dubai"

→ Копировать фильтры, модифицировать локацию
→ Проверка dedup фильтров (hash check) предотвращает случайные дубликаты

Связь Gathering Run ↔ Segment

Каждый запуск сбора МОЖЕТ быть связан с конкретным сегментом:

  • Предлагать недо-собранные сегменты: "AU-PH Corridor имеет 0 запусков сбора — рассмотрите начало"

Как dedup работает:

  1. GatheringRun #1 находит домен acme.com → создаёт DiscoveredCompany + company_source_link(run=1)
  2. GatheringRun #2 находит acme.com снова → находит существующий DC по домену → только создаёт новый link(run=2)
  3. SELECT COUNT(DISTINCT gathering_run_id) FROM company_source_links WHERE discovered_company_id = X = "найден 2 источниками"
  4. source_data сохраняет что каждый источник сказал об этой компании (разные employee counts, индустрии и т.д.)

TTL re-scrape: Scheduled job запрашивает WHERE is_current = true AND expires_at < now(). Когда re-scraped:

  1. Старая запись: is_current = false

Блэклист и Gates

Как система предотвращает повторный аутрич и обеспечивает человеческое одобрение на каждом платном шаге.

Блэклист — Project-Scoped Fast Lookup

Критическое решение по дизайну: Project-Scoped Blacklisting

Проблема: Компания X получает outreach от EasyStaff RU (проект 40, payroll оффер). Inxy (проект 10, crypto payments) начинает собирать и находит Company X. Должна ли она быть заблокирована?

Ответ: НЕТ. Разные проекты продают разные продукты разным ICPs. Company X может легитимно получить outreach от обоих проектов — это разные value propositions. Auto-blocking через проекты искусственно сокращает TAM.

Правило: Только кампании принадлежащие ОДНОМУ проекту вызывают auto-rejection. Кампании других проектов показываются как предупреждения (никогда не auto-rejected).

CREATE MATERIALIZED VIEW active_campaign_domains AS
SELECT DISTINCT
lower(c.domain) AS domain,
c.project_id,
p.name AS project_name,
array_agg(DISTINCT camp.id) AS campaign_ids,
array_agg(DISTINCT camp.name) AS campaign_names,
count(DISTINCT c.id) AS contact_count
FROM contacts c
JOIN projects p ON p.id = c.project_id
JOIN campaigns camp ON camp.project_id = c.project_id AND camp.status = 'active'
WHERE c.domain IS NOT NULL AND c.domain != ''
GROUP BY lower(c.domain), c.project_id, p.name;

CREATE UNIQUE INDEX ON active_campaign_domains(domain, project_id);
CREATE INDEX ON active_campaign_domains(project_id);

MCP Adapter Pattern

Базовый класс

class GatheringAdapter(ABC):
"""Base class for all gathering source adapters.

The system is source-agnostic. The DB stores opaque JSONB filters.
Each adapter owns its filter schema and execution logic.
Adding a new source = write adapter + register. Zero DB changes.
"""

source_type: str                              # "apollo.companies.emulator"
filter_model: Optional[Type[BaseModel]]       # Pydantic model (optional — can be None for fully dynamic)

@abstractmethod
async def validate(self, raw_filters: dict) -> dict:
"""Validate & normalize filters. Returns cleaned dict. Raises on invalid.
If filter_model is set, validates via Pydantic. Otherwise, passes through."""

@abstractmethod
async def estimate(self, filters: dict) -> EstimateResult:
"""Estimate cost/results without executing."""

@abstractmethod
async def execute(self, filters: dict, on_progress: Callable = None) -> GatheringResult:
"""Execute gathering. Returns companies list + metadata."""

def get_filter_schema(self) -> Optional[dict]:
"""JSON Schema for MCP tool registration. None if fully dynamic."""
if self.filter_model:
    return self.filter_model.model_json_schema()
return None

def get_capabilities(self) -> dict:
"""What this adapter can do. Used by UI and MCP for discovery."""
return {
    "source_type": self.source_type,
    "has_estimate": True,       # can estimate before executing
    "has_filter_schema": self.filter_model is not None,
    "cost_model": "free",       # free | per_result | per_page | per_credit
    "requires_auth": False,     # needs API key in integration_settings
}

Реестр адаптеров — Open for Extension

# Global registry — adapters self-register on import
ADAPTER_REGISTRY: dict[str, Type[GatheringAdapter]] = {}

def register_adapter(cls: Type[GatheringAdapter]) -> Type[GatheringAdapter]:
"""Decorator. @register_adapter on adapter class auto-registers it."""
ADAPTER_REGISTRY[cls.source_type] = cls
return cls

def get_adapter(source_type: str) -> GatheringAdapter:
cls = ADAPTER_REGISTRY.get(source_type)
if not cls:
raise ValueError(f"Unknown source: {source_type}. Available: {sorted(ADAPTER_REGISTRY.keys())}")
return cls()

def list_adapters() -> list[dict]:
"""Returns all registered adapters + capabilities. Used by /gathering/sources endpoint."""
return [get_adapter(st).get_capabilities() for st in sorted(ADAPTER_REGISTRY.keys())]

Граничные случаи

Критические сценарии, которые система должна обрабатывать — дубликаты, сбои, rate limits и проблемы целостности данных.

Критические граничные случаи и решения по дизайну

1. Company Identity Resolution (Multi-Domain)

Проблема: Одна и та же компания может иметь несколько доменов: acme.com, acme.ae, acme.co.uk. Apollo говорит "Frizzon Studios", Clay говорит "Frizzon Productions". Dedup только по домену это пропускает.

Решение: Two-layer dedup:

  • Layer 1 (instant, free): Exact domain match после нормализации (strip www, lowercase)
  • Layer 2 (fuzzy, в PRE-FILTER phase): Company name similarity (Levenshtein или trigram) + LinkedIn URL match. Если две DiscoveredCompanies share LinkedIn company URL → merge.

Реализация: discovered_companies.linkedin_company_url как secondary unique identifier. На insert, проверить оба домен И linkedin URL. Если linkedin matches existing с другим доменом → link как alias, не создавать duplicate.

2. Domain Resolution Phase

Проблема: 8,766 компаний из Companies tab имеют LinkedIn URLs но БЕЗ доменов. Не могу скрапить сайты или проверить email enrichment без доменов.

Решение: New pipeline phase RESOLVE между PRE-FILTER и SCRAPE:

  1. LinkedIn → domain: Apollo enrich_organization(linkedin_url) возвращает primary_domain (1 кредит на вызов, но высокая ценность)
  2. Company name → domain: Google поиск "{company_name}" site:{country_tld} → extract домен из первого результата
  3. Skip если unnecessary: Только запустить для компаний missing domain после GATHER
  4. Budget-gated: Оператор одобряет batch size перед трат Apollo credits

3. Re-Run Lineage

Проблема: Оператор хочет повторить поиск из 2 недель назад потому что "Apollo может иметь новые компании сейчас." Но filter_hash dedup отклонит это как duplicate.

Решение: parent_run_id на gathering_runs:

gathering_runs.parent_run_id  FK(gathering_runs) NULL — "this is a re-run of run #42"

Когда parent_run_id установлен, filter_hash dedup пропускается. Система сравнивает результаты: "Run #42 found 5,602 компаний. Re-run #65 found 5,840. Net new: 238."

4. Conversion Provenance (Full Funnel Tracking)

Проблема: "Какой запуск сбора фактически привёл к сделке, которую мы закрыли?" Не могу ответить без отслеживания цепи: gathering_run → DiscoveredCompany → ExtractedContact → Contact → Campaign → ProcessedReply → Meeting.

Решение: Уже частично решено:

  • extracted_contacts.discovered_company_id → links к DiscoveredCompany
  • contacts.provenance JSON хранит gathering_details (из pipeline_service.promote_to_crm())
  • DiscoveredCompany → company_source_links → gathering_runs

Добавить: contacts.gathering_run_id FK для прямой lineage:

# На Contact model:
gathering_run_id = Column(Integer, ForeignKey('gathering_runs.id'))

5. Central Quota Management

Проблема: Два оператора запускают Apollo поиски одновременно → оба попадают в 429 rate limit. Или MCP агент сжигает все Clay credits в одном запуске.

Решение: Adapter-level quota tracker (не БД — in-memory с Redis или простой file lock).

6. Parallel Gathering Strategy

Проблема: Для нового проекта, оператор хочет искать Apollo People + Apollo Companies + Clay TAM одновременно, затем merge результаты.

Решение: Несколько gathering_runs sharing одно и то же pipeline_run_id. Пайплайн ждёт ВСЕ runs с этим pipeline_run_id завершиться перед переходом к DEDUP.

7. Effectiveness Learning Loop

Проблема: После 10 запусков сбора, оператор хочет знать: "Какая комбинация source type + filter дала лучший целевой процент?"

Решение: Добавить вычисляемые метрики в gathering_runs (populated после ANALYZE phase): 

gathering_runs (additional columns):
├── target_rate            FLOAT     -- targets_found / new_companies_count
├── avg_analysis_confidence FLOAT    -- mean confidence of targets from this run
├── cost_per_target_usd    NUMERIC   -- total_cost_usd / targets_found
├── enrichment_hit_rate    FLOAT     -- emails_found / targets_enriched (populated after ENRICH)

8. Export Flexibility (Pluggable Outputs)

Проблема: Результаты нужно идти в разные destinations: Google Sheets (review), CSV (download), SmartLead (campaign), GetSales (LinkedIn), Clay table (further enrichment), webhook (custom integration).

Решение: Output adapters (же pattern как gathering adapters).

Шаги 0–2
ПОИСК КОМПАНИЙ — поиск компаний через Clay/Apollo используя описание Ideal Customer Profile (ICP). Удаляем дубликаты и уже сокращённые компании. Стоимость: Clay ~$0.01 за компанию, Apollo бесплатно (через UI emulator).
Шаги 3–4
ФИЛЬТРАЦИЯ И СКРЕЙП — удаляем мусор (offline бизнесы, мёртвые сайты, .gov/.edu домены). Качаем контент сайтов для анализа.
Шаг 5
КЛАССИФИКАЦИЯ — GPT анализирует сайт каждой компании и решает: это наш клиент или нет? Присваивает сегмент (platform/agency/OTHER) и confidence score.
Шаги 6–8
ПОДГОТОВКА ВЕРИФИКАЦИИ — подготовка к email-верификации (оценка стоимости). CP3: оператор одобряет расходы перед зарядом FindyMail.
Шаг 9
ЭКСПОРТ — экспорт одобренных таргет-компаний из БД. Разбиваем по сегментам, сохраняем CSV локально и в Google Sheets. Эти компании — основа для поиска людей на следующем шаге.
Шаг 10
ИМПОРТ ЛЮДЕЙ — оператор вручную ищет людей (decision-makers) в Apollo People UI используя список таргет-компаний из шага 9. Экспортирует как CSV. Этот шаг импортирует CSV: маппинг колонок Apollo → наш формат, обогащение domain/segment/social_proof, дедупликация.
Шаг 11
ПОИСК EMAIL — FindyMail ищет рабочий email по LinkedIn URL. Платим только за найденные email ($0.01 каждый). Типичный hit rate: 60–80% (из 500 LinkedIn профилей → ~350–400 верифицированных email).

ЗАГРУЗКА В SMARTLEAD — создаём кампанию, привязываем email-аккаунты, загружаем лиды, загружаем email-секвенции, устанавливаем расписание. Каждый подшаг требует подтверждения оператора. Кампания НИКОГДА не активируется автоматически.

★ CP1

ProjectConfig #

Центральный класс конфигурации. Загружается из БД при каждом запуске pipeline. Содержит: сегменты (кому мы продаём — названия, должности decision-makers, social proof по регионам), промпт классификации (GPT решает "наш клиент или нет"), email-аккаунты для отправки через SmartLead, расписание отправки (дни, время, интервал). Для изменения конфигурации — обновьте данные в БД через API, скрипт подхватит изменения при следующем запуске.

class ProjectConfig:
    def __init__(self, project_id: int):
        self.project_id = project_id
        self.project_name = ""
        self.segments = {}          # slug → данные сегмента из kb_segments
        self.prompt_id = None       # последний активный промпт из gathering_prompts
        self.prompt_text = ""       # текст промпта (фоллбэк)
        self.email_accounts = []    # ID аккаунтов SmartLead
        self.schedule = {}          # конфиг расписания SmartLead
        self.state_dir = None       # директория состояния для этого проекта
        self.csv_dir = None         # директория выходных CSV

load() — что загружается из БД

Вызывается один раз при запуске pipeline. Загружает 5 типов данных:

1 Имя проекта
GET /contacts/projects/{id} → self.project_name. Используется в названиях кампаний и CSV файлов.
2 Сегменты из kb_segments
GET /knowledge-base/segments → фильтруем по project_id и is_active. Каждый сегмент хранится как slug → {id, name, titles, seniorities, social_proof, campaign_name_template, default_filters, ...}. Сегменты определяют кому мы продаём — должности decision-makers, social proof текст по регионам, фильтры Clay поиска.
3 Последний промпт из gathering_prompts
get_latest_prompt(project_id) → (prompt_id, prompt_text). Выбирает промпт с максимальным ID среди активных. Показывает count использований и average target rate для мониторинга качества промпта по итерациям.
4 Конфиг SmartLead из project_knowledge
Два ключа: smartlead/email_accounts → список ID аккаунтов, smartlead/config → расписание (дни, таймзона, start/end час, интервал). Оба хранятся как JSON строки в поле value.
5 Директории состояния и выходных данных
state_dir = ../state/{project_slug}/ — JSON файлы состояния для восстановления после сбоя. csv_dir = output/{project_slug}/pipeline/ — CSV экспорты. Обе создаются с mkdir -p при каждом запуске.

Методы

get_segment(slug)
Возвращает конфиг сегмента по slug. Если не найден — выводит доступные сегменты и sys.exit(1). Fail-fast дизайн: лучше остановиться немедленно, чем молча использовать неправильные данные.
get_social_proof(country, slug)
Возвращает country-specific социальный proof текст для персонализации писем. Ищет segment.social_proof[country], фоллбэк на ключ _default. Пример: UK получает "trusted by 200+ UK agencies", India получает "used by Flipkart, Nykaa".
get_titles(slug)
Target должности для Apollo People поиска. Дефолты: ["CEO", "Founder", "CTO", "Head of Product"]. Кастомизируется на сегмент.
get_seniorities(slug)
Уровни seniority для Apollo фильтра поиска. Дефолты: ["owner", "founder", "c_suite", "vp", "head", "director"].
get_campaign_name(slug, contacts)

SmartLead игнорирует \n в теле письма — требует HTML-теги <br>. Парсер конвертирует все переносы: \n\n → <br><br>, затем \n → <br>.
Замена em dash
GPT часто генерирует длинные тире (—) в темах и телах писем. Некоторые email-клиенты отображают их некорректно, поэтому они заменяются на обычные дефисы (-) для максимальной совместимости.

Сгенерированные секвенции сохраняются как markdown для редактирования человеком перед следующим использованием: generated_sequence_{segment}_{date}.md. Оператор всегда проверяет перед загрузкой — никогда не загружается автоматически. GPT-4o вызов запускается через docker exec против backend контейнера OpenAI client.

Режимы ввода #

Четыре способа подать фильтры в pipeline. Фильтры определяют какие компании будут искать Clay/Apollo. Каждый режим производит одинаковый выход: {segment, filters} dict.

Режим 1: Natural (--mode natural)

Claude генерирует фильтры в диалоге с оператором, передаёт результат как --filters JSON. Свободное описание ICP → структурированный запрос. Сегмент авто-определяется из контента фильтров (ключевые слова "agency"/"agencies" → IM_FIRST_AGENCIES, иначе INFLUENCER_PLATFORMS).

python3 universal_pipeline.py --project-id 42 --mode natural \
  --input "SaaS companies in MENA with 50-500 employees" \
  --filters '{"country_names": ["UAE", "Saudi Arabia"], "max_results": 3000}'

Режим 2: Structured (--mode structured)

Использует default_filters из конфига сегмента в БД. Самый частый для повторных ранов. Требует --segment slug. Если сегмент не найден — выводит доступные slugs и выходит.

Даёте примеры хороших клиентов (домены) → система анализирует их общие черты → генерирует фильтры для поиска похожих компаний. Пример: "find companies similar to impact.com and modash.io".

api() — Общение с бэкендом

Единая функция для всех API-вызовов к бэкенду. Обработка ошибок и fail-fast при HTTP 4xx/5xx.

def api(method: str, path: str, raise_on_error: bool = True, **kwargs) -> dict:
    url = f"{BACKEND_BASE}/api{path}"
    r = getattr(httpx, method)(url, headers=BACKEND_HEADERS, timeout=300, **kwargs)
    if r.status_code >= 400:
        if raise_on_error:
            print(f"  API ERROR {r.status_code}: {r.text[:500]}")
            sys.exit(1)
        return {"_error": r.status_code, "_detail": r.text[:500]}
    return r.json()

api_long() — Устойчивые вызовы долгих операций

Обёртка для API-операций, которые занимают минуты (scrape, analyze). Проблема: скрейпинг 500 сайтов занимает 5–15 минут, анализ 500 компаний — 10–20 минут. За это время HTTP-соединение может разорваться (timeout, перезапуск бэкенда, проблемы с сетью). Но бэкенд сохраняет результаты в базу данных инкрементально — данные не теряются. Решение: если соединение упало — не паникуем, а опрашиваем статус рана каждые 30 секунд, пока фаза не продвинется.

def api_long(method: str, path: str, expected_phase: str, run_id: int,
             timeout: int = 3600, poll_interval: int = 30, **kwargs) -> dict:
    url = f"{BACKEND_BASE}/api{path}"
    try:
        r = getattr(httpx, method)(url, headers=BACKEND_HEADERS, timeout=timeout, **kwargs)
        if r.status_code >= 400:
            return {"_error": r.status_code, "_detail": r.text[:500]}
        return r.json()
    except (httpx.ReadTimeout, httpx.RemoteProtocolError,
            httpx.ConnectError, httpx.ReadError) as e:
        print(f"  Connection lost ({type(e).__name__}). Backend may still be working...")
        print(f"  Polling until phase reaches '{expected_phase}'...")

        start = time.time()
        while time.time() - start < timeout:
            time.sleep(poll_interval)
            try:
                r2 = httpx.get(f"{BACKEND_BASE}/api/pipeline/gathering/runs/{run_id}",
                               headers=BACKEND_HEADERS, timeout=30)
                if r2.status_code == 200:
                    phase = r2.json().get("current_phase", "")
                    elapsed = int(time.time() - start)
                    print(f"  [{elapsed}s] Phase: {phase}")
                    if phase == expected_phase or phase.startswith("awaiting_"):
                        print(f"  Backend finished — phase is now '{phase}'")
                        return r2.json()
            except Exception:
                print(f"  [{int(time.time()-start)}s] Backend unreachable, waiting...")

        print(f"  Timeout after {timeout}s. Check manually.")
        return {"_timeout": True}
expected_phase
Фаза, которую ран должен достичь ПОСЛЕ завершения операции. Для scrape: "scraped". Для analyze: "analyzed". Polling останавливается, когда обнаружена эта фаза (или любая awaiting_*).
Обрабатываемые ошибки
httpx.ReadTimeout, httpx.RemoteProtocolError, httpx.ConnectError, httpx.ReadError. Любая из них запускает fallback с polling вместо краша.
Таймаут
По умолчанию 3600 секунд (1 час). Если ожидаемая фаза не достигнута за это время, возвращает {"_timeout": True} — оператор должен проверить вручную.
Интервал опроса
Каждые 30 секунд по умолчанию. Каждый опрос печатает прошедшее время и текущую фазу для прозрачности.
Почему не просто retry?
Повторный POST перезапустил бы операцию на бэкенде (дублирование работы). Polling GET-эндпоинта статуса безопасен — он read-only и показывает, закончил ли бэкенд работу сам.
Используется в
step4_scrape() и step5_analyze() — обе функции заменили api() на api_long() для устойчивости к разрывам соединения при долгих операциях.

Батчинг и авто-pipeline #

Бэкенд не может надёжно обработать scrape/analyze 3000+ компаний одним запросом — соединения рвутся, память растёт, ошибки каскадируются. Система батчинга разбивает большие списки доменов на раны по 500 и проводит каждый ран через полный pipeline автоматически, без участия оператора на checkpoint'ах.

Константа BATCH_SIZE

BATCH_SIZE = 500  # Max domains per gathering run (for scrape stability)

Если список доменов ≤ 500, создаётся один ран. Если больше — список разбивается на чанки по 500, каждый становится отдельным gathering run. Все раны делят один проект и попадают в общую БД — результаты объединяются.

create_batched_runs() — Разбивка доменов на раны

def create_batched_runs(config: ProjectConfig, domains: list[str],
                        mode: str, notes_prefix: str = "") -> list[int]:
    """Create multiple gathering runs if domains > BATCH_SIZE.
    Returns list of run IDs."""
    if len(domains) <= BATCH_SIZE:
        batches = [domains]
    else:
        batches = [domains[i:i+BATCH_SIZE]
                   for i in range(0, len(domains), BATCH_SIZE)]
        print(f"\n  Splitting {len(domains)} domains into {len(batches)} batches of ~{BATCH_SIZE}")

    run_ids = []
    for i, batch in enumerate(batches):
        batch_label = f" (batch {i+1}/{len(batches)})" if len(batches) > 1 else ""
        result = api("post", "/pipeline/gathering/start", json={
            "project_id": config.project_id,
            "source_type": "manual.companies.manual",
            "filters": {"domains": batch},
            "triggered_by": "operator",
            "input_mode": mode,
            "notes": f"{notes_prefix}{batch_label} — {len(batch)} domains",
        })
        run_id = result["id"]
        run_ids.append(run_id)
        print(f"  Run #{run_id}: {len(batch)} domains{batch_label}")

    return run_ids
Source type
manual.companies.manual — домены передаются напрямую (без поиска Clay/Apollo). Используется для повторной обработки известных доменов или expand mode.
Notes
Каждый ран получает описательную заметку с номером батча и количеством доменов для лёгкой идентификации в UI.
Возвращает
Список run_id (целые числа). Вызывающий код итерирует по ним и вызывает process_run_pipeline() для каждого.

process_run_pipeline() — Полный авто-pipeline на ран

Проводит один ран через весь pipeline автоматически: gather → blacklist → prefilter → scrape → analyze → approve → blacklist. Без checkpoint'ов — все gate'ы авто-одобряются. Использует api_long() для scrape и analyze, чтобы обрабатывать разрывы соединения без краша.

def process_run_pipeline(config: ProjectConfig, run_id: int,
                         prompt_text: str = None, skip_gather_wait: bool = False):
    # 1. Ждём завершения фазы gather (poll каждые 10 сек, до 10 мин)
    if not skip_gather_wait:
        for i in range(60):
            time.sleep(10)
            phase = api("get", f"/pipeline/gathering/runs/{run_id}").get("current_phase")
            if phase != "gather": break

    # 2. Проверяем текущую фазу — пропускаем уже пройденные шаги
    phase = api("get", f"/pipeline/gathering/runs/{run_id}").get("current_phase")

    # 3. Blacklist check + авто-одобрение CP1
    if phase == "gathered":
        api("post", f"/pipeline/gathering/runs/{run_id}/blacklist-check")
        approve_pending_gate(config, run_id)

    # 4. Pre-filter (убираем мусорные домены)
    if phase == "scope_approved":
        api("post", f"/pipeline/gathering/runs/{run_id}/pre-filter")

    # 5. Scrape — устойчив через api_long()
    if phase == "filtered":
        step4_scrape(run_id)

    # 6. Analyze — устойчив через api_long(), использует prompt_text
    if phase == "scraped":
        api_long("post", f"/pipeline/gathering/runs/{run_id}/analyze",
                 expected_phase="analyzed", run_id=run_id, timeout=3600,
                 params={"prompt_text": prompt_text, "model": "gpt-4o-mini"})

    # 7. Авто-одобрение таргетов (CP2) + добавление в blacklist
    approve_pending_gate(config, run_id)
    blacklist_approved_targets(config, run_id)
Phase-aware
На каждом шаге функция проверяет текущую фазу через API. Если ран был прерван и возобновлён — продолжает с того места, где остановился, не повторяя пройденные шаги.
Авто-одобрение
Все gate'ы (CP1, CP2) авто-одобряются через approve_pending_gate(). Безопасно для батчей, потому что оператор уже одобрил общий список доменов до начала батчинга.
Устойчивость
Scrape и analyze используют api_long() — если соединение с бэкендом упадёт посреди 15-минутного скрейпинга, pipeline ждёт через polling вместо краша.
skip_gather_wait
Если True — пропускает ожидание фазы gather. Используется, когда домены переданы через manual source (мгновенная дедупликация, без задержки Clay).

approve_pending_gate() — Авто-одобрение checkpoint

def approve_pending_gate(config: ProjectConfig, run_id: int) -> bool:
    """Find and approve pending gate for this run."""
    gates = api("get",
        f"/pipeline/gathering/approval-gates?project_id={config.project_id}",
        raise_on_error=False)
    items = gates if isinstance(gates, list) else gates.get("items", [])
    for g in items:
        if g.get("gathering_run_id") == run_id and g.get("status") == "pending":
            api("post", f"/pipeline/gathering/approval-gates/{g['id']}/approve",
                json={}, raise_on_error=False)
            print(f"  Gate #{g['id']} approved")
            return True
    return False
Как работает
Запрашивает все approval gate'ы проекта, находит тот, который соответствует этому run_id со статусом "pending", и одобряет автоматически.
Отличие от approve_gate()
approve_gate(gate_id) требует известный gate ID (для интерактивного режима). approve_pending_gate(run_id) находит gate автоматически — для batch/auto режима.
Обработка ошибок
Использует raise_on_error=False и try/except — если pending gate не существует, возвращает False молча. Безопасно вызывать на любой стадии pipeline.

--project-id
ID проекта из базы данных (целое число). Обязателен для каждого запуска. Пример: --project-id 42

Выбор режима

--mode
Режим ввода: natural, structured (по умолч.), lookalike, expand. Определяет как генерируются фильтры.
--segment
Slug сегмента для structured режима. Должен совпадать со slug в kb_segments БД. Пример: --segment influencer_platforms
--input
Описание на естественном языке для natural режима. Пример: --input "SaaS companies in MENA with 50-500 employees"
--filters
JSON-фильтры для natural режима (генерирует Claude в диалоге). Пример: --filters '{"country_names": ["UAE"], "max_results": 3000}'
--examples
Домены через запятую для lookalike режима. Пример: --examples "impact.com,modash.io,grin.co"
--base-run
ID рана для клонирования в expand режиме. Пример: --base-run 198
--override
JSON-переопределения для expand режима. Мержатся поверх фильтров базового рана. Пример: --override '{"country_names": ["Singapore"]}'

Управление потоком

--from-step
Запустить pipeline с определённого шага. Пропускает всё до него. Значения: start, blacklist, prefilter, scrape, analyze, verify, export, people, findymail, upload
--run-id
Возобновить существующий gathering run по ID. Если не указан — загружается из run_state.json.
--dry-run
Режим превью — печатает фильтры, сегмент, страны и шаги, которые выполнились бы. Никаких API-вызовов.
--force
Игнорировать кэш-файлы (targets.json, contacts.json, enriched.json). Принудительное выполнение каждого шага заново.

Импорт данных

--apollo-csv
Путь к CSV-экспорту Apollo People. Обязателен для --from-step people.
--apollo-csv-agencies
Второй CSV для сегмента agencies. Двух-сегментный импорт в одном запуске — platforms из --apollo-csv, agencies из этого флага. Оба мержатся в один список контактов.
--max-findymail
Максимум контактов для обогащения через FindyMail (по умолч.: 1500). Защитный лимит для контроля расходов.

Продвинутые

--prompt-file
Кастомный промпт классификации из текстового файла. Перезаписывает промпт из БД. Для тестирования новых промптов.
--re-analyze
Перезапустить GPT-классификацию на существующем ране без повторного сбора. Требует --run-id. Использует последний промпт из БД или --prompt-file.

Массив STEPS и логика резюме

Pipeline определяет упорядоченную последовательность шагов. --from-step обрезает массив, определяя какие шаги выполнять:

STEPS = ["start", "blacklist", "prefilter", "scrape", "analyze", "verify",
         "export", "people", "findymail", "upload"]

# --from-step people → выполняет: ["people", "findymail", "upload"]
# --from-step export → выполняет: ["export", "people", "findymail", "upload"]

Типичные паттерны возобновления после checkpoint'ов:

# После CP1 (блэклист одобрен через UI):
python3 universal_pipeline.py --project-id 42 --from-step prefilter --run-id 205

# После CP2 (таргеты одобрены через UI):
python3 universal_pipeline.py --project-id 42 --from-step verify --run-id 205

# После CP3 (стоимость одобрена) — перейти к экспорту:
python3 universal_pipeline.py --project-id 42 --from-step export --run-id 205

# Импорт людей + обогащение + загрузка (самый частый resume point):
python3 universal_pipeline.py --project-id 42 --from-step people \
  --apollo-csv platforms.csv --apollo-csv-agencies agencies.csv

# Переанализ с новым промптом, затем продолжить:
python3 universal_pipeline.py --project-id 42 --re-analyze --run-id 205 \
  --prompt-file new_prompt_v9.txt
