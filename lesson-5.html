Lesson 5

CRM Blacklist & Approval Gates

Learn how the system prevents duplicate outreach and controls spending before credit-consuming operations.

The Scenario

Your operator is running a gathering for Project 10 (Inxy, crypto payments). They've found 134 domains. 45 are in their own campaigns. But 89 appear in EasyStaff RU's active campaigns (Project 40, payroll offering). Should the Inxy gathering auto-reject all 89 to avoid "contacting the same companies"?

The Critical Design Decision

Answer: No, absolutely not. Those 89 domains appear in EasyStaff's campaigns, not Inxy's. They represent different value propositions being sold to potentially different buyer personas within the same company.

Cross-Project Rule: Only same-project campaigns trigger auto-rejection. Other projects' campaigns are shown as warnings. This preserves TAM across projects and respects the autonomy of parallel business lines.

Blacklist Check Layers (in order)

The system checks domains against four layers, in strict sequence. Each layer has different scope and action.

# Layer Scope Action What it catches
1 Project blacklist Per-project Auto-reject project_blacklist table — manual bans by operator
2 Same-project campaigns Per-project Auto-reject Contacts with domains in THIS project's active campaigns
3 Enterprise blacklist Global Auto-reject enterprise_blacklist.json — competitors, banned orgs
4 Cross-project campaigns All projects Warning only Domains in OTHER projects' campaigns (informational)

When a blacklist check completes, the system returns two arrays: rejected domains (with reasons) and warning domains (informational only). Here's the exact JSON structure:

Edge Cases

These six scenarios show how the system behaves in tricky situations:

Scenario Behavior
Domain in project A campaigns, gathering for project B Warning (not rejected) — different value prop
Domain in project A campaigns, gathering for project A Rejected — already in outreach
Domain in enterprise blacklist Rejected regardless of project
Domain in project blacklist for project A, gathering for project B Not rejected — blacklist is per-project
Campaign wrongly assigned to project Operator sees campaign names in detail → can fix mapping
Contact has NULL project_id Ignored — unassigned contacts don't trigger blacklist

Materialized View: Fast Domain Lookups

The system maintains a project-scoped materialized view of all active campaign domains. This enables sub-millisecond checks during gathering:

CREATE MATERIALIZED VIEW active_campaign_domains (SQL) 
CREATE MATERIALIZED VIEW active_campaign_domains AS
SELECT DISTINCT
  lower(c.domain) AS domain,
  c.project_id,
  p.name AS project_name,
  array_agg(DISTINCT camp.id) AS campaign_ids,
  array_agg(DISTINCT camp.name) AS campaign_names,
  count(DISTINCT c.id) AS contact_count
FROM contacts c
JOIN projects p ON p.id = c.project_id
JOIN campaigns camp ON camp.project_id = c.project_id
  AND camp.status = 'active'
WHERE c.domain IS NOT NULL AND c.domain != ''
GROUP BY lower(c.domain), c.project_id, p.name;

CREATE UNIQUE INDEX ON active_campaign_domains(domain, project_id);
CREATE INDEX ON active_campaign_domains(project_id);

Query patterns:

  • Same-project check: WHERE domain = ANY(:domains) AND project_id = :pid
  • Cross-project check: WHERE domain = ANY(:domains) AND project_id != :pid

Refresh strategy: REFRESH MATERIALIZED VIEW CONCURRENTLY after campaign sync in crm_scheduler.py and after any campaign push. No locks needed.

Also Check: Previously Discovered Companies

The system also prevents re-processing companies from earlier gathering runs in the same project:

SELECT query for discovered_companies (SQL)
-- Companies already in the system for THIS project (not globally) SELECT DISTINCT domain FROM discovered_companies WHERE domain = ANY(:domains) AND project_id = :pid AND status != 'REJECTED'

This prevents re-processing companies from previous gathering runs within the same project, saving time and avoiding duplicate analysis.

Approval Gates: Operator Checkpoints

The pipeline pauses before any credit-spending operation. The operator reviews the scope, estimated cost, and list of companies, then approves or rejects the gate.

Gate Types

Five types of gates protect different pipeline phases:

Gate Type When it triggers Cost
pre_scrape_crona Before batch scraping via Crona API Crona credits
pre_analysis Before AI analysis (GPT/Gemini batch) LLM tokens
pre_enrichment Before Apollo enrichment batch 1 credit/person
pre_verification Before FindyMail email verification FindyMail credits
pre_push Before SmartLead/GetSales push None (final review)

Free scraping: httpx scraping is free — no gate needed. Only credit-spending steps get gates.

Gate Lifecycle (8 Steps)

  1. Pipeline creates gate: status = 'pending', created_at = now()
  2. Database stores: Full scope (company IDs, estimated cost, credit usage)
  3. Operator sees alert: Dashboard shows pending gates, grouped by project and type
  4. Operator reviews details: Company names, estimated costs, reasoning
  5. Operator approves or rejects: POST /api/approval-gates/{id}/approve or /reject
  6. Pipeline resumes: If approved, continue to the gated step. If rejected, mark companies as 'not_approved' and stop.
  7. Auto-expiry: If not decided after 48 hours, gate automatically expires (optional, per config)
  8. Audit trail: All decisions logged: decided_by, decided_at, decision_note

Crash Safety: No Lost Work

If the system crashes after a gate is approved but before the gated operation completes, the pipeline automatically resumes. The gate record itself is immutable — only the dependent work (enrichment, verification) retries.

Full Example: BLACKLIST → pre_enrichment Gate

Here's a 10-step flow showing exactly what happens when an operator gathers 200 companies, the system filters them, and creates an enrichment gate:

  1. Gathering completes: 200 unique domains discovered (found by Apollo People, source_type = "apollo.people.emulator")
  2. Blacklist check runs:
    • 45 domains rejected (same-project campaigns)
    • 12 domains rejected (project blacklist)
    • 8 domains rejected (enterprise blacklist)
    • 89 domains returned as cross-project warnings (not rejected)
    • Net approved: 135 domains
  3. Scraping happens: 135 approved domains are scraped (free via httpx)
  4. Scraping result: 128 sites fully scraped, 7 unreachable
  5. Pre-filter runs: Deterministic checks (no AI)
    • Removes 8 domains (offline industries, parked sites, junk domains)
    • Net after pre-filter: 120 companies
  6. Analysis phase: 120 companies sent to GPT-4o-mini for classification
  7. Analysis complete: 45 marked as targets, 75 as non-targets
  8. Pre-enrichment gate created:
    • Scope: 45 target company IDs
    • Estimated cost: 45 Apollo credits (~$0.90)
    • Estimated people per company: 2.3 (avg)
    • Total estimated enrichment: ~103 people contacts
    • Label: "Approve 45 companies for Apollo enrichment (~$0.90)"
  9. Operator approves gate: decision = approved, decided_by = "alice@company.com"
  10. Enrichment executes: Pipeline continues, calls enrich_apollo_batch(45 companies). 103 contact records created in extracted_contacts table. Status progresses to next phase.
Key Takeaway

Cross-project autonomy + project-scoped safeguards. The blacklist system respects that different projects sell different products to potentially different decision-makers at the same company. Warnings keep operators informed; rejections prevent duplicate outreach. Approval gates ensure operators stay in control of spending.
