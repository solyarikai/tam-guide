LESSON 4

Source-Agnostic Architecture & Adapters

How to add a new gathering source without touching the database — the Open/Closed Principle in action.

What you'll learn
  • Why we reject typed database models per source (combinatorial explosion)
  • How the adapter pattern decouples sources from the database
  • Adapter-level validation with Pydantic and extra='allow'
  • The complete adapter registry and how to add a new source
  • All 15 current adapters and their filter fields
  • How clay adapters handle the 5,000 limit internally
  • Adding Indeed or any new source tomorrow (zero DB changes)
Real-World Scenario

Your VP says: "Add Indeed as a source — find all Python developers in Austin. Don't touch the database."

You smile, write IndeedJobsAdapter, register it, deploy. Done. The DB never changes. The pipeline discovers it automatically.

1. The Problem: Why Not Typed Models Per Source?

Your instinct might be: "Let's create a Pydantic model for each source in the database. Apollo filters here, Clay filters there."

Reality check:

  • Apollo alone: People UI, Companies UI, People API, Org API — 4 variants with different filters
  • Clay: Companies UI, Companies API, People UI, People API — 4 more
  • Other platforms: Sales Navigator, Google Maps, LinkedIn, Crunchbase, Indeed, Google Sheets, CSV, Manual, Google SERP — 9 more
  • Filter explosion: Each platform adds new filters every month. Model updates = schema migrations = downtime
  • Combinatorial nightmare: platforms × targets × methods = too many models to maintain

Solution: Source-agnostic database + adapter pattern.

The Three Dimensions of Any Source

Every gathering source can be described by three orthogonal dimensions:

Dimension Column Examples
Platform source_platform apollo, clay, sales_navigator, google_maps, crunchbase, csv, google_sheets, manual, google
Target source_target companies, people, jobs
Method source_method api, emulator, manual

The database combines these into: source_type = "{platform}.{target}.{method}" — e.g. apollo.companies.emulator, clay.people.api, indeed.jobs.api.

But here's the key: source_type is just a free VARCHAR. The database never parses it. It has no opinion about what sources exist. The adapter owns all the semantics.

Tomorrow: Add indeed.jobs.api to the registry. Next week: linkedin_recruiter.people.api. The DB schema? Unchanged.

2. Filter Design — The Database Doesn't Care

The gathering_runs table has one column for filters:

filters JSONB NOT NULL

That's it. The database stores whatever the adapter puts in. It never queries individual filter fields. It uses filters for:

  1. Store — write once when gathering run starts
  2. Display — read and show to operator in UI
  3. Hash — SHA256 for dedup detection (same filters = same hash)
  4. Re-execute — pass back to adapter unchanged

There is no Pydantic validation at the database level. Validation is the adapter's job, not the schema's.

3. Adapter-Level Validation — extra='allow'

Each adapter defines its OWN Pydantic model. This is adapter code, not schema code. And every adapter's filter model has one critical setting: 

class Config:
    extra = "allow"

Why? When Apollo adds a new filter next week, users pass it through. The adapter doesn't yet know about it, but the field is preserved in the JSONB. When you update the adapter code to handle it, old records load fine. No breakage.

This is the secret to future-proofing filter schemas without database migrations.

💡

Validation always lives in adapter code. Unknown fields pass through. When a new filter is needed, the adapter implements it — the DB is already ready because JSONB is schemaless.

4. The GatheringAdapter Base Class

All adapters inherit from GatheringAdapter, an abstract base class that defines the contract:

Full GatheringAdapter Python Code 
from abc import ABC, abstractmethod
from typing import Optional, Type, Callable
from pydantic import BaseModel

class EstimateResult:
    """Result of estimating a gathering before execution."""
    def __init__(self, estimated_companies: int, estimated_credits: float, cost_usd: float):
        self.estimated_companies = estimated_companies
        self.estimated_credits = estimated_credits
        self.cost_usd = cost_usd

class GatheringResult:
    """Result of executing a gathering."""
    def __init__(self, companies: list, metadata: dict):
        self.companies = companies
        self.metadata = metadata

class GatheringAdapter(ABC):
    """Base class for all gathering source adapters.

    The system is source-agnostic. The DB stores opaque JSONB filters.
    Each adapter owns its filter schema and execution logic.
    Adding a new source = write adapter + register. Zero DB changes.
    """

    source_type: str = None                            # "apollo.companies.emulator"
    filter_model: Optional[Type[BaseModel]] = None     # Pydantic model (optional)

    @abstractmethod
    async def validate(self, raw_filters: dict) -> dict:
        """Validate & normalize filters. Returns cleaned dict. Raises on invalid.

        If filter_model is set, validates via Pydantic. Otherwise, passes through.
        All filter models MUST have Config.extra = 'allow' for forward compat.

        Args:
            raw_filters: Dictionary of raw filter input from user/API

        Returns:
            Cleaned and normalized filter dictionary

        Raises:
            ValueError: If filters are invalid for this adapter
        """
        pass

    @abstractmethod
    async def estimate(self, filters: dict) -> EstimateResult:
        """Estimate cost/results without executing.

        Used to show user the approximate number of results and cost before running.
        For API adapters, may call API with dry_run flag. For UI adapters, may
        load the page and count visible results without scrolling.

        Args:
            filters: Validated filter dictionary

        Returns:
            EstimateResult with estimated_companies, estimated_credits, cost_usd
        """
        pass

    @abstractmethod
    async def execute(self, filters: dict, on_progress: Callable = None) -> GatheringResult:
        """Execute gathering. Returns companies list + metadata.

        Handles pagination, rate limiting, retries internally.
        Yields progress updates via on_progress callback.
        Normalizes all results to standard company dict format.

        Args:
            filters: Validated filter dictionary
            on_progress: Optional callback fn(current, total, message) for progress tracking

        Returns:
            GatheringResult with companies list and metadata
        """
        pass

    def get_filter_schema(self) -> Optional[dict]:
        """JSON Schema for MCP tool registration. None if fully dynamic.

        Returns the Pydantic model's JSON schema for use in API docs
        and MCP tool definitions. If this adapter doesn't have a strict
        filter schema, returns None.

        Returns:
            JSON Schema dict or None
        """
        if self.filter_model:
            return self.filter_model.model_json_schema()
        return None

    def get_capabilities(self) -> dict:
        """What this adapter can do. Used by UI and MCP for discovery.

        Returns a dictionary describing the adapter's capabilities,
        used by the UI to show which adapters are available and what
        features they support.

        Returns:
            Dictionary with keys:
                - source_type: str (e.g. "apollo.companies.emulator")
                - has_estimate: bool (can estimate before executing)
                - has_filter_schema: bool (has formal filter schema)
                - cost_model: str (free, per_result, per_page, per_credit)
                - requires_auth: bool (needs API key in integration_settings)
        """
        return {
            "source_type": self.source_type,
            "has_estimate": True,
            "has_filter_schema": self.filter_model is not None,
            "cost_model": "free",         # free | per_result | per_page | per_credit
            "requires_auth": False,       # needs API key in integration_settings
        }

The base class defines four key methods:

  • validate(raw_filters): Pydantic validation + normalization. All models use extra='allow'.
  • estimate(filters): Dry-run to show user the cost before executing.
  • execute(filters, on_progress): Actually gather the data. Handles pagination, auth, retries internally.
  • get_filter_schema(): Returns JSON Schema for API docs.
  • get_capabilities(): Metadata for UI discovery.

5. The Adapter Registry

A global registry holds all registered adapters. Adapters self-register via a decorator.

Full Adapter Registry Python Code 
from typing import Type, List

# Global registry — adapters self-register on import
ADAPTER_REGISTRY: dict[str, Type[GatheringAdapter]] = {}

def register_adapter(cls: Type[GatheringAdapter]) -> Type[GatheringAdapter]:
    """Decorator. @register_adapter on adapter class auto-registers it.

    Usage:
        @register_adapter
        class MyAdapter(GatheringAdapter):
            source_type = "my_platform.companies.api"
            ...

    Args:
        cls: The adapter class to register

    Returns:
        The adapter class (for use as a decorator)
    """
    if not cls.source_type:
        raise ValueError(f"Adapter {cls.__name__} must define source_type")
    ADAPTER_REGISTRY[cls.source_type] = cls
    return cls

def get_adapter(source_type: str) -> GatheringAdapter:
    """Get an adapter instance by source_type.

    Args:
        source_type: e.g. "apollo.companies.emulator"

    Returns:
        An instance of the adapter

    Raises:
        ValueError: If source_type not in registry
    """
    cls = ADAPTER_REGISTRY.get(source_type)
    if not cls:
        available = sorted(ADAPTER_REGISTRY.keys())
        raise ValueError(
            f"Unknown source: {source_type}. Available: {available}"
        )
    return cls()

def list_adapters() -> List[dict]:
    """Returns all registered adapters + capabilities.

    Used by /gathering/sources endpoint and MCP tool discovery.
    Returns capabilities for each registered adapter.

    Returns:
        List of capability dicts, one per adapter
    """
    return [
        get_adapter(st).get_capabilities()
        for st in sorted(ADAPTER_REGISTRY.keys())
    ]

That's it. When the backend starts, all adapters import and auto-register. The API endpoint /gathering/sources returns the list dynamically. The UI discovers new adapters without restart.

7. Adding a New Source — Zero Database Changes

Let's say you need to add Indeed as a job source. Here's the complete code:

Example: IndeedJobsAdapter Full Implementation 
# backend/app/services/gathering_adapters/indeed_jobs.py

from pydantic import BaseModel, Field
from typing import Optional
from .base import GatheringAdapter, EstimateResult, GatheringResult, register_adapter

class IndeedJobsFilters(BaseModel):
    """Filter model for Indeed job search."""

    job_title: str = Field(..., description="Job title to search for")
    location: str = Field(..., description="Location (city, state, or postal code)")
    radius_miles: Optional[int] = Field(default=25, description="Search radius")

    company_exclude: Optional[list[str]] = Field(default=[], description="Companies to exclude")
    job_type: Optional[list[str]] = Field(default=[], description="Full-time, part-time, contract, temporary")
    salary_min: Optional[int] = Field(default=None, description="Minimum salary in USD")
    date_posted: Optional[str] = Field(default="anytime", description="Last 7 days, last 30 days, anytime")

    max_results: Optional[int] = Field(default=500, description="Max jobs to retrieve")

    class Config:
        extra = "allow"  # Future-proof: unknown fields pass through

@register_adapter
class IndeedJobsAdapter(GatheringAdapter):
    """Adapter for Indeed job search API."""

    source_type = "indeed.jobs.api"
    filter_model = IndeedJobsFilters

    async def validate(self, raw_filters: dict) -> dict:
        """Validate filters via Pydantic."""
        validated = IndeedJobsFilters(**raw_filters)
        return validated.model_dump()

    async def estimate(self, filters: dict) -> EstimateResult:
        """Estimate job count via Indeed API dry-run."""
        # Call Indeed API with dry_run=true
        # Parse response to get estimated count
        # Assume 0 cost (Indeed doesn't charge per query)
        import httpx

        async with httpx.AsyncClient() as client:
            response = await client.get(
                "https://api.indeed.com/v2/jobs/search",
                params={
                    "query": filters["job_title"],
                    "location": filters["location"],
                    "radius": filters.get("radius_miles", 25),
                    "sort": "date",
                    "count": 0  # dry-run, no results
                },
                headers={"Authorization": f"Bearer {self.indeed_api_key}"}
            )
            data = response.json()
            estimated_jobs = data.get("totalResults", 100)

        return EstimateResult(
            estimated_companies=estimated_jobs,
            estimated_credits=0,
            cost_usd=0
        )

    async def execute(self, filters: dict, on_progress=None) -> GatheringResult:
        """Execute Indeed job search and normalize results."""
        import httpx

        companies = []
        max_results = filters.get("max_results", 500)
        retrieved = 0
        page = 0

        async with httpx.AsyncClient() as client:
            while retrieved < max_results:
                response = await client.get(
                    "https://api.indeed.com/v2/jobs/search",
                    params={
                        "query": filters["job_title"],
                        "location": filters["location"],
                        "radius": filters.get("radius_miles", 25),
                        "offset": page * 25,
                        "count": 25,
                        "sort": "date"
                    },
                    headers={"Authorization": f"Bearer {self.indeed_api_key}"}
                )

                data = response.json()
                jobs = data.get("results", [])

                if not jobs:
                    break

                for job in jobs:
                    if retrieved >= max_results:
                        break

                    # Normalize to company dict
                    company = {
                        "domain": self._extract_domain(job.get("company_url", "")),
                        "name": job.get("company_name"),
                        "industry": None,
                        "employees": None,
                        "country": None,
                        "city": filters.get("location"),
                        "source_data": {
                            "job_id": job.get("job_id"),
                            "job_title": job.get("job_title"),
                            "salary": job.get("salary"),
                            "url": job.get("url"),
                        }
                    }

                    if company["domain"] and company["name"]:
                        companies.append(company)
                        retrieved += 1

                        if on_progress:
                            on_progress(retrieved, max_results, f"Fetched {retrieved} jobs")

                page += 1

        return GatheringResult(
            companies=companies,
            metadata={
                "total_fetched": retrieved,
                "method": "indeed_jobs_api",
                "filters_applied": filters
            }
        )

    def _extract_domain(self, url: str) -> Optional[str]:
        """Extract domain from company URL."""
        if not url:
            return None
        from urllib.parse import urlparse
        parsed = urlparse(url)
        return parsed.netloc if parsed.netloc else None

    def get_capabilities(self) -> dict:
        return {
            "source_type": self.source_type,
            "has_estimate": True,
            "has_filter_schema": True,
            "cost_model": "free",
            "requires_auth": True,  # Needs Indeed API key
        }

That's it. Three simple steps to add Indeed or any new source:

  1. Write the adapter class with Pydantic filter model
  2. Decorate with @register_adapter
  3. Deploy. The system discovers it automatically.

The database? Zero changes. The API? Auto-updated. The UI? Discovers it on refresh. New adapters work with the existing pipeline, approval gates, and dedup logic without touching core code.

8. Adapter Internals — Transparent to Pipeline

Adapters handle complexity internally, invisible to the rest of the system:

Clay 5,000 Limit

When Clay's estimated result count exceeds 5,000, the adapter automatically splits by geography. It creates multiple gathering_runs under the same pipeline_run_id, each with a _geo_split_label in the filters. The rest of the system sees one request; the adapter handles the splits seamlessly.

Apollo Pagination

Apollo's UI and API adapters handle page iteration internally. The adapter loops through pages, aggregates results, and returns them as one normalized list. The pipeline never sees pagination logic.

Rate Limiting

Each adapter implements rate limiting internally:

  • Apollo: 0.3 seconds per request (conservative)
  • Clay: Respects export timing constraints
  • Sales Navigator: Respects LinkedIn's rate limits
  • Google APIs: Handles quota exhaustion gracefully

Authentication

API adapters read authentication keys from integration_settings (user's stored API keys). UI adapters (Puppeteer-based) handle login directly with stored credentials.

Retry & Backoff

All adapters implement exponential backoff for 429 (rate limit), 503 (service unavailable), and transient errors. Timeouts are handled gracefully, failing fast if a source is down.

Key Takeaway

The adapter pattern decouples source implementations from the database schema. New sources are added as code (adapters) + registration, not database migrations. The extra='allow' pattern in Pydantic ensures filters can evolve without breaking old records. This is the Open/Closed Principle in action: open for new sources, closed for database changes.
