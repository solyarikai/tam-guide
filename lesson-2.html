Lesson 2

Project Context & Operator Input Modes

Learn what the system knows about your project, the four ways operators can request a gathering run, and how the system connects runs to segments for tracking and intelligence.

Scenario

An operator opens the TAM Gathering System, wants to find 5000 digital agencies in the UAE, and starts typing a request. The system instantly knows about the project's ICP, competitors, case studies, existing contacts, and past filtering attempts. It uses all that context to map the operator's simple request into precise source filters — without the operator having to understand Apollo's seniority codes or Clay's company size ranges.

What You'll Learn
  • The 10 data sources the system reads from each project
  • How ICP, segments, and case studies inform gathering decisions
  • Four operator input modes: Natural Language, Structured Filters, Lookalike, Expand/Repeat
  • How gathering runs link to segments for TAM tracking
  • The end-to-end flow when an operator gives natural language input

What The System Knows About Each Project

The gathering system doesn't operate in a vacuum. It reads the project's existing knowledge to make smart decisions.

Project Context (already exists in DB — reuse, don't rebuild)

Source Table/Field How Gathering Uses It
ICP Definition projects.target_segments, projects.target_industries → AI maps to source-specific filters. "Agencies <50 emp in UAE" becomes Apollo location + seniority + size filters
Segments kb_segments (flexible data JSON per segment) → Each segment can have its OWN gathering configuration. Segment "Dubai Agencies" → apollo.companies.emulator with UAE filters. Segment "AU-PH Corridor" → clay.people.emulator with Filipino language filters
Products kb_products (what we sell) → AI analysis prompt: "Does this company need {product}?"
Competitors kb_competitors (domains, strengths, weaknesses) → Auto-blacklist competitor domains. Don't gather companies already using competitor
Case Studies kb_case_studies (client industry, size, challenge, solution) → Lookalike search: "Find companies similar to these customers"
Company Profile kb_company_profile (who WE are) → AI analysis context: "We are {company}, we offer {products}"
Project Knowledge project_knowledge (category=icp/outreach/contacts/gtm) → Rich ICP context, outreach rules, GTM strategy
Existing Contacts contacts + campaigns (CRM) → Blacklist: don't re-target companies already in outreach
Enterprise Blacklist project_blacklist + enterprise_blacklist.json → Hard exclusion before any processing
Past Gathering Runs gathering_runs (this system) → "Don't repeat filters that yielded <5% target rate"

Operator Input Modes

When an operator (or MCP agent) starts a gathering run, they can provide input in several ways:

Mode 1: Natural Language (easiest, MCP-optimized)

Example: "Find 5000 digital agencies in UAE with <100 employees"

ℹ️

→ AI maps to filters using project's ICP + segment context
→ Suggests best source (Apollo for broad, Clay for specific)
→ Returns estimated results + cost

Mode 2: Structured Filters (experienced operators, precise)

Example:

Show JSON 
{
  "source_type": "apollo.people.emulator",
  "filters": {
    "person_locations": ["Dubai, United Arab Emirates"],
    "person_seniorities": ["founder", "c_suite"],
    "organization_num_employees_ranges": ["1,10", "11,50", "51,100"]
  }
}
ℹ️

→ Bypasses AI mapping, direct execution

Mode 3: Lookalike (from case studies or existing customers)

Example: "Find companies similar to: frizzon.ae, 10xbrand.com, zopreneurs.com"

ℹ️

→ Reverse-engineer patterns (industry, size, location, tech stack)
→ Generate filters that match the pattern
→ Already partially exists: reverse_engineering_service.py

Mode 4: Expand/Repeat (iterate on previous runs)

Example: "Run the same search as gathering_run #42 but for Singapore instead of Dubai"

ℹ️

→ Copy filters, modify location
→ Filter dedup (hash check) prevents accidental duplicates

Gathering Run ↔ Segment Link

Each gathering run CAN be linked to a specific segment:

Show code 
gathering_runs.segment_id  FK(kb_segments) NULL

This lets the system:

  • Track which segments have been gathered (and which haven't)
  • Calculate segment-level TAM: "Dubai Agencies segment: 5,150 companies found, 1,200 analyzed, 340 targets, 89 enriched"
  • Suggest under-gathered segments: "AU-PH Corridor has 0 gathering runs — consider starting"

How It All Connects: Natural Language to Filters

When an operator gives natural language input, here's the 10-step journey:

  1. Operator speaks: "Find 5000 digital agencies in UAE with <100 employees"
  2. System loads project context: Fetches ICP, segments, products, competitors, case studies from KB
  3. AI maps intent to dimensions: "agencies" = industry tag, "UAE" = location, "<100 employees" = size bucket
  4. AI chooses best source: Looks at source_label + historical success rate. Apollo's "companies.emulator" → good for broad searches. Clay's "people.api" → better for founder-specific targeting. Picks Apollo.
  5. AI generates canonical filters: Reads Apollo's filter schema from source_config, translates "digital agencies in UAE" to apollo-specific filters: {"company_industries": ["software", "digital_marketing"], "company_locations": ["Dubai, UAE"], "organization_num_employees_ranges": ["1,10", "11,50", "51,100"]}
  6. System checks filter_hash: Computes SHA256 of sorted canonical JSON. Checks if this filter combination has ever run before. If yes, shows last run's stats: "Similar filters found 3,847 companies last week; 412 passed analysis (10.7% target rate)"
  7. System estimates cost + results: "Estimated 4,200 results, Apollo will cost ~$210 (at $0.05/company), Clay would cost ~$840. Use Apollo."
  8. System shows preview: "Ready to run? Gathering run will: source=apollo.companies.emulator, filters={...}, estimated result 4,200, cost $210. Approve?"
  9. Operator approves. System creates gathering_runs record with status=pending, filters=JSON, filter_hash=SHA256, segment_id=(optional)
  10. Execute and track: Gathers raw results, dedups against discovered_companies, blacklists, and progresses through the pipeline — all using the filters stored in the gathering_run record.
Key Takeaway

Project context isn't just decorative — it's the engine behind smart defaults. The system reads 10 data sources (ICP, segments, products, competitors, case studies, company profile, project knowledge, existing contacts, blacklists, past runs), and uses them to map operator intent into precise, source-specific filters. Operators can request in 4 ways (natural language, structured filters, lookalike, expand/repeat), and each gathering run can be linked to a segment for TAM tracking. This design makes the system both operator-friendly and deterministic: the filters are always explicit, always stored, and always reproducible.
