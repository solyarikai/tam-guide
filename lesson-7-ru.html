8 критических граничных случаев

1. Разрешение идентификации компании (мультидомен)

Проблема: Одна компания может иметь несколько доменов: acme.com, acme.ae, acme.co.uk. Apollo говорит "Frizzon Studios", Clay говорит "Frizzon Productions". Дедупликация по домену не ловит эти случаи.

Решение: Двухуровневая дедупликация:

  • Уровень 1 (мгновенный, бесплатный): Точное совпадение домена после нормализации (удалить www, нижний регистр)
  • Уровень 2 (нечёткий, в фазе PRE-FILTER): Сходство названия компании (Levenshtein или trigram) + сопоставление LinkedIn URL. Если две DiscoveredCompanies совместно используют LinkedIn company URL → объединить.

Реализация: discovered_companies.linkedin_company_url как вторичный уникальный идентификатор. При вставке проверить оба домена И linkedin URL. Если linkedin совпадает с существующей с другим доменом → связать как alias, не создавать дублик.

company_aliases схема таблицы ▼ 
-- Опционально: таблица aliases компаний для мультидоменных сущностей
company_aliases
├── primary_company_id    FK(discovered_companies)
├── alias_domain          VARCHAR(255)
├── alias_linkedin_url    TEXT
├── alias_name            VARCHAR(500)
├── detected_by           VARCHAR(50)   -- domain_match | linkedin_match | name_fuzzy | manual

2. Фаза разрешения доменов

Проблема: 8 766 компаний из вкладки Companies имеют LinkedIn URLs, но НЕ имеют доменов. Невозможно скрепить веб-сайты или проверить email enrichment без доменов.

Решение: Новая фаза конвейера RESOLVE между PRE-FILTER и SCRAPE:

  1. LinkedIn → домен: Apollo enrich_organization(linkedin_url) возвращает primary_domain (1 credit за вызов, но высокое значение)
  2. Имя компании → домен: Google search "{company_name}" site:{country_tld} → извлечь домен из первого результата
  3. Пропустить, если необязательно: Запускать только для компаний, у которых отсутствует домен после GATHER
  4. Бюджет-ограничено: Оператор утверждает размер партии перед потратой Apollo credits

3. Lineage повторного запуска

Проблема: Оператор хочет повторно запустить поиск с 2 недель назад, потому что "Apollo может иметь новые компании сейчас." Но filter_hash дедупликация отклонит это как дублик.

Решение: parent_run_id на gathering_runs:

parent_run_id схема и логика ▼ 
gathering_runs.parent_run_id  FK(gathering_runs) NULL  -- "это повторный запуск run #42"

Когда parent_run_id установлен, дедупликация filter_hash пропускается. Система сравнивает результаты: "Run #42 нашла 5 602 компании. Re-run #65 нашла 5 840. Net new: 238."

4. Provenance конверсии (полное отслеживание воронки)

Проблема: "Какой gathering run действительно привёл к сделке, которую мы закрыли?" Невозможно ответить без отслеживания цепи: gathering_run → DiscoveredCompany → ExtractedContact → Contact → Campaign → ProcessedReply → Meeting.

Решение: Частично решено. Добавить: contacts.gathering_run_id FK для прямой lineage.

Запрос полного отслеживания воронки ▼ 
SELECT gr.source_type, gr.filters->>'person_locations' AS location,
  COUNT(DISTINCT dc.id) AS companies, COUNT(DISTINCT ec.id) AS contacts,
  COUNT(DISTINCT c.id) AS crm_contacts, COUNT(DISTINCT m.id) AS meetings
FROM gathering_runs gr
JOIN company_source_links csl ON csl.gathering_run_id = gr.id
JOIN discovered_companies dc ON dc.id = csl.discovered_company_id
LEFT JOIN extracted_contacts ec ON ec.discovered_company_id = dc.id
LEFT JOIN contacts c ON c.gathering_run_id = gr.id
LEFT JOIN meetings m ON m.contact_id = c.id
WHERE gr.project_id = :project_id
GROUP BY gr.id
ORDER BY meetings DESC;

5. Центральное управление квотой

Проблема: Два оператора одновременно запускают Apollo поиск → оба попадают на лимит 429. Или MCP agent израсходует все Clay credits в одном запуске.

Решение: Трекер квоты уровня адаптера (не БД — в памяти с Redis или простой file lock).

Класс QuotaManager (Python) ▼ 
class QuotaManager:
    """Центральный трекер квоты за платформу источника."""

    async def acquire(self, platform: str, credits: int, timeout: int = 300) -> bool:
        """Запросить credits. Блокирует до доступности или timeout."""

    async def release(self, platform: str, credits: int) -> None:
        """Вернуть неиспользованные credits (например, поиск вернул меньше результатов, чем ожидалось)."""

    async def get_usage(self, platform: str) -> QuotaUsage:
        """Текущее использование: {used_today, limit_today, used_this_hour, limit_this_hour}."""

Каждый адаптер вызывает quota_manager.acquire() перед вызовами API. Менеджер уважает лимиты для каждой платформы (Apollo 200/min, Clay 5000 exports/day, и т.д.).

6. Стратегия параллельного сбора

Проблема: Для нового проекта оператор хочет одновременно искать Apollo People + Apollo Companies + Clay TAM, затем объединить результаты.

Решение: Несколько gathering_runs совместно используют одно и то же pipeline_run_id. Конвейер ждёт, пока ВСЕ запуски с этим pipeline_run_id завершатся, прежде чем переходить к DEDUP.

7. Loop обучения эффективности

Проблема: После 10 gathering runs, оператор хочет узнать: "Какая комбинация source type + filter дала лучший target rate?"

Решение: Добавить вычисляемые метрики в gathering_runs (заполняется после фазы ANALYZE).

Метрики сбора и запрос dashboard ▼ 
gathering_runs (дополнительные столбцы):
├── target_rate            FLOAT     -- targets_found / new_companies_count
├── avg_analysis_confidence FLOAT    -- средняя confidence целей из этого запуска
├── cost_per_target_usd    NUMERIC   -- total_cost_usd / targets_found
├── enrichment_hit_rate    FLOAT     -- emails_found / targets_enriched
SELECT source_type, source_subtype,
  COUNT(*) AS runs,
  AVG(target_rate) AS avg_target_rate,
  AVG(cost_per_target_usd) AS avg_cost_per_target
FROM gathering_runs
WHERE project_id = :pid AND status = 'completed' AND target_rate IS NOT NULL
GROUP BY source_type, source_subtype
ORDER BY avg_target_rate DESC;

Этот feedback loop позволяет системе (или MCP агенту) рекомендовать: "Apollo seniority search имеет 22% target rate для этого проекта. Clay companies search имеет 8%. Рекомендуется использовать Apollo."

8. Гибкость экспорта (подключаемые выходы)

Проблема: Результаты должны идти в разные направления: Google Sheets (отзыв), CSV (скачать), SmartLead (кампания), GetSales (LinkedIn), Clay table (дальнейший enrichment), webhook (пользовательская интеграция).

Решение: Output адаптеры (тот же паттерн, что и gathering адаптеры):

OutputAdapter паттерн и 6 типов выходов ▼ 
class OutputAdapter(ABC):
    output_type: str  # "google_sheets" | "csv" | "smartlead" | "getsales" | "clay_table" | "webhook"

    async def export(self, companies: list, contacts: list, config: dict) -> ExportResult

6 типов выходов:

  • google_sheets — экспорт в Google Sheets для отзыва и ручной сортировки
  • csv — скачать как CSV для внешних инструментов
  • smartlead — отправить кампанию напрямую в SmartLead (leads + sequence)
  • getsales — интеграция LinkedIn для GetSales кампаний
  • clay_table — добавить в Clay table для дальнейшего enrichment
  • webhook — POST в custom webhook для интеграций

Частично существует: clay_service.export_to_google_sheets(), pipeline_service.promote_to_crm(), SmartLead push. Нужно только формализовать как адаптеры.

Реальный пример: EasyStaff Global (Project 9)

Этот конвейер использовался для сбора, анализа и проверки 7 900+ целевых компаний в 20+ городах. Конвейер создал 8 новых SmartLead кампаний (региональные разделения: US, UK, Gulf, India, APAC, Australia, LatAm-Africa, плюс варианты). EasyStaff Global имеет примерно 170 кампаний в общей сложности — большинство были созданы до существования этого конвейера, посредством ручных процессов.

Пошагово: Полный 11-шаговый конвейер

1GATHER из Apollo
Применить фильтры keyword + city + size к Apollo Companies UI. ~80 ключевых слов × 20 городов = 50+ gathering runs. Фильтры каждого run сохранены в gathering_runs.filters (JSONB).
Puppeteer emulator — бесплатно
2DEDUP + BLACKLIST
Нормализация доменов → company_source_links (multi-source bridge). Проверка против 170 существующих кампаний EasyStaff → отклонение пересечений.
★ CP1 — оператор подтверждает scope проекта
3SCRAPE веб-сайтов
50 параллельных соединений, streaming per-company commits. Crash-safe: on_result callback сохраняет каждую компанию индивидуально.
httpx + Apify proxy — бесплатно
4PRE-FILTER алгоритмически
Без AI — детерминированный. Удалить: веб-сайт недоступен, офлайн индустрии (ресторан, отель, строительство), мусорные домены (.gov, .edu), пустые/припаркованные сайты. ~40-60% отклонение — дёшево, быстро, без ложных отрицаний.
Детерминированный — $0
5ANALYZE от GPT-4o-mini
AI классифицирует каждую компанию: is_target? confidence? segment? reasoning? Использует ICP prompt конкретно для этого проекта.
~$0.01-0.05 за batch из 500
6VERIFY от Opus (Claude)
16 параллельных Opus агентов проверяют ВСЕ GPT targets. Каждый агент получает ~260 компаний, проверяет вердикт GPT. Находит false positives: SaaS products, solo consultants, неправильная география, government entities.
7ADJUST PROMPT
Повторить шаги 5→6 до ≥90% accuracy. Это критический loop (см. все 8 версий ниже).
↺ Если < 90% → назад к Шагу 5
8PEOPLE SEARCH
Найти до 3 лиц, принимающих решения, на проверенную целевую компанию. Фильтры: founder, c_suite, vp, director seniority. Результат: ~1.4 контакта на компанию в среднем.
Apollo People emulator — бесплатно
9FINDYMAIL email verification
Для contacts, где Apollo не предоставляет проверенные emails. 438 contacts проверены в одном 88-min batch.
$0.01/email ★ CP3 — оператор утверждает потрату
10GOD_SEQUENCE
Генерация campaign sequence через Gemini 2.5 Pro. Система собирает knowledge из 3 уровней в один prompt (см. GOD_SEQUENCE 3-level knowledge system ниже).
~$0.08 за generation
11SMARTLEAD campaign creation + lead upload
Создаёт SmartLead campaign (DRAFT state). Загружает leads с custom fields: city, segment, sender_name. Добавляет 12 sender email accounts за кампанию. 8 региональных кампаний создано через этот конвейер.

Глубокий разбор Шага 7: Loop итерации промпта (все 8 версий)

Это то, что отличает хороший gathering от плохого. Конвейер не "запустить GPT один раз и отправить." Это закрытый loop, где Opus проверяет работу GPT, выявляет паттерны false-positive и кормит их обратно в уточнённые промпты. EasyStaff требовала 8 итераций:

Версия Точность Размер выборки Ключевое изменение / Находка
V1 0% — Сложная scoring rubric, неправильные segments в целом. Полный перезапуск необходим.
V2 76% — Via negativa подход, CAPS_LOCKED segments. Лучше, но всё ещё пропускает граничные случаи.
V3 93% Small sample Geography filter, solo consultant exclusion. Близко, но нужна полная Opus validation.
V4 83% Full Opus review Strict location, investment exclusion. Over-corrected, отклонил некоторые валидные targets.
V5 86% — Entity type patterns, gov exclusion, country name detection. Фаза рафинирования.
V6 88% — Interior design, company formation, fake site detection. Приближаясь к threshold.
V7 93.6% 645/689 verified Уточнённое SERVICE vs PRODUCT distinction. Opus ловит оставшиеся false positives.
V8 95.1% ✓ 2,645/2,782 verified Финальный SERVICE business focus. Loop выходит когда ≥90% accuracy подтверждена.

Логика loop: На каждой итерации Opus находит FP (false positive) patterns → добавить exclusions в промпт → re-analyze с обновленным GPT промптом → re-verify с Opus → измерить accuracy → если ≥90%, DONE; иначе iterate.

Что Opus ловит, что GPT пропускает:

  • Game studios строящие own IP (не занимаются client work)
  • Management/strategy consulting (использует employees, не freelancers)
  • SaaS products (не service businesses)
  • Solo consultants / fractional CxOs (1-person operations)
  • Неправильная география (Indian "Pvt Ltd" entities, Oman, Lebanon)
  • Government subsidiaries
  • Investment/VC firms

Глубокий разбор Шага 10: GOD_SEQUENCE — 3-уровневая система знаний

Campaign sequence не является AI магией — это knowledge engineering. Gemini получает 3 уровня контекста:

УРОВЕНЬ 1: УНИВЕРСАЛЬНЫЙ

Campaign patterns, которые применяются ко ВСЕМ проектам (без business context):

Subject line: {{first_name}} – [pain point question]
Timing: Day 0/3/4/7/7 (Steps 2+4 tied at 31% warm replies каждый)
Tone: Casual-professional, no hype words
Body arc: Hook → Value → Proof → CTA
CTA: Offer value ("calculate cost benefit?"), не просить время
Flow: Value → Competition → Price → Channel → Empathy

УРОВЕНЬ 2: БИЗНЕС

Product knowledge, общий для проектов ОДНОГО и того же бизнеса (grouped by sender_company):

Пример: Project 9 + 40 = оба из "easystaff.io"
Включает:
  • Competitors & competitive positioning
  • Pricing strategy (hourly, fixed, retainer)
  • Displacement stories (кого заменяете)
  • Proof points (case studies, ROI metrics)
  • Outreach/GTM patterns изученные из sibling projects

УРОВЕНЬ 3: ПРОЕКТ

Специфичный контекст для ЭТОГО проекта только:

Включает:
  • ICP (Ideal Customer Profile): точные компании sizes, industries, geographies
  • Market segment (например, "staffing agencies в UK LatAm focus")
  • Language & region (en-US, en-GB, es-ES)
  • Sender identity (name, title, company voice)
  • Existing contacts & relationship map (избежать дубликатов, персонализировать для accounts)

Все 3 уровня → ~3,000 token промпт → Gemini 2.5 Pro → 5-step sequence

API calls:

  • POST /api/campaign-intelligence/generate-sequence/ — Generate draft sequence
  • POST /api/campaign-intelligence/generated/{id}/approve/ — Mark as approved
  • POST /api/campaign-intelligence/generated/{id}/push/ — Push to SmartLead

Стоимость: ~$0.08 за generation

Отслеживание эффективности ключевых слов

Каждый gathering run хранит свои фильтры. После анализа, target_rate вычисляется за run. Это позволяет нам ранжировать keywords по ROI:

Tier 1 (35-45%) — Лучшее
Keywords: staffing agency, design agency, marketing agency
Они дают самое высокое значение. Приоритет здесь.
Tier 2 (26-35%) — Хорошее
Keywords: outsourcing company, digital agency, creative agency
Solid performance. Стоит продолжать.
Tier 3 (14-23%) — Низкое
Keywords: software development, consulting firm, IT services
Ниже ideal, но всё ещё полезно для coverage.
НЕ ИСПОЛЬЗОВАТЬ (<5%) — Отходы
Keywords: fintech, saas, blockchain, crypto
Стоп. Это time wasters для этого ICP.
SQL запрос для эффективности ключевых слов ▼ 
SELECT source_subtype, AVG(target_rate) AS avg_target_rate, COUNT(*) AS runs
FROM gathering_runs
WHERE project_id = 9 AND status = 'completed'
GROUP BY source_subtype
ORDER BY avg_target_rate DESC;

Ключевые выводы

1. Puppeteer > API для gathering

Apollo emulator бесплатен, API стоит credits. Использовали API только для 10K credits blitz для охвата большего количества keywords быстрее. Для большинства проектов свободный Puppeteer выигрывает на ROI.

2. Loop итерации промпта незаменим

V1 имела 0% accuracy, V8 достигла 95.1%. Восемь итераций, каждая управляется Opus false-positive analysis. Нет shortcut. Это core конкурентное преимущество.

3. Эффективность ключевых слов варьируется в 10 раз

"staffing agency" попадает на 45% target rate, "fintech" попадает меньше чем на 5%. Отследить target_rate за run избежать потери времени на плохих keywords. Data-driven keyword selection сокращает costs наполовину.

4. Streaming scrape незаменим

Старый batch scrape терял весь прогресс при крахе. Новый streaming подход commits per-company через on_result callback. 50 параллельных соединений, ~344 companies/min. Resilience бьёт raw speed.

5. People search заполняет gap

Apollo Companies даёт domains, но не contacts. Отдельный People search step находит 1-3 decision-makers на компанию. FindyMail только необходим для emails, которые Apollo не может проверить (10-15% contacts).

Итог: 11 принципов, которые делают это god-level

1. Source-agnostic

БД хранит opaque JSONB. Новый source = новый адаптер, нулевые изменения схемы БД. Добавить 10 sources без refactoring.

2. Filter memory

Каждый поиск запомнен с точными параметрами. Re-runnable & auditable. "Какие keywords использовали 3 месяца назад?"

3. Multi-source dedup

Одна компания найдена 5 sources → 1 DiscoveredCompany + 5 source_links. Нет дубликатов, полная lineage.

4. Cheapest first

Free pattern matching → free blacklist → cheap scraping → expensive AI → costly enrichment. Никогда не потратить на AI то, что free pattern-matching может сделать.

5. Full provenance

gathering_run → DiscoveredCompany → ExtractedContact → CRM Contact → Meeting → Deal. Ответить: "Какой gathering run привёл к этой сделке?"

6. Versioned scraping

Несколько страниц на компанию, TTL-based re-scrape, все версии сохранены. Веб-сайт изменился? Проверить историю версий.

7. Multi-run analysis

Разные модели, разные промпты, сравнить результаты, оператор override. Запустить GPT + Opus параллельно, выбрать лучшее или объединить.

8. Approval gates

Human-in-the-loop перед каждым credit-spending шагом. Нет runaway costs. Оператор подтверждает scope в DEDUP, снова перед FINDYMAIL.

9. MCP-ready

Адаптеры экспортируют JSON schemas. tam_gather_{source} tools auto-registered. Claude может вызвать: "gather 500 designers из New York."

10. Learning loop

target_rate, cost_per_target отслеживаются за run. Система рекомендует лучшие sources. "Apollo seniority search: 22% target rate. Clay companies: 8%. Рекомендуется Apollo."

11. Project-aware

Читает ICP, products, competitors, case studies, existing contacts. Не dumb filter machine. Каждый промпт формируется project context.
