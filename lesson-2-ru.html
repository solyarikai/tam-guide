Урок 2

Контекст проекта и режимы ввода

Узнайте, что система знает о вашем проекте, четыре способа запроса запуска сбора данных оператором и как система связывает запуски со сегментами для отслеживания и анализа.

Сценарий

Оператор открывает систему сбора TAM, хочет найти 5000 цифровых агентств в ОАЭ и начинает печатать запрос. Система мгновенно знает о целевом профиле (ICP) проекта, конкурентов, тематические исследования, существующих контактов и предыдущие попытки фильтрации. Она использует весь этот контекст, чтобы преобразовать простой запрос оператора в точные фильтры источников — без необходимости оператору понимать коды старшинства Apollo или диапазоны размеров компаний Clay.

Чему вы научитесь
  • 10 источников данных, которые система читает из каждого проекта
  • Как ICP, сегменты и тематические исследования информируют решения по сбору
  • Четыре режима ввода оператора: естественный язык, структурированные фильтры, похожие, Expand/Repeat
  • Как запуски сбора связаны со сегментами для отслеживания TAM
  • Полный путь от конца к концу, когда оператор дает ввод на естественном языке

Что система знает о каждом проекте

Система сбора данных работает не в вакууме. Она читает существующие знания проекта, чтобы принимать умные решения.

Контекст проекта (уже существует в БД — переиспользовать, не перестраивать)

Источник Таблица/Поле Как сбор использует это
Определение ICP projects.target_segments, projects.target_industries → AI преобразует в фильтры, специфичные для источника. «Агентства <50 сотр в ОАЭ» становится фильтрами Apollo по локации, старшинству и размеру
Сегменты kb_segments (гибкие данные JSON на сегмент) → Каждый сегмент может иметь собственную конфигурацию сбора. Сегмент «Dubai Agencies» → apollo.companies.emulator с фильтрами ОАЭ. Сегмент «AU-PH Corridor» → clay.people.emulator с фильтрами филиппинского языка
Продукты kb_products (что мы продаем) → Подсказка анализа AI: «Нуждается ли эта компания в {product}?»
Конкуренты kb_competitors (домены, сильные стороны, слабые стороны) → Автоматическая блокировка доменов конкурентов. Не собирайте компании, которые уже используют конкурента
Тематические исследования kb_case_studies (отрасль клиента, размер, вызов, решение) → Поиск похожих: «Найдите компании, похожие на этих клиентов»
Профиль компании kb_company_profile (кто МЫ есть) → Контекст анализа AI: «Мы есть {company}, мы предлагаем {products}»
Знания проекта project_knowledge (category=icp/outreach/contacts/gtm) → Богатый контекст ICP, правила outreach, GTM стратегия
Существующие контакты contacts + campaigns (CRM) → Черный список: не переориентировать компании, уже находящиеся в outreach
Enterprise черный список project_blacklist + enterprise_blacklist.json → Жесткое исключение перед любой обработкой
Предыдущие запуски сбора gathering_runs (эта система) → «Не повторяйте фильтры, которые дали <5% целевой коэффициент»

Режимы ввода оператора

Когда оператор (или агент MCP) запускает запуск сбора, он может предоставить ввод несколькими способами:

Режим 1: Естественный язык (самый простой, оптимизирован для MCP)

Пример: «Найти 5000 цифровых агентств в ОАЭ с <100 сотрудниками»

ℹ️

→ AI преобразует в фильтры, используя контекст ICP и сегмента проекта
→ Предлагает лучший источник (Apollo для широкого, Clay для специфичного)
→ Возвращает предполагаемые результаты + стоимость

Режим 2: Структурированные фильтры (опытные операторы, точность)

Пример:

Показать JSON 
{
  "source_type": "apollo.people.emulator",
  "filters": {
    "person_locations": ["Dubai, United Arab Emirates"],
    "person_seniorities": ["founder", "c_suite"],
    "organization_num_employees_ranges": ["1,10", "11,50", "51,100"]
  }
}
ℹ️

→ Пропускает преобразование AI, прямое выполнение

Режим 3: Похожие (из тематических исследований или существующих клиентов)

Пример: «Найти компании, похожие на: frizzon.ae, 10xbrand.com, zopreneurs.com»

ℹ️

→ Обратный инжиниринг шаблонов (отрасль, размер, локация, tech stack)
→ Создание фильтров, которые соответствуют шаблону
→ Уже частично существует: reverse_engineering_service.py

Режим 4: Expand/Repeat (итерация по предыдущим запускам)

Пример: «Запустить тот же поиск, что и gathering_run #42, но для Сингапура вместо Дубая»

ℹ️

→ Копировать фильтры, изменить локацию
→ Дедупликация фильтров (проверка хэша) предотвращает случайные дубликаты

Связь запуска сбора ↔ сегмента

Каждый запуск сбора МОЖЕТ быть связан с определенным сегментом:

Показать код 
gathering_runs.segment_id  FK(kb_segments) NULL

Это позволяет системе:

  • Отслеживать, какие сегменты были собраны (и какие нет)
  • Рассчитать TAM на уровне сегмента: «Сегмент Dubai Agencies: 5150 найдено компаний, 1200 проанализировано, 340 целевых, 89 обогащено»
  • Предложить недостаточно собранные сегменты: «AU-PH Corridor имеет 0 запусков сбора — рассмотрите начало»

Как все это связывается: естественный язык к фильтрам

Когда оператор дает ввод на естественном языке, вот путь в 10 шагов:

  1. Оператор говорит: «Найти 5000 цифровых агентств в ОАЭ с <100 сотрудниками»
  2. Система загружает контекст проекта: Извлекает ICP, сегменты, продукты, конкурентов, тематические исследования из KB
  3. AI преобразует намерение в измерения: «агентства» = тег отрасли, «ОАЭ» = локация, «<100 сотрудников» = диапазон размера
  4. AI выбирает лучший источник: Смотрит на source_label + исторический уровень успеха. Apollo's «companies.emulator» → хорошо для широких поисков. Clay's «people.api» → лучше для таргетирования, специфичного для основателя. Выбирает Apollo.
  5. AI создает канонические фильтры: Читает схему фильтров Apollo из source_config, переводит «цифровые агентства в ОАЭ» в фильтры, специфичные для Apollo: {"company_industries": ["software", "digital_marketing"], "company_locations": ["Dubai, UAE"], "organization_num_employees_ranges": ["1,10", "11,50", "51,100"]}
  6. Система проверяет filter_hash: Вычисляет SHA256 отсортированного канонического JSON. Проверяет, запускалась ли эта комбинация фильтров раньше. Если да, показывает статистику последнего запуска: «Похожие фильтры нашли 3847 компаний на прошлой неделе; 412 прошли анализ (10.7% целевой коэффициент)»
  7. Система оценивает стоимость + результаты: «Предполагаемо 4200 результатов, Apollo будет стоить ~$210 (по $0.05/компания), Clay будет стоить ~$840. Используйте Apollo.»
  8. Система показывает предварительный просмотр: «Готовы к запуску? Запуск сбора будет: source=apollo.companies.emulator, filters={...}, предполагаемый результат 4200, стоимость $210. Одобрить?»
  9. Оператор одобряет. Система создает запись gathering_runs с status=pending, filters=JSON, filter_hash=SHA256, segment_id=(дополнительно)
  10. Выполнить и отслеживать: Собирает сырые результаты, дедублирует против discovered_companies, черные списки и продвигается через трубопровод — все используя фильтры, сохраненные в записи gathering_run.
Ключевой вывод

Контекст проекта — это не просто украшение — это двигатель умных значений по умолчанию. Система читает 10 источников данных (ICP, сегменты, продукты, конкуренты, тематические исследования, профиль компании, знания проекта, существующие контакты, черные списки, прошлые запуски) и использует их для преобразования намерения оператора в точные, специфичные для источника фильтры. Операторы могут запросить 4 способами (естественный язык, структурированные фильтры, похожие, expand/repeat), и каждый запуск сбора может быть связан со сегментом для отслеживания TAM. Этот дизайн делает систему как удобной для оператора, так и детерминированной: фильтры всегда явные, всегда сохраняются и всегда воспроизводимы.
