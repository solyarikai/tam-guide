Сервисный слой, API и план реализации

Воплотите систему сбора в жизнь: реализуйте сервис-оркестратор, API-эндпоинты, перенесите существующие данные и выполните поэтапную сборку.

Сценарий

Вам только что поручили построить систему сбора TAM. Вам нужно знать: Что орхестрирует конвейер? Какой API вызывают клиенты фронтенда и MCP? Какие файлы я создаю, изменяю или переиспользую? В каком порядке я их строю? Как мне импортировать 25 000 существующих корпоративных записей, не потеряв атрибуцию источника?

GatheringService — оркестратор

Класс GatheringService — это один класс Python, который орхестрирует весь конвейер сбора данных. Он валидирует входные данные, вызывает адаптеры, сохраняет результаты, триггерит последующую обработку и предоставляет эндпоинты истории.

class GatheringService:
    """Orchestrates the full TAM gathering pipeline."""

    # ── GATHER ──
    async def start_gathering(
        self, project_id: int, source_type: str, filters: dict,
        triggered_by: str = "operator", notes: str = None
    ) -> GatheringRun:
        """Create GatheringRun, validate filters, execute adapter, store results."""

    # ── DEDUP ──
    async def dedup_and_store(
        self, gathering_run_id: int, companies: list[dict]
    ) -> DedupeResult:
        """Normalize domains, check existing DiscoveredCompanies, create/link."""

    # ── BLACKLIST (deterministic, $0) ──
    async def run_blacklist_check(
        self, gathering_run_id: int
    ) -> BlacklistResult:
        """Check vs project_blacklist, enterprise_blacklist, CRM mat.view, existing DCs."""

    # ── SCRAPE ──
    async def scrape_companies(
        self, gathering_run_id: int, pages: list[str] = ['/'],
        method: str = 'httpx', force: bool = False
    ) -> ScrapeResult:
        """Scrape website content. Skip if current non-expired scrape exists (unless force=True)."""

    # ── ANALYZE ──
    async def start_analysis(
        self, project_id: int, model: str, prompt_text: str,
        scope_filter: dict, triggered_by: str = "operator"
    ) -> AnalysisRun:
        """Create analysis run, process companies, store results."""

    async def compare_analysis_runs(
        self, run_id_1: int, run_id_2: int
    ) -> ComparisonResult:
        """Compare two runs: agreements, disagreements, confidence deltas."""

    # ── APPROVAL ──
    async def create_gate(
        self, project_id: int, gate_type: str, scope: dict, label: str
    ) -> ApprovalGate:
        """Create approval gate. Pipeline pauses until approved."""

    async def approve_gate(self, gate_id: int, operator: str, note: str = None) -> None
    async def reject_gate(self, gate_id: int, operator: str, note: str = None) -> None

    # ── CONTINUE PIPELINE ──
    async def continue_pipeline(
        self, gathering_run_id: int, next_phase: str
    ) -> PhaseResult:
        """Resume pipeline from a specific phase."""

    # ── HISTORY ──
    async def get_runs(self, project_id: int, source_type: str = None) -> list[GatheringRun]
    async def get_run_detail(self, run_id: int) -> GatheringRunDetail
    async def get_run_companies(self, run_id: int, page: int = 1) -> PaginatedCompanies

Как работает start_gathering() внутри

Когда API получает запрос POST /api/pipeline/gathering/start, сервис орхестрирует эти 9 шагов:

  1. Создать запись GatheringRun — сохраните project_id, source_type, filters, status='running', created_by, notes.
  2. Валидировать фильтры — используйте схему Pydantic для source_type, чтобы проверить структуру фильтра и обязательные поля.
  3. Загрузить адаптер — найдите класс адаптера в реестре по source_type (например, "apollo.people.emulator" → ApolloOrgAPIAdapter).
  4. Выполнить адаптер — вызовите adapter.execute(filters) → возвращает список сырых словарей компаний.
  5. Обновить raw_results_count — установите GatheringRun.raw_results_count = len(results).
  6. Дедублировать — вызовите dedup_and_store(run_id, results) → нормализирует домены, проверяет/создает DiscoveredCompanies, создает CompanySourceLinks.
  7. Запустить проверку черного списка — вызовите run_blacklist_check(run_id) → отметьте отклоненные компании, создайте gate, если найдены.
  8. Добавить скрейпинг в очередь — для принятых компаний добавьте scrape_companies(run_id) в очередь фоновых заданий (Celery/RQ).
  9. Вернуть GatheringRun — верните созданный запуск со статусом='running' и job_id для опроса фронтенда.

API Эндпоинты — все 16

Эти эндпоинты образуют полный публичный интерфейс для конвейера сбора:

Method Эндпоинт Описание
POST /api/pipeline/gathering/start Запустить новый запуск сбора. Body: project_id, source_type, filters, triggered_by, notes. Возвращает: GatheringRun со статусом и job_id.
GET /api/pipeline/gathering/runs Список всех запусков сбора для проекта. Query: project_id, source_type (опционально), status (опционально), date_range (опционально). Возвращает: пагинированный список GatheringRun со счетчиками.
GET /api/pipeline/gathering/runs/{id} Получить полную информацию о запуске: метаданные, статистика (raw_count, new_companies, duplicates, rejected, accepted), сводка фильтра, временная шкала фазы. Возвращает: GatheringRunDetail.
GET /api/pipeline/gathering/runs/{id}/companies Получить пагинированные компании из запуска. Query: page (default 1), limit (default 50). Возвращает: PaginatedCompanies с метаданными DiscoveredCompany + source link.
POST /api/pipeline/gathering/continue/{id} Возобновить конвейер с определенной фазы. Body: next_phase (например, 'scrape', 'analyze'). Возвращает: PhaseResult с job_id и статусом фазы.
POST /api/pipeline/gathering/estimate Получить оценку стоимости без выполнения. Body: source_type, filters. Возвращает: { estimated_cost_usd, estimated_duration_seconds, billable_entities }.
GET /api/pipeline/gathering/sources Список всех доступных адаптеров источников с их JSON-схемами фильтров. Возвращает: список { source_type, description, schema, example_filters }.
GET /api/pipeline/gathering/sources/{type}/schema Получить JSON-схему для определенного типа источника. Возвращает: схему Pydantic как JSON Schema spec для генерации формы фронтенда.
GET /api/pipeline/gathering/approval-gates Список ожидающих gates одобрения (status='pending'). Query: project_id (опционально), gate_type (опционально). Возвращает: список ApprovalGate с деталями scope.
POST /api/pipeline/gathering/approval-gates/{id}/approve Одобрить gate. Body: operator (user ID), note (опционально). Обновляет gate.status='approved', возобновляет конвейер.
POST /api/pipeline/gathering/approval-gates/{id}/reject Отклонить gate. Body: operator (user ID), note (обязательно). Обновляет gate.status='rejected', триггерит логику очистки/повтора.
GET /api/pipeline/gathering/scrapes/{company_id} Получить все скрейпы для компании. Возвращает: список CompanyScrape с версиями, временами, страницами, сводками содержания.
POST /api/pipeline/gathering/scrapes/refresh Триггер повторного скрейпинга устаревшего контента. Body: company_ids (список). Возвращает: job_id и предполагаемое время завершения.
GET /api/pipeline/gathering/analysis-runs Список запусков анализа. Query: project_id, status, date_range. Возвращает: пагинированный список AnalysisRun со сводкой результатов.
GET /api/pipeline/gathering/analysis-runs/{id} Получить деталь запуска анализа со сводкой результатов по уровню доверия (high/medium/low), метаданные модели.
GET /api/pipeline/gathering/analysis-runs/{a}/compare/{b} Сравнить два запуска анализа. Возвращает: { agreements, disagreements (по дельте доверия), new findings, dropped findings, metrics diff }.
GET /api/pipeline/gathering/blacklist-check Сухой запуск проверки черного списка по доменам. Query: domains (разделенные запятыми). Возвращает: { blacklisted: [...], reasons: {...}, sources: [...] }.

Миграция существующих данных EasyStaff

У нас есть примерно 25 000 компаний и 12 000 записей людей, уже собранные. Мы должны импортировать их в новую систему, сохраняя атрибуцию источника и метаданные запуска.

Что существует

Файл Записи Источник
data/dubai_agency_companies_full.json 295 apollo_people_ui, Strategy A (32 ключевых слова)
data/uae_god_search_companies.json 5,602 apollo_people_ui, Strategy B (seniority + location)
data/uae_god_search_people.json 12,201 apollo_people_ui, Strategy B (записи людей)
data/uae_20k_companies.json 7,782 apollo_companies_ui (теги отрасли + ключевые слова)

Сценарий миграции: scripts/migrate_existing_tam.py — 5 шагов

Шаг 1: Создать GatheringRuns для каждого исторического поиска

  • Запуск 1 (Strategy A, поиск по ключевым словам):
    • source_type: apollo.people.emulator
    • source_subtype: strategy_a
    • filters: { "person_locations": ["Dubai, UAE"], "q_organization_name": "32 keywords", "max_pages": 10 }
    • status: completed
    • created_by: migration_script
    • raw_results_count: 295 (установить после дедупликации)
  • Запуск 2 (Strategy B, seniority + location):
    • source_type: apollo.people.emulator
    • source_subtype: strategy_b
    • filters: { "person_locations": ["Dubai, UAE", "Abu Dhabi, UAE", "Sharjah, UAE"], "person_seniorities": ["founder", "c_suite", "owner"], "organization_num_employees_ranges": ["1,10", "11,50", "51,200", "201,500", "501,1000", "1001,5000", "5001,10000", "10001,+" ], "max_pages": 10 }
    • status: completed
    • raw_results_count: 5,602
  • Запуск 3 (Вкладка Companies, теги отрасли + сотрудники):
    • source_type: apollo.companies.emulator
    • source_subtype: industry_tags
    • filters: { "organization_locations": ["United Arab Emirates"], "organization_industry_tag_ids": [...], "organization_num_employees_ranges": [...], "max_pages": 100 }
    • status: completed
    • raw_results_count: 7,782

Шаг 2: Создать/обновить DiscoveredCompanies для каждого уникального домена

  • Прочитайте каждую запись компании из JSON файлов.
  • Извлеките домен (или сконструируйте из имени компании, если отсутствует).
  • Вызовите domain_service.normalize_domain(raw_domain) для стандартизации (строчные, удалить www и т.д.).
  • Upsert DiscoveredCompany по (project_id, domain). Если существует, обновите только пустые поля (name, employees, linkedin_url) с самыми богатыми данными источника.
  • Свяжите с GatheringRun через столбец gathering_run_id.
  • Установите first_source_date на временную метку миграции, last_source_date = now, source_count = 1 (увеличивается при связи с несколькими запусками ниже).

Шаг 3: Создать company_source_links для каждой пары (company, run)

  • Для каждой компании в JSON файлах создайте запись CompanySourceLink.
  • Поля: company_id (FK к DiscoveredCompany), gathering_run_id (FK к GatheringRun из шага 1), source_data (сырая запись JSON), source_rank (позиция в исходном списке результатов, с индексом 1).
  • Это сохраняет полную атрибуцию: какая компания пришла из какого запуска и в каком рейтинге.
  • Когда компания появляется в нескольких запусках (например, и Strategy A, и Strategy B), увеличьте source_count в DiscoveredCompany.

Шаг 4: Импортировать людей как ExtractedContacts (из uae_god_search_people.json)

  • Разберите 12 201 запись людей из uae_god_search_people.json.
  • Для каждого человека извлеките name, title, email, company, location.
  • Свяжите с DiscoveredCompany по домену (извлеченному из имени компании или предыдущего сопоставления).
  • Создайте запись ExtractedContact с source=APOLLO, gathering_run_id=Run2 (Strategy B).
  • Сохраните все сырые поля в contact_data (JSON) для последующего обогащения или дедупликации.

Шаг 5: Обновить счетчики GatheringRun

  • Для каждого GatheringRun вычислите и сохраните счетчики:
  • raw_results_count = количество записей в JSON файле.
  • new_companies_count = количество созданных DiscoveredCompanies (не уже в проекте).
  • duplicate_count = raw_count - new_count (компании, которые уже существовали).
  • rejected_count = количество записей, которые не прошли проверку черного списка или валидацию домена (если имеются).
  • accepted_count = new + duplicates (все успешно сохраненные).

Порядок миграции

  1. Запустить миграцию для крупнейшего набора данных первым (uae_god_search = 5,602 компании). Это устанавливает базовый набор DiscoveredCompany и избегает более поздних слияний.
  2. Затем мигрируйте поиск по ключевым словам (295) — в основном дубликаты Strategy B, но создает source_links для атрибуции.
  3. Затем мигрируйте вкладку companies (7,782) — много новых (другой метод обнаружения), некоторые перекрывающиеся. Обновляет DiscoveredCompany более богатыми данными сотрудников/отрасли.
  4. Отдельно импортируйте записи людей (12,201) — свяжите с существующими DiscoveredCompanies по домену.

Ключевой вывод

GatheringService является единственным оркестратором для всего конвейера. Он вызывает адаптеры для получения данных, сохраняет результаты в нормализованной схеме с полной атрибуцией источника, запускает детерминированные фильтры (черный список, валидация домена), триггерит скрейпинг и анализ, управляет approval gates и подает историю/detail endpoints. 16 API эндпоинтов раскрывают все эти возможности клиентам фронтенда и MCP. Путем построения по фазам (data → core → scraping → migration → gates), вы держите каждый слой тестируемым и независимым. Сценарий миграции вносит 25K исторических записей без потери данных или нарушенной линии происхождения.
