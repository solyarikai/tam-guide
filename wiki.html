Getting Started

What the system does, why it exists, and core design principles behind every decision.

What We're Building #

A reusable system that remembers everything about TAM gathering:

  • Which filters were applied in which source (Apollo, Clay, Sales Navigator, etc.)
  • Which companies were found from which filters
  • Website content scraped (multiple pages, versioned with TTL)
  • AI analysis runs (multiple models, multiple prompts, all stored)
  • Blacklist checks against CRM/campaigns
  • Approval gates before credit-spending steps

MCP-ready: Each source is a pluggable adapter module. Today they're Puppeteer emulators; tomorrow users connect their own Apollo/Clay API keys. The DB layer is the same.

Reusable across projects: EasyStaff Global, OnSocial, Inxy, any future project — same tables, same pipeline.

Runs on Hetzner only: All scripts, scrapers, migrations, DB queries execute on the production server (46.62.210.24). Local machines are for code editing only. Claude Code SSHes to Hetzner for all execution.

Key Concepts & Terminology #

Gathering Run
One execution of a source adapter with specific filters. Stores EXACTLY which filters were applied, results count, cost. Immutable.
Discovered Company
Unique company domain (normalized). Can be found by multiple gathering runs (multi-source). Linked via company_source_links.
Company Source Link
Bridge table: links one company to the gathering run(s) that found it. Preserves per-source metadata.
Company Scrape
Website content for one company page. Versioned, TTL-based re-scrape. One company can have scrapes of /, /about, /contact, etc.
Analysis Run
One AI analysis pass over companies (with specific model + prompt). Multiple runs with different prompts to compare.
Approval Gate
Pauses the pipeline before credit-spending steps. Operator reviews scope, cost, then approves or rejects.
Filter Hash
SHA256 of canonical JSON filters. Used to detect duplicate searches (same filters = same hash).
Pipeline Phase
One step in the linear pipeline: GATHER, DEDUP, BLACKLIST, CP1, PRE-FILTER, RESOLVE, SCRAPE, ANALYZE, CP2, VERIFY, CP3, GOD_SEQ, PUSH.

Pipeline

How companies flow through the system — from search to email campaign. Steps, checkpoints, cost gates, and data sources.

Pipeline Flow #

Strict linear progression, no skipping:

GATHER + DEDUP → BLACKLIST → ★ CP1 — STOP → PRE-FILTER → SCRAPE → ANALYZE → ★ CP2 — STOP → VERIFY → ★ CP3 — STOP → GOD_SEQ → PUSH

Phase state machine (gathering_run.current_phase):

gathered → awaiting_scope_ok → scope_approved → filtered → scraped → analyzed → awaiting_targets_ok → targets_approved → awaiting_verify_ok → verify_approved → verified → pushed

At awaiting_* phases, the run is physically stuck in the database. An approval_gate record exists. The ONLY way forward is POST /approval-gates/{gate_id}/approve. No code path skips this. It survives crashes, session reloads, server restarts.

Phase Details (10 Phases + 3 Checkpoints) #

# Phase Cost Auto? What it does
1 GATHER varies auto Execute adapter, get raw company list from source
2 DEDUP $0 auto Normalize domains, match to existing DiscoveredCompanies, create source_links. "Already known" = exists in discovered_companies for THIS project from a previous gathering run.
3 BLACKLIST $0 auto Project-scoped CRM check + project_blacklist + enterprise_blacklist
— ★ CHECKPOINT 1 — STOP Project confirmation + blacklist review. Shows: project name/ID, total contacts, ALL active campaigns (name, platform, leads), per-campaign rejection breakdown, enterprise blacklist, cross-project warnings. Operator confirms: "Yes, this is my project, and the scope is correct." Code-enforced — cannot be bypassed. If wrong project → cancel run. If wrong campaign → fix campaign_filters.
4 PRE-FILTER $0 auto Deterministic rejection: offline industries (restaurant, hotel, construction), junk domains (.gov, .edu), trash patterns. NO AI. Rejects 40-60%.
5 RESOLVE $0-low auto For companies missing domain: LinkedIn URL → domain. Company name → Google lookup. Skip if domain already known.
6 SCRAPE $0 (httpx) auto Fetch website content. TTL check — skip if fresh. Multiple pages. Cheap — no approval needed.
7 ANALYZE ~$0.01-0.05 auto AI analysis (GPT-4o-mini). Scores: industry_match, size_match, service_model. Cheap — no approval needed.
— ★ CHECKPOINT 2 — STOP Operator reviews target list. Must see: every company marked as target, its confidence score, reasoning, segment. Operator can accept, reject individual companies, override verdicts. Only after operator confirms the target list does the pipeline proceed. This is where the operator ensures the AI didn't hallucinate. No credits are spent until this is approved.
8 VERIFY $$$ BLOCKED FindyMail email verification. This is the expensive step. Only runs on operator-approved targets.
— ★ CHECKPOINT 3 — STOP Operator approves FindyMail spend. Must see: how many emails to verify, estimated cost, which companies. Operator can remove companies before verification.
9 GOD_SEQUENCE ~$0.08 auto Generate 5-step sequence from 3-level knowledge: universal patterns + business knowledge (same sender_company) + project ICP. Operator reviews draft before push. POST /generate-sequence/ → POST /approve/
10 PUSH $0 BLOCKED POST /generated/{id}/push/ creates SmartLead campaign (DRAFT) with GOD_SEQUENCE output. Operator adds leads + activates.

Session Crash Recovery #

All checkpoint state is persisted in the database (gathering_run.current_phase + approval_gates table).

What happens if Claude Code session crashes at checkpoint 1:

  1. Next session: Claude Code reads CLAUDE.md which says "check for in-progress runs first"
  2. Queries: GET /runs?project_id=X → finds run with current_phase=awaiting_scope_ok
  3. Queries: GET /approval-gates?project_id=X → finds pending gate with gate_type=scope_verification
  4. Reads gate.scope which contains the FULL blacklist detail (rejected_domains, warning_domains, campaign breakdown)
  5. Shows the same checkpoint 1 results to the operator
  6. Operator approves → pipeline continues

No data is lost. The gate's scope JSON stores everything needed to reconstruct the checkpoint display.

Scrape Failure Reporting #

When analysis runs, companies without scraped text are SKIPPED (not analyzed). The analysis response includes:

  • total_eligible: how many companies could have been analyzed
  • skipped_no_scraped_text: how many were skipped because scraping failed
  • total_analyzed: how many were actually analyzed

Claude Code MUST report this at checkpoint 2: "Analyzed 300 of 500 eligible companies. 200 skipped (scrape failed — no website text)."

Operator Input Modes (4 modes) #

Mode 1: Natural Language (easiest, MCP-optimized) 
"Find 5000 digital agencies in UAE with <100 employees"

→ AI maps to filters using project's ICP + segment context
→ Suggests best source (Apollo for broad, Clay for specific)
→ Returns estimated results + cost

Mode 2: Structured Filters (experienced operators, precise) 
{
  "source_type": "apollo.people.emulator",
  "filters": {
    "person_locations": ["Dubai, United Arab Emirates"],
    "person_seniorities": ["founder", "c_suite"],
    "organization_num_employees_ranges": ["1,10", "11,50", "51,100"]
  }
}

→ Bypasses AI mapping, direct execution

Mode 3: Lookalike (from case studies or existing customers) 
"Find companies similar to: frizzon.ae, 10xbrand.com, zopreneurs.com"

→ Reverse-engineer patterns (industry, size, location, tech stack)
→ Generate filters that match the pattern
→ Already partially exists: reverse_engineering_service.py

Mode 4: Expand/Repeat (iterate on previous runs) 
"Run the same search as gathering_run #42 but for Singapore instead of Dubai"

→ Copy filters, modify location
→ Filter dedup (hash check) prevents accidental duplicates

Blacklist & Gates

How the system prevents duplicate outreach and enforces human approval at every cost-incurring step.

Project-Scoped Blacklisting #

Problem: Company X is being contacted by EasyStaff RU (project 40, payroll offer). Inxy (project 10, crypto payments) starts gathering and finds Company X. Should it be blacklisted?

Answer: NO. Different projects sell different products to different ICPs. Company X can legitimately receive outreach from both projects — they're separate value propositions. Auto-blocking across projects would artificially reduce TAM.

Rule: Only campaigns belonging to the SAME project trigger auto-rejection. Other projects' campaigns are shown as warnings (never auto-rejected).

Blacklist Check Layers (in order) #

# Layer Scope Action What it catches
1 Project blacklist Per-project Auto-reject project_blacklist table — manual bans by operator
2 Same-project campaigns Per-project Auto-reject Contacts with domains in THIS project's active campaigns
3 Enterprise blacklist Global Auto-reject enterprise_blacklist.json — competitors, banned orgs
4 Cross-project campaigns All projects Warning only Domains in OTHER projects' campaigns (informational)

Materialized View (Project-Scoped) #

CREATE MATERIALIZED VIEW active_campaign_domains AS
SELECT DISTINCT
  lower(c.domain) AS domain,
  c.project_id,
  p.name AS project_name,
  array_agg(DISTINCT camp.id) AS campaign_ids,
  array_agg(DISTINCT camp.name) AS campaign_names,
  count(DISTINCT c.id) AS contact_count
FROM contacts c
JOIN projects p ON p.id = c.project_id
JOIN campaigns camp ON camp.project_id = c.project_id AND camp.status = 'active'
WHERE c.domain IS NOT NULL AND c.domain != ''
GROUP BY lower(c.domain), c.project_id, p.name;

CREATE UNIQUE INDEX ON active_campaign_domains(domain, project_id);
CREATE INDEX ON active_campaign_domains(project_id);

Same-project check: WHERE domain = ANY(:domains) AND project_id = :pid

Cross-project check: WHERE domain = ANY(:domains) AND project_id != :pid

Refresh: After campaign sync in crm_scheduler.py + after any campaign push. REFRESH MATERIALIZED VIEW CONCURRENTLY (no locks).

Adapter System

Pluggable data source connectors — Clay, Apollo, Google Sheets, and more. All sources share the same interface.

MCP Adapter Pattern & Base Class #

Critical design decision: The DB is source-agnostic. It does NOT know about Apollo, Clay, or any specific platform. Adding a new source = new adapter class + register. Zero DB changes.

class GatheringAdapter(ABC):
    """Base class for all gathering source adapters."""

    source_type: str                              # "apollo.companies.emulator"
    filter_model: Optional[Type[BaseModel]]       # Pydantic model

    @abstractmethod
    async def validate(self, raw_filters: dict) -> dict:
        """Validate & normalize filters. Returns cleaned dict."""

    @abstractmethod
    async def estimate(self, filters: dict) -> EstimateResult:
        """Estimate cost/results without executing."""

    @abstractmethod
    async def execute(self, filters: dict, on_progress: Callable = None) -> GatheringResult:
        """Execute gathering. Returns companies list + metadata."""

    def get_filter_schema(self) -> Optional[dict]:
        """JSON Schema for MCP tool registration."""
        if self.filter_model:
            return self.filter_model.model_json_schema()
        return None

    def get_capabilities(self) -> dict:
        """What this adapter can do."""
        return {
            "source_type": self.source_type,
            "has_estimate": True,
            "has_filter_schema": self.filter_model is not None,
            "cost_model": "free",
            "requires_auth": False,
        }

Adapter Registry & Adding New Sources #

ADAPTER_REGISTRY: dict[str, Type[GatheringAdapter]] = {}

def register_adapter(cls: Type[GatheringAdapter]) -> Type[GatheringAdapter]:
    """Decorator. @register_adapter on adapter class auto-registers it."""
    ADAPTER_REGISTRY[cls.source_type] = cls
    return cls

def get_adapter(source_type: str) -> GatheringAdapter:
    cls = ADAPTER_REGISTRY.get(source_type)
    if not cls:
        raise ValueError(f"Unknown source: {source_type}")
    return cls()

def list_adapters() -> list[dict]:
    """Returns all registered adapters + capabilities."""
    return [get_adapter(st).get_capabilities() for st in sorted(ADAPTER_REGISTRY.keys())]

8 Critical Edge Cases #

Design decisions and solutions for complex scenarios:

1. Company Identity Resolution (Multi-Domain)

Problem: Same company can have multiple domains: acme.com, acme.ae, acme.co.uk. Apollo says "Frizzon Studios", Clay says "Frizzon Productions". Dedup by domain alone misses these.

Solution: Two-layer dedup:

  • Layer 1 (instant, free): Exact domain match after normalization (strip www, lowercase)
  • Layer 2 (fuzzy, in PRE-FILTER phase): Company name similarity (Levenshtein or trigram) + LinkedIn URL match. If two DiscoveredCompanies share a LinkedIn company URL → merge.

Implementation: discovered_companies.linkedin_company_url as secondary unique identifier. On insert, check both domain AND linkedin URL. If linkedin matches existing with different domain → link as alias.

company_aliases
├── primary_company_id    FK(discovered_companies)
├── alias_domain          VARCHAR(255)
├── alias_linkedin_url    TEXT
├── alias_name            VARCHAR(500)
├── detected_by           VARCHAR(50)

2. Domain Resolution Phase

Problem: 8,766 companies from Companies tab have LinkedIn URLs but NO domains. Can't scrape websites or check email enrichment without domains.

Solution: New pipeline phase RESOLVE between PRE-FILTER and SCRAPE:

  1. LinkedIn → domain: Apollo enrich_organization(linkedin_url) returns primary_domain (1 credit per call, high value)
  2. Company name → domain: Google search "{company_name}" site:{country_tld} → extract domain from first result
  3. Skip if unnecessary: Only run for companies missing domain after GATHER
  4. Budget-gated: Operator approves batch size before spending Apollo credits

3. Re-Run Lineage

Problem: Operator wants to re-run search from 2 weeks ago because "Apollo might have new companies now." But filter_hash dedup would reject it as duplicate.

Solution: parent_run_id on gathering_runs:

gathering_runs.parent_run_id  FK(gathering_runs) NULL

When parent_run_id is set, filter_hash dedup is bypassed. The system compares results: "Run #42 found 5,602 companies. Re-run #65 found 5,840. Net new: 238."

4. Conversion Provenance (Full Funnel Tracking)

Problem: "Which gathering run actually led to the deal we closed?" Can't answer without tracking the chain: gathering_run → DiscoveredCompany → ExtractedContact → Contact → Campaign → ProcessedReply → Meeting.

Solution: Already partially solved. Add: contacts.gathering_run_id FK for direct lineage:

contacts.gathering_run_id = Column(Integer, ForeignKey('gathering_runs.id'))

Full funnel query:

SELECT gr.source_type, gr.filters->>'person_locations' AS location,
  COUNT(DISTINCT dc.id) AS companies, COUNT(DISTINCT ec.id) AS contacts,
  COUNT(DISTINCT c.id) AS crm_contacts, COUNT(DISTINCT m.id) AS meetings
FROM gathering_runs gr
JOIN company_source_links csl ON csl.gathering_run_id = gr.id
JOIN discovered_companies dc ON dc.id = csl.discovered_company_id
LEFT JOIN extracted_contacts ec ON ec.discovered_company_id = dc.id
LEFT JOIN contacts c ON c.gathering_run_id = gr.id
LEFT JOIN meetings m ON m.contact_id = c.id
WHERE gr.project_id = :project_id
GROUP BY gr.id
ORDER BY meetings DESC;

5. Central Quota Management

Problem: Two operators run Apollo searches simultaneously → both hit 429 rate limit. Or MCP agent burns through all Clay credits in one run.

Solution: Adapter-level quota tracker (not DB — in-memory with Redis or file lock):

class QuotaManager:
    async def acquire(self, platform: str, credits: int, timeout: int = 300) -> bool:
        """Request credits. Blocks until available or timeout."""

    async def release(self, platform: str, credits: int) -> None:
        """Return unused credits."""

    async def get_usage(self, platform: str) -> QuotaUsage:
        """Current usage: {used_today, limit_today, used_this_hour, limit_this_hour}."""

Each adapter calls quota_manager.acquire() before API calls. Manager respects per-platform limits (Apollo 200/min, Clay 5000 exports/day, etc.).

7. Effectiveness Learning Loop

Problem: After 10 gathering runs, the operator wants to know: "Which source type + filter combination yielded the best target rate?"

Solution: Add computed metrics to gathering_runs (populated after ANALYZE phase):

gathering_runs (additional columns):
├── target_rate            FLOAT

  COUNT(*) AS runs,
  AVG(target_rate) AS avg_target_rate,

api_long() — Resilient Long-Running API Calls

Wrapper for API operations that take minutes to complete (scrape, analyze). The problem: scraping 500 websites takes 5–15 minutes, analyzing 500 companies — 10–20 minutes. During that time, HTTP connections can drop (timeout, backend restart, network hiccup). But the backend saves results to the database incrementally — data is never lost. Solution: if the connection drops, don't panic — poll the run status every 30 seconds until the phase advances.

def api_long(method: str, path: str, expected_phase: str, run_id: int,
             timeout: int = 3600, poll_interval: int = 30, **kwargs) -> dict:
    url = f"{BACKEND_BASE}/api{path}"
    try:
        r = getattr(httpx, method)(url, headers=BACKEND_HEADERS, timeout=timeout, **kwargs)
        if r.status_code >= 400:
            return {"_error": r.status_code, "_detail": r.text[:500]}
        return r.json()
    except (httpx.ReadTimeout, httpx.RemoteProtocolError,
            httpx.ConnectError, httpx.ReadError) as e:
        print(f"  Connection lost ({type(e).__name__}). Backend may still be working...")
        print(f"  Polling until phase reaches '{expected_phase}'...")

        start = time.time()
        while time.time() - start < timeout:
            time.sleep(poll_interval)
            try:
                r2 = httpx.get(f"{BACKEND_BASE}/api/pipeline/gathering/runs/{run_id}",
                               headers=BACKEND_HEADERS, timeout=30)
                if r2.status_code == 200:
                    phase = r2.json().get("current_phase", "")
                    elapsed = int(time.time() - start)
                    print(f"  [{elapsed}s] Phase: {phase}")
                    if phase == expected_phase or phase.startswith("awaiting_"):
                        print(f"  Backend finished — phase is now '{phase}'")
                        return r2.json()
            except Exception:
                print(f"  [{int(time.time()-start)}s] Backend unreachable, waiting...")

        print(f"  Timeout after {timeout}s. Check manually.")
        return {"_timeout": True}
expected_phase
The phase the run should reach AFTER the operation completes. For scrape: "scraped". For analyze: "analyzed". Polling stops when this phase (or any awaiting_* phase) is detected.
Connection errors handled
httpx.ReadTimeout, httpx.RemoteProtocolError, httpx.ConnectError, httpx.ReadError. Any of these triggers the polling fallback instead of crashing.
Timeout
Default 3600 seconds (1 hour). If the expected phase is not reached within this time, returns {"_timeout": True} — operator must check manually.
Poll interval
Every 30 seconds by default. Each poll prints elapsed time and current phase for operator visibility.
Why not just retry?
Retrying the POST would restart the operation on the backend (duplicate work). Polling the GET status endpoint is safe — it's read-only and shows whether the backend finished the job on its own.
Used by
step4_scrape() and step5_analyze() — both replaced api() with api_long() to handle long-running backend operations resilient to connection drops.

Batching & Auto-Pipeline #

The backend can't reliably scrape/analyze 3000+ companies in a single run — connections time out, memory grows, and errors cascade. The batching system splits large domain lists into runs of 500 and processes each run through the full pipeline automatically, without operator intervention at checkpoints.

BATCH_SIZE Constant

BATCH_SIZE = 500  # Max domains per gathering run (for scrape stability)

If the domain list is ≤ 500, a single run is created. If larger — the list is split into chunks of 500, each becoming a separate gathering run. All runs share the same project and land in the same database — results are unified.

create_batched_runs() — Split Domains Into Runs

def create_batched_runs(config: ProjectConfig, domains: list[str],
                        mode: str, notes_prefix: str = "") -> list[int]:
    """Create multiple gathering runs if domains > BATCH_SIZE.
    Returns list of run IDs."""
    if len(domains) <= BATCH_SIZE:
        batches = [domains]
    else:
        batches = [domains[i:i+BATCH_SIZE]
                   for i in range(0, len(domains), BATCH_SIZE)]
        print(f"\n  Splitting {len(domains)} domains into {len(batches)} batches of ~{BATCH_SIZE}")

    run_ids = []
    for i, batch in enumerate(batches):
        batch_label = f" (batch {i+1}/{len(batches)})" if len(batches) > 1 else ""
        result = api("post", "/pipeline/gathering/start", json={
            "project_id": config.project_id,
            "source_type": "manual.companies.manual",
            "filters": {"domains": batch},
            "triggered_by": "operator",
            "input_mode": mode,
            "notes": f"{notes_prefix}{batch_label} — {len(batch)} domains",
        })
        run_id = result["id"]
        run_ids.append(run_id)
        print(f"  Run #{run_id}: {len(batch)} domains{batch_label}")

    return run_ids
Source type
manual.companies.manual — domains are passed directly (no Clay/Apollo search). Used for re-processing known domain lists or expand mode.
Notes
Each run gets a descriptive note with batch number and domain count for easy identification in the UI.
Return value
List of run_id integers. Caller iterates over them and calls process_run_pipeline() for each.

process_run_pipeline() — Full Auto-Pipeline Per Run

Processes a single run through the entire gathering pipeline automatically: gather → blacklist → prefilter → scrape → analyze → approve → blacklist. No operator checkpoints — all gates are auto-approved. Uses api_long() for scrape and analyze steps to handle connection drops gracefully.

def process_run_pipeline(config: ProjectConfig, run_id: int,
                         prompt_text: str = None, skip_gather_wait: bool = False):
    # 1. Wait for gather phase to complete (poll every 10s, up to 10 min)
    if not skip_gather_wait:
        for i in range(60):
            time.sleep(10)
            phase = api("get", f"/pipeline/gathering/runs/{run_id}").get("current_phase")
            if phase != "gather": break

    # 2. Check current phase — skip already-completed steps
    phase = api("get", f"/pipeline/gathering/runs/{run_id}").get("current_phase")

    # 3. Blacklist check + auto-approve CP1
    if phase == "gathered":
        api("post", f"/pipeline/gathering/runs/{run_id}/blacklist-check")
        approve_pending_gate(config, run_id)

    # 4. Pre-filter (remove junk domains)
    if phase == "scope_approved":
        api("post", f"/pipeline/gathering/runs/{run_id}/pre-filter")

    # 5. Scrape — resilient via api_long()
    if phase == "filtered":
        step4_scrape(run_id)

    # 6. Analyze — resilient via api_long(), uses prompt_text
    if phase == "scraped":
        api_long("post", f"/pipeline/gathering/runs/{run_id}/analyze",
                 expected_phase="analyzed", run_id=run_id, timeout=3600,
                 params={"prompt_text": prompt_text, "model": "gpt-4o-mini"})

    # 7. Auto-approve targets (CP2) + blacklist approved domains
    approve_pending_gate(config, run_id)
    blacklist_approved_targets(config, run_id)
Phase-aware
At each step, the function checks the current phase via API before acting. If the run was interrupted and resumed, it picks up from where it left off — never repeats completed steps.
Auto-approval
All gates (CP1, CP2) are auto-approved via approve_pending_gate(). This is safe for batched runs because the operator already approved the overall domain list before batching started.
Resilience
Scrape and analyze use api_long() — if the backend connection drops during a 15-minute scrape, the pipeline polls until it completes instead of crashing.
skip_gather_wait
If True, skips the initial gather wait loop. Used when domains are passed via manual source (instant dedup, no Clay delay).

approve_pending_gate() — Auto-Approve Checkpoint

def approve_pending_gate(config: ProjectConfig, run_id: int) -> bool:
    """Find and approve pending gate for this run."""
    gates = api("get",
        f"/pipeline/gathering/approval-gates?project_id={config.project_id}",
        raise_on_error=False)
    items = gates if isinstance(gates, list) else gates.get("items", [])
    for g in items:
        if g.get("gathering_run_id") == run_id and g.get("status") == "pending":
            api("post", f"/pipeline/gathering/approval-gates/{g['id']}/approve",
                json={}, raise_on_error=False)
            print(f"  Gate #{g['id']} approved")
            return True
    return False
How it works
Queries all approval gates for the project, finds the one matching this run_id with status "pending", and approves it automatically.
Difference from approve_gate()
approve_gate(gate_id) requires a known gate ID (used in interactive mode). approve_pending_gate(run_id) finds the gate automatically — used in batch/auto mode.
Error handling
Uses raise_on_error=False and try/except — if no pending gate exists, returns False silently. Safe to call at any pipeline stage.

Path to Apollo People CSV export. Required for --from-step people.

Advanced

--prompt-file
Custom classification prompt from a text file. Overrides the prompt from DB. Used for testing new prompts.
--re-analyze
Re-run GPT classification on an existing run without re-gathering. Requires --run-id. Uses latest prompt from DB or --prompt-file.
