Урок 3

Слой данных

Полная схема для запусков сбора, дедупликации, веб-скрейпинга, анализа ИИ и врата одобрения. Это самый большой и критически важный урок для разработчиков, пишущих миграции и код сервисов.

Сценарий

Вы пишете миграцию Alembic для добавления новой системы сбора в production. Вам нужно понять каждое поле в каждой таблице, каждый индекс, внешние ключи и логику проектирования для каждого выбора. Вам также нужно знать, как таблицы связаны между собой и какие паттерны данных вы будете запрашивать.

Чему вы научитесь
  • Полная схема для gathering_runs (ID источника, фильтры, дедупликация через хеш)
  • Почему БД агностична к источникам и как добавлять новые адаптеры без миграций
  • company_source_links: Мост мультиисточниковой дедупликации с UNIQUE ограничением
  • company_scrapes: Версионированный контент сайта с переправой на основе TTL
  • gathering_prompts: Переиспользуемые шаблоны ИИ с отслеживанием эффективности
  • analysis_runs & analysis_results: Сравнение разных моделей ИИ и промптов
  • approval_gates: Контрольные точки оператора перед дорогостоящими операциями
  • Расширения существующих таблиц (discovered_companies, search_jobs)

Таблица 1: gathering_runs — Память фильтров (критически отсутствующая часть)

Каждое выполнение поиска = один запись. Эта таблица помнит ТОЧНО какие фильтры были применены, их результаты и сколько они стоили. Это основа для дедупликации, переисполнения и принятия решений оператором.

Ключевые решения при проектировании

source_type как свободный VARCHAR (не enum)

Критическое решение при проектировании: БД агностична к источникам. Она НЕ знает об Apollo, Clay или любой другой конкретной платформе. Добавление нового источника = новый класс адаптера + регистрация. Нулевые изменения БД.

Почему не типизированные модели по источникам в БД?

  • Один только Apollo имеет: People UI, Companies UI, People API, Org API — 4 варианта
  • Clay имеет: Companies UI, Companies API, People UI, People API — еще 4
  • Sales Navigator, Google Maps, LinkedIn, Crunchbase, Indeed... — продолжает расти
  • Каждая платформа добавляет новые фильтры регулярно — обновления модели = миграции = простой
  • Взрыв: платформы x цели x методы = слишком много моделей для поддержки

Трехмерный подход

Измерение Колонка Примеры
Платформа префикс source_type apollo, clay, sales_navigator, google_maps, crunchbase, csv
Цель середина source_type companies, people, jobs
Метод суффикс source_type api, emulator, manual

source_type = {platform}.{target}.{method} — например apollo.companies.emulator, clay.people.api

Но source_type это просто свободный VARCHAR — система никогда его не парсит. Адаптер владеет семантикой. Завтра вы добавляете indeed.jobs.api и ничего не меняется в БД.

filters как непрозрачный JSONB

БД хранит все, что адаптер в нее помещает. Она никогда не запрашивает отдельные поля фильтров. Она использует фильтры для:

  1. Сохранение — запись один раз когда запуск сбора начинается
  2. Отображение — чтение и показ оператору в UI
  3. Хеширование — SHA256 для обнаружения дедупа (одинаковые фильтры = одинаковый хеш)
  4. Переисполнение — передача обратно адаптеру без изменений

Нет валидации Pydantic на уровне БД. Валидация — это работа адаптера. Каждый адаптер определяет СВОЮ модель Pydantic с extra='allow' — неизвестные поля сохраняются. Когда Apollo добавляет новый фильтр, он проходит без изменения кода.

filter_hash для дедупа

filter_hash = SHA256(отсортированного канонического JSON фильтров)

Одинаковые фильтры = одинаковый хеш. Используйте это для обнаружения и предотвращения дублирующихся запусков. Запрос: WHERE project_id = X AND filter_hash = :hash → найти если этот точный набор фильтров уже был запущен.

Таблица 2: company_source_links — Мост мультиисточниковой дедупликации

Решает: "Acme.com был найден поиском ключевых слов Apollo, поиском по старшинству Apollo И экспортом TAM Clay." Эта таблица отслеживает каждый источник, который нашел компанию, сохраняя что каждый источник сказал о ней.

Полное дерево схемы 
company_source_links
├── id                      SERIAL PK
├── discovered_company_id   FK(discovered_companies, CASCADE) NOT NULL
├── gathering_run_id        FK(gathering_runs, CASCADE) NOT NULL
├── source_rank             INTEGER                     -- позиция в результатах источника (1-я, 50-я, и т.д.)
├── source_data             JSONB                       -- сырая запись от этого источника для этой компании
├── source_confidence       FLOAT                       -- оценка релевантности от источника
├── found_at                TIMESTAMPTZ DEFAULT now()
│
├── UNIQUE (discovered_company_id, gathering_run_id)
├── INDEX (gathering_run_id)
└── INDEX (discovered_company_id)

Как работает дедупликация (4-шаговый поток)

  1. GatheringRun #1 находит домен `acme.com` → создает DiscoveredCompany + company_source_link(run=1, rank=5)
  2. GatheringRun #2 находит `acme.com` снова → находит существующую DC по домену → создает только новую ссылку(run=2, rank=2)
  3. Запрос мультиисточникового количества: SELECT COUNT(DISTINCT gathering_run_id) FROM company_source_links WHERE discovered_company_id = X = "найдено 2 источниками"
  4. source_data сохраняет различия — Apollo сказал 500 сотрудников, Clay сказал 450. Обе мнения хранятся в JSONB source_data для каждой ссылки.

Таблица 3: company_scrapes — Версионированный контент сайта с TTL

Несколько страниц на компанию. Версионировано. TTL-основанная переправа. Отслеживает метод скрейпинга (httpx, crona, puppeteer) и статус (success, timeout, blocked, js_only).

Полное дерево схемы 
company_scrapes
├── id                      SERIAL PK
├── discovered_company_id   FK(discovered_companies, CASCADE) NOT NULL
├── url                     TEXT NOT NULL                -- https://acme.com/about
├── page_path               VARCHAR(255) DEFAULT '/'    -- /, /about, /contact, /team, /careers
├── raw_html                TEXT                         -- сырой HTML (макс 100KB)
├── clean_text              TEXT                         -- извлеченный читаемый текст
├── metadata                JSONB                        -- {title, description, language, cyrillic_ratio, word_count, og_tags}
├── scraped_at              TIMESTAMPTZ DEFAULT now()
├── ttl_days                INTEGER DEFAULT 180          -- 6 месяцев по умолчанию
├── expires_at              TIMESTAMPTZ                  -- вычисляется: scraped_at + interval(ttl_days)
├── is_current              BOOLEAN DEFAULT true         -- последняя версия для этой (company, page_path)
├── version                 INTEGER DEFAULT 1            -- увеличивается при переправе
├── scrape_method           VARCHAR(50) DEFAULT 'httpx'  -- httpx | crona | puppeteer | apify
├── scrape_status           VARCHAR(30) DEFAULT 'success' -- success | error | timeout | blocked | redirect | empty | js_only
├── error_message           TEXT
├── http_status_code        INTEGER
├── html_size_bytes         INTEGER
├── text_size_bytes         INTEGER
├── created_at              TIMESTAMPTZ DEFAULT now()
│
├── INDEX (discovered_company_id, page_path, is_current)
├── INDEX (discovered_company_id) WHERE is_current = true
├── INDEX (expires_at) WHERE is_current = true           -- планировщик переправы
└── INDEX (scrape_status)

TTL поток переправы

Запланированная работа запрашивает WHERE is_current = true AND expires_at < now(). При переправе:

  1. Старая запись: is_current = false
  2. Новая запись: is_current = true, version = old.version + 1
  3. Все исторические скрейпы сохранены — можно сравнить изменения веб-сайта с течением времени

Обратная совместимость

Новый код пишет в company_scrapes. Также копирует последнюю домашнюю страницу (page_path='/') в discovered_companies.scraped_html/scraped_text/scraped_at для существующих путей кода, которые еще не были обновлены.

Таблица 4a: gathering_prompts — Переиспользуемые шаблоны AI промптов

Один и тот же промпт переиспользуется в разных запусках анализа. Дедупирован по хешу SHA256. Отслеживает эффективность на промпт (целевой коэффициент, уверенность, количество использований).

Полное дерево схемы 
gathering_prompts
├── id                    SERIAL PK
├── company_id            FK(companies) NOT NULL
├── project_id            FK(projects) NULL              -- NULL = глобальный/общий промпт
├── name                  VARCHAR(255) NOT NULL           -- "EasyStaff UAE Agency ICP v2"
├── prompt_text           TEXT NOT NULL
├── prompt_hash           VARCHAR(64) NOT NULL UNIQUE     -- SHA256, одинаковый текст = одинаковый промпт
├── category              VARCHAR(50) DEFAULT 'icp_analysis'  -- icp_analysis | segment_classification | pre_filter | enrichment
├── model_default         VARCHAR(100) DEFAULT 'gpt-4o-mini'
├── version               INTEGER DEFAULT 1
├── parent_prompt_id      FK(gathering_prompts) NULL      -- цепочка версий для итераций
│
│   ── TRACKING EFFECTIVENESS (обновляется после каждого запуска анализа) ──
├── usage_count           INTEGER DEFAULT 0
├── avg_target_rate       FLOAT                           -- targets / всего проанализировано
├── avg_confidence        FLOAT
├── total_companies_analyzed  INTEGER DEFAULT 0
│
├── created_by            VARCHAR(100)
├── is_active             BOOLEAN DEFAULT true
├── created_at            TIMESTAMPTZ DEFAULT now()
├── updated_at            TIMESTAMPTZ DEFAULT now()
│
├── INDEX (company_id, project_id)
└── INDEX (company_id, category)

Жизненный цикл промпта (4 шага)

  1. Оператор пишет промпт → get_or_create_prompt() создает запись (или находит существующую по хешу)
  2. Запуск анализа связывает с промптом через FK prompt_id
  3. После завершения запуска → update_prompt_stats() пересчитывает эффективность
  4. Панель управления: "Какой промпт имеет лучший целевой коэффициент?" → сортировка по avg_target_rate

Итерация версии

Когда улучшаете промпт, установите parent_prompt_id на старую версию. Обе остаются используемыми; система отслеживает какая версия работает лучше. Запрос: SELECT * FROM gathering_prompts WHERE parent_prompt_id = X ORDER BY avg_target_rate DESC для сравнения всех версий промпта.

Таблицы 4b: analysis_runs & analysis_results — Версионирование анализа AI

Множество проходов анализа с разными моделями/промптами. Сравнивайте результаты. Сохраняйте все. Проектирование поддерживает A/B тестирование и итерации промптов.

Полное дерево схемы: analysis_runs 
analysis_runs
├── id                    SERIAL PK
├── project_id            FK(projects) NOT NULL
├── company_id            FK(companies) NOT NULL
├── prompt_id             FK(gathering_prompts) NULL      -- ссылка на переиспользуемый промпт
├── model                 VARCHAR(100) NOT NULL           -- gemini-2.5-pro | gpt-4o-mini | gpt-4o | claude-sonnet-4
├── prompt_hash           VARCHAR(64) NOT NULL            -- SHA256 текста промпта
├── prompt_text           TEXT NULL                       -- встроенный fallback если нет prompt_id
├── scope_type            VARCHAR(50) DEFAULT 'batch'   -- batch | single | re_analysis | comparison
├── scope_filter          JSONB                         -- {gathering_run_id: 5} или {status: "new"} или {company_ids: [1,2,3]}
├── status                VARCHAR(30) DEFAULT 'pending' -- pending | running | completed | failed | cancelled
├── started_at            TIMESTAMPTZ
├── completed_at          TIMESTAMPTZ
├── total_analyzed        INTEGER DEFAULT 0
├── targets_found         INTEGER DEFAULT 0
├── rejected_count        INTEGER DEFAULT 0
├── avg_confidence        FLOAT
├── total_cost_usd        NUMERIC(10,4) DEFAULT 0
├── total_tokens          INTEGER DEFAULT 0
├── triggered_by          VARCHAR(100)

Полное дерево схемы: analysis_results 
analysis_results
├── id                      SERIAL PK
├── analysis_run_id         FK(analysis_runs, CASCADE) NOT NULL
├── discovered_company_id   FK(discovered_companies, CASCADE) NOT NULL
├── is_target               BOOLEAN DEFAULT false
├── confidence              FLOAT                       -- 0.0-1.0
├── segment                 VARCHAR(100)                -- совпавшее имя сегмента
├── reasoning               TEXT                        -- объяснение AI
├── scores                  JSONB                       -- {industry: 0.9, size: 0.7, service: 0.6, digital: 0.8}
├── raw_output              TEXT                        -- полный ответ AI (для отладки)
├── override_verdict        BOOLEAN                     -- ручное переопределение оператором
├── override_reason         TEXT
├── overridden_at           TIMESTAMPTZ
├── tokens_used             INTEGER
├── cost_usd                NUMERIC(10,6)
├── created_at              TIMESTAMPTZ DEFAULT now()
│
├── UNIQUE (analysis_run_id, discovered_company_id)
├── INDEX (discovered_company_id)
└── INDEX (analysis_run_id, is_target)

Сравнение двух запусков анализа (SQL запрос)

Показать SQL запрос 
SELECT dc.domain, dc.name,
  a1.is_target AS run1, a1.confidence AS conf1, a1.reasoning AS why1,
  a2.is_target AS run2, a2.confidence AS conf2, a2.reasoning AS why2
FROM discovered_companies dc
JOIN analysis_results a1 ON a1.discovered_company_id = dc.id AND a1.analysis_run_id = :run1
JOIN analysis_results a2 ON a2.discovered_company_id = dc.id AND a2.analysis_run_id = :run2
WHERE a1.is_target != a2.is_target  -- несогласия только
ORDER BY ABS(a1.confidence - a2.confidence) DESC;

Этот запрос находит где два модели не согласились — существенно для отладки и выбора лучшего подхода.

Таблица 5: approval_gates — Контрольные точки оператора

Конвейер паузирует перед шагами траты кредитов. Оператор просматривает область + предполагаемую стоимость, затем одобряет/отклоняет. Предотвращает случайные дорогостоящие запуски.

Полное дерево схемы 
approval_gates
├── id                    SERIAL PK
├── project_id            FK(projects) NOT NULL
├── pipeline_run_id       FK(pipeline_runs) NULL
├── gathering_run_id      FK(gathering_runs) NULL
├── gate_type             VARCHAR(50) NOT NULL          -- pre_scrape_crona | pre_analysis | pre_enrichment | pre_verification | pre_push
├── gate_label            VARCHAR(255) NOT NULL         -- "Одобрить 150 компаний для обогащения Apollo (~$4.50)"
├── scope                 JSONB NOT NULL                -- {count: 150, company_ids: [...], estimated_cost_usd: 4.50, estimated_credits: 150}
├── status                VARCHAR(30) DEFAULT 'pending' -- pending | approved | rejected | expired
├── decided_by            VARCHAR(100)
├── decided_at            TIMESTAMPTZ
├── decision_note         TEXT
├── expires_at            TIMESTAMPTZ                   -- автоматически истечет если не решено
├── created_at            TIMESTAMPTZ DEFAULT now()
│
├── INDEX (project_id, status)
└── INDEX (status) WHERE status = 'pending'

Когда врата создаются (5 типов)

  • pre_scrape_crona: Crona стоит кредитов — врата перед пакетным скрейпингом через Crona
  • pre_analysis: AI анализ стоит токены — врата перед пакетным анализом GPT/Gemini
  • pre_enrichment: Apollo enrich стоит 1 кредит/персона — врата перед enrich_apollo_batch
  • pre_verification: FindyMail стоит кредиты — врата перед verify_emails_batch
  • pre_push: Финальный обзор перед пушем SmartLead/GetSales

Скрейпинг httpx бесплатен

Врата не требуется. Только шаги траты кредитов получают врата. Свободные операции (скрейпинг httpx, базовый анализ с включенными моделями) проходят без контрольных точек.

Ключевой вывод

Слой данных спроектирован для гибкости, аудиторствуемости и дедупа. Таблицы никогда не ограничивают вас определенными платформами или источниками. Фильтры остаются непрозрачными, хеши предотвращают дубликаты, и версионирование (скрейпы сайта, промпты, запуски анализа) позволяет вам сравнивать изменения с течением времени. Каждая таблица включает индексы оптимизированные для запросов, которые вы фактически будете запускать. Когда вы пишете свои миграции, включайте каждое поле, каждый индекс и каждое ограничение — ничто не опционально.
