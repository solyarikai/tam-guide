Lesson 7

Edge Cases, Real-World Example & Key Learnings

The final and most detailed lesson: 8 critical edge cases, the full EasyStaff Global story (11 steps, all 8 prompt versions, GOD_SEQUENCE explained), and the 11 principles that make the pipeline god-level.

Scenario

You've learned the architecture. Now your lead wants you to see the real, messy edge cases that emerge in production and walk through the complete EasyStaff Global story: 7,900+ companies, 20+ cities, 8 regional campaigns. You'll also see all the variations that separate best-in-class from mediocre.

The 8 Critical Edge Cases

1. Company Identity Resolution (Multi-Domain)

Problem: Same company can have multiple domains: acme.com, acme.ae, acme.co.uk. Apollo says "Frizzon Studios", Clay says "Frizzon Productions". Dedup by domain alone misses these.

Solution: Two-layer dedup:

  • Layer 1 (instant, free): Exact domain match after normalization (strip www, lowercase)
  • Layer 2 (fuzzy, in PRE-FILTER phase): Company name similarity (Levenshtein or trigram) + LinkedIn URL match. If two DiscoveredCompanies share a LinkedIn company URL → merge.

Implementation: discovered_companies.linkedin_company_url as secondary unique identifier. On insert, check both domain AND linkedin URL. If linkedin matches existing with different domain → link as alias, don't create duplicate.

company_aliases table schema ▼ 
-- Optional: company aliases table for multi-domain entities
company_aliases
├── primary_company_id    FK(discovered_companies)
├── alias_domain          VARCHAR(255)
├── alias_linkedin_url    TEXT
├── alias_name            VARCHAR(500)
├── detected_by           VARCHAR(50)   -- domain_match | linkedin_match | name_fuzzy | manual

2. Domain Resolution Phase

Problem: 8,766 companies from Companies tab have LinkedIn URLs but NO domains. Can't scrape websites or check email enrichment without domains.

Solution: New pipeline phase RESOLVE between PRE-FILTER and SCRAPE:

  1. LinkedIn → domain: Apollo enrich_organization(linkedin_url) returns primary_domain (1 credit per call, but high value)
  2. Company name → domain: Google search "{company_name}" site:{country_tld} → extract domain from first result
  3. Skip if unnecessary: Only run for companies missing domain after GATHER
  4. Budget-gated: Operator approves batch size before spending Apollo credits

3. Re-Run Lineage

Problem: Operator wants to re-run search from 2 weeks ago because "Apollo might have new companies now." But filter_hash dedup would reject it as duplicate.

Solution: parent_run_id on gathering_runs:

parent_run_id schema & logic ▼ 
gathering_runs.parent_run_id  FK(gathering_runs) NULL  -- "this is a re-run of run #42"

When parent_run_id is set, filter_hash dedup is bypassed. The system compares results: "Run #42 found 5,602 companies. Re-run #65 found 5,840. Net new: 238."

4. Conversion Provenance (Full Funnel Tracking)

Problem: "Which gathering run actually led to the deal we closed?" Can't answer without tracking the chain: gathering_run → DiscoveredCompany → ExtractedContact → Contact → Campaign → ProcessedReply → Meeting.

Solution: Already partially solved. Add: contacts.gathering_run_id FK for direct lineage.

Full funnel tracking query ▼ 
SELECT gr.source_type, gr.filters->>'person_locations' AS location,
  COUNT(DISTINCT dc.id) AS companies, COUNT(DISTINCT ec.id) AS contacts,
  COUNT(DISTINCT c.id) AS crm_contacts, COUNT(DISTINCT m.id) AS meetings
FROM gathering_runs gr
JOIN company_source_links csl ON csl.gathering_run_id = gr.id
JOIN discovered_companies dc ON dc.id = csl.discovered_company_id
LEFT JOIN extracted_contacts ec ON ec.discovered_company_id = dc.id
LEFT JOIN contacts c ON c.gathering_run_id = gr.id
LEFT JOIN meetings m ON m.contact_id = c.id
WHERE gr.project_id = :project_id
GROUP BY gr.id
ORDER BY meetings DESC;

5. Central Quota Management

Problem: Two operators run Apollo searches simultaneously → both hit 429 rate limit. Or MCP agent burns through all Clay credits in one run.

Solution: Adapter-level quota tracker (not DB — in-memory with Redis or simple file lock).

QuotaManager class (Python) ▼ 
class QuotaManager:
    """Central quota tracker per source platform."""

    async def acquire(self, platform: str, credits: int, timeout: int = 300) -> bool:
        """Request credits. Blocks until available or timeout."""

    async def release(self, platform: str, credits: int) -> None:
        """Return unused credits (e.g., search returned fewer results than expected)."""

    async def get_usage(self, platform: str) -> QuotaUsage:
        """Current usage: {used_today, limit_today, used_this_hour, limit_this_hour}."""

Each adapter calls quota_manager.acquire() before API calls. The manager respects per-platform limits (Apollo 200/min, Clay 5000 exports/day, etc.).

6. Parallel Gathering Strategy

Problem: For a new project, operator wants to search Apollo People + Apollo Companies + Clay TAM simultaneously, then merge results.

Solution: Multiple gathering_runs sharing the same pipeline_run_id. The pipeline waits for ALL runs with that pipeline_run_id to complete before proceeding to DEDUP.

7. Effectiveness Learning Loop

Problem: After 10 gathering runs, the operator wants to know: "Which source type + filter combination yielded the best target rate?"

Solution: Add computed metrics to gathering_runs (populated after ANALYZE phase).

Gathering metrics & dashboard query ▼ 
gathering_runs (additional columns):
├── target_rate            FLOAT     -- targets_found / new_companies_count
├── avg_analysis_confidence FLOAT    -- mean confidence of targets from this run
├── cost_per_target_usd    NUMERIC   -- total_cost_usd / targets_found
├── enrichment_hit_rate    FLOAT     -- emails_found / targets_enriched
SELECT source_type, source_subtype,
  COUNT(*) AS runs,
  AVG(target_rate) AS avg_target_rate,
  AVG(cost_per_target_usd) AS avg_cost_per_target
FROM gathering_runs
WHERE project_id = :pid AND status = 'completed' AND target_rate IS NOT NULL
GROUP BY source_type, source_subtype
ORDER BY avg_target_rate DESC;

This feedback loop lets the system (or MCP agent) recommend: "Apollo seniority search has 22% target rate for this project. Clay companies search has 8%. Recommend using Apollo."

8. Export Flexibility (Pluggable Outputs)

Problem: Results need to go to different destinations: Google Sheets (review), CSV (download), SmartLead (campaign), GetSales (LinkedIn), Clay table (further enrichment), webhook (custom integration).

Solution: Output adapters (same pattern as gathering adapters):

OutputAdapter pattern & 6 output types ▼ 
class OutputAdapter(ABC):
    output_type: str  # "google_sheets" | "csv" | "smartlead" | "getsales" | "clay_table" | "webhook"

    async def export(self, companies: list, contacts: list, config: dict) -> ExportResult

6 output types:

  • google_sheets — export to Google Sheets for review & manual sorting
  • csv — download as CSV for external tools
  • smartlead — push campaign directly to SmartLead (leads + sequence)
  • getsales — LinkedIn integration for GetSales campaigns
  • clay_table — append to Clay table for further enrichment
  • webhook — POST to custom webhook for integrations

Already partially exists: clay_service.export_to_google_sheets(), pipeline_service.promote_to_crm(), SmartLead push. Just need to formalize as adapters.

Real-World Example: EasyStaff Global (Project 9)

This pipeline was used to gather, analyze, and verify 7,900+ target companies across 20+ cities. The pipeline produced 8 new SmartLead campaigns (regional splits: US, UK, Gulf, India, APAC, Australia, LatAm-Africa, plus variants). EasyStaff Global has ~170 campaigns total — most were created before this pipeline existed, via manual processes.

Step-by-Step: The Complete 11-Step Pipeline

1GATHER from Apollo
Apply keyword + city + size filters to Apollo Companies UI. ~80 keywords × 20 cities = 50+ gathering runs. Each run's filters stored in gathering_runs.filters (JSONB). Tracks which keywords produce the best targets later.
Puppeteer emulator — free
2DEDUP + BLACKLIST
Domain normalization → company_source_links (multi-source bridge). Check against 170 existing EasyStaff campaigns → reject overlaps.
★ CP1 — operator confirms project scope
3SCRAPE websites
50 concurrent connections, streaming per-company commits. Crash-safe: on_result callback saves each company individually.
httpx + Apify proxy — free
4PRE-FILTER algorithmically
No AI — deterministic. Remove: website unreachable, offline industries (restaurant, hotel, construction), junk domains (.gov, .edu), empty/parked sites. ~40-60% rejection rate — cheap, fast, no false negatives.
Deterministic — $0
5ANALYZE by GPT-4o-mini
AI classifies each company: is_target? confidence? segment? reasoning? Uses ICP prompt specific to this project.
~$0.01-0.05 per batch of 500
6VERIFY by Opus (Claude)
16 parallel Opus agents review ALL GPT targets. Each agent gets ~260 companies, checks GPT's verdict. Identifies false positives: SaaS products, solo consultants, wrong geography, government entities, game studios building own IP.
7ADJUST PROMPT
Repeat Steps 5→6 until ≥90% accuracy. This is the critical loop (see all 8 versions below).
↺ If < 90% → back to Step 5
8PEOPLE SEARCH
Find up to 3 decision-makers per verified target company. Filters: founder, c_suite, vp, director seniority. Result: ~1.4 contacts per company average.
Apollo People emulator — free
9FINDYMAIL email verification
For contacts where Apollo doesn't provide verified emails. 438 contacts verified in one 88-min batch.
$0.01/email ★ CP3 — operator approves spend
10GOD_SEQUENCE
Generate campaign sequence via Gemini 2.5 Pro. System assembles knowledge from 3 levels into one prompt (see GOD_SEQUENCE 3-level knowledge system below).
~$0.08 per generation
11SMARTLEAD campaign creation + lead upload
Creates SmartLead campaign (DRAFT state). Uploads leads with custom fields: city, segment, sender_name. Adds 12 sender email accounts per campaign. 8 regional campaigns created via this pipeline.

Step 7 Deep Dive: The Prompt Iteration Loop (ALL 8 Versions)

This is what separates good gathering from bad. The pipeline is not "run GPT once and push." It's a closed loop where Opus verifies GPT's work, identifies false-positive patterns, and feeds those back into refined prompts. EasyStaff required 8 iterations:

Version Accuracy Sample Size Key Change / Finding
V1 0% — Complex scoring rubric, wrong segments entirely. Complete restart needed.
V2 76% — Via negativa approach, CAPS_LOCKED segments. Better but still missing edge cases.
V3 93% Small sample Geography filter, solo consultant exclusion. Getting close but needs full Opus validation.
V4 83% Full Opus review Strict location, investment exclusion. Over-corrected, rejected some valid targets.
V5 86% — Entity type patterns, gov exclusion, country name detection. Refinement phase.
V6 88% — Interior design, company formation, fake site detection. Approaching threshold.
V7 93.6% 645/689 verified Refined SERVICE vs PRODUCT distinction. Opus catches remaining false positives.
V8 95.1% ✓ 2,645/2,782 verified Final SERVICE business focus. Loop exits when ≥90% accuracy confirmed.

Loop logic: Each iteration, Opus finds FP (false positive) patterns → add exclusions to prompt → re-analyze with updated GPT prompt → re-verify with Opus → measure accuracy → if ≥90%, DONE; else iterate.

What Opus catches that GPT misses:

  • Game studios building own IP (not doing client work)
  • Management/strategy consulting (uses employees, not freelancers)
  • SaaS products (not service businesses)
  • Solo consultants / fractional CxOs (1-person operations)
  • Wrong geography (Indian "Pvt Ltd" entities, Oman, Lebanon)
  • Government subsidiaries
  • Investment/VC firms

Step 10 Deep Dive: GOD_SEQUENCE — 3-Level Knowledge System

The campaign sequence is not AI magic — it's knowledge engineering. Gemini gets fed 3 levels of context:

LEVEL 1: UNIVERSAL

Campaign patterns that apply to ALL projects (no business context):

Subject line: {{first_name}} – [pain point question]
Timing: Day 0/3/4/7/7 (Steps 2+4 tied at 31% warm replies each)
Tone: Casual-professional, no hype words
Body arc: Hook → Value → Proof → CTA
CTA: Offer value ("calculate cost benefit?"), don't ask for time
Flow: Value → Competition → Price → Channel → Empathy

LEVEL 2: BUSINESS

Product knowledge shared across projects of the SAME business (grouped by sender_company):

Example: Project 9 + 40 = both from "easystaff.io"
Includes:
  • Competitors & competitive positioning
  • Pricing strategy (hourly, fixed, retainer)
  • Displacement stories (who you're replacing)
  • Proof points (case studies, ROI metrics)
  • Outreach/GTM patterns learned from sibling projects

LEVEL 3: PROJECT

Specific context for THIS project only:

Includes:
  • ICP (Ideal Customer Profile): exact company sizes, industries, geographies
  • Market segment (e.g., "staffing agencies in UK LatAm focus")
  • Language & region (en-US, en-GB, es-ES)
  • Sender identity (name, title, company voice)
  • Existing contacts & relationship map (to avoid duplicates, personalize for accounts)

All 3 levels → ~3,000 token prompt → Gemini 2.5 Pro → 5-step sequence

API calls:

  • POST /api/campaign-intelligence/generate-sequence/ — Generate draft sequence
  • POST /api/campaign-intelligence/generated/{id}/approve/ — Mark as approved
  • POST /api/campaign-intelligence/generated/{id}/push/ — Push to SmartLead

Cost: ~$0.08 per generation

Key Learnings

1. Puppeteer > API for gathering

Apollo emulator is free, API costs credits. Used API only for the 10K credits blitz to cover more keywords faster. For most projects, free Puppeteer wins on ROI.

2. The prompt iteration loop is essential

V1 had 0% accuracy, V8 reached 95.1%. Eight iterations, each driven by Opus false-positive analysis. There is no shortcut. This is the core competitive advantage.

3. Keyword effectiveness varies 10x

"staffing agency" hits 45% target rate, "fintech" hits less than 5%. Track target_rate per run to avoid wasting time on bad keywords. Data-driven keyword selection cuts costs in half.

4. Streaming scrape is essential

Old batch scrape lost all progress on crash. New streaming approach commits per-company via on_result callback. 50 concurrent connections, ~344 companies/min. Resilience beats raw speed.

5. People search fills the gap

Apollo Companies gives domains, but not contacts. A separate People search step finds 1-3 decision-makers per company. FindyMail only needed for emails Apollo can't verify (10-15% of contacts).

Summary: The 11 Principles That Make This God-Level

1. Source-agnostic

DB stores opaque JSONB. New source = new adapter, zero DB schema changes. Add 10 sources without refactoring.

2. Filter memory

Every search remembered with exact parameters. Re-runnable & auditable. "What keywords did we use 3 months ago?"

3. Multi-source dedup

Same company found by 5 sources → 1 DiscoveredCompany + 5 source_links. No duplicates, full lineage.

4. Cheapest first

Free pattern matching → free blacklist → cheap scraping → expensive AI → costly enrichment. Never waste on AI what free pattern-matching can do.

5. Full provenance

gathering_run → DiscoveredCompany → ExtractedContact → CRM Contact → Meeting → Deal. Answer: "Which gathering run led to this deal?"

6. Versioned scraping

Multiple pages per company, TTL-based re-scrape, all versions preserved. Website changed? Check version history.

7. Multi-run analysis

Different models, different prompts, compare results, operator override. Run GPT + Opus in parallel, pick best or merge.

8. Approval gates

Human-in-the-loop before every credit-spending step. No runaway costs. Operator confirms scope at DEDUP, again before FINDYMAIL.

9. MCP-ready

Adapters expose JSON schemas. tam_gather_{source} tools auto-registered. Claude can call: "gather 500 designers from New York."

10. Learning loop

target_rate, cost_per_target tracked per run. System recommends best sources. "Apollo seniority search: 22% target rate. Clay companies: 8%. Recommend Apollo."

11. Project-aware

Reads ICP, products, competitors, case studies, existing contacts. Not a dumb filter machine. Every prompt is shaped by project context.
